# テンソルと計算グラフ：配列をどう持ち、どう微分するか

配列の章では一次元の可変長列を見ました。数値計算と機械学習の言語
（NumPy を載せた Python、Julia、PyTorch、JAX）が中心に据えるのは、その
多次元版 ── **テンソル**（tensor、多次元配列）です。ここで面白いのは
データ構造の二点に尽きます。**(1) 多次元の配列を、メモリ上の一本の列で
どう表すか**、そして **(2) その上の計算を、微分できるデータ構造として
どう記録するか**。本章はこの二つの表現を解剖します。

## ストライド付き配列：shape と strides

n 次元のテンソルも、メモリ上は**一本の連続したバッファ**です。多次元の
見かけは、次の数個のメタデータで与えます。

- **dtype**：要素の型（`float32`・`int8` など。1 要素のバイト数）。
- **shape**：各次元の長さ（例 `(2, 3)`）。
- **strides**：各次元を 1 進むとバッファ上で何バイト飛ぶか。
- **offset**：バッファの先頭から何バイト目が `[0,0,…]` か。

要素 `[i, j, …]` の在りかは、**`offset + Σ index[k] × strides[k]`** という
一次式で求まります。配列の章の「添字 → アドレス」の計算が、次元の数だけ
足し合わさっただけです。

```ruby
# 概念図：ストライドで多次元配列を一本のバッファから引く
class Tensor
  attr_reader :buf, :shape, :strides, :offset
  def initialize(buf, shape, strides, offset = 0)
    @buf, @shape, @strides, @offset = buf, shape, strides, offset
  end

  def [](*idx)
    pos = @offset
    idx.each_with_index { |v, k| pos += v * @strides[k] }  # 一次式で位置を出す
    @buf[pos]
  end
end

# 2x3 を行優先(C 順)で：行を進むと 3、列を進むと 1
t = Tensor.new([10, 20, 30, 40, 50, 60], [2, 3], [3, 1])
p t[0, 0]   # => 10
p t[1, 2]   # => 60   offset 0 + 1*3 + 2*1 = 5 番目
```

**strides をどう取るかが行優先／列優先の正体**です。行優先（C 順、NumPy
既定）は最後の次元の stride が最小、列優先（Fortran 順、Julia 既定）は
最初の次元が最小。APL の章で見た「平坦な本体＋shape」に **strides** を
足したのがテンソルで、この一手で次の魔法が効きます。

## ビューはストライドの操作にすぎない

テンソルの肝は、**形を変える操作の多くがデータをコピーしない**ことです。
`reshape`・`transpose`・スライスは、**バッファはそのままに shape・strides・
offset だけを書き換えた新しいテンソル（ビュー, view）**を返します。

- **転置**は strides の**入れ替え**だけ。`(2,3)` で strides `[3,1]` のものを
  転置すると、shape `(3,2)`・strides `[1,3]` ── バッファは一切動きません。
- **スライス** `t[1.., ::2]` は offset を進め、strides を整数倍し、shape を
  縮めるだけ。
- **ブロードキャスト**（後述）は stride 0 の次元を足すだけ。

```ruby
# 転置はストライドを入れ替えるだけ（ゼロコピー）
def transpose(t)
  Tensor.new(t.buf, t.shape.reverse, t.strides.reverse, t.offset)
end
tt = transpose(t)        # 2x3 → 3x2 のビュー
p tt[2, 1]               # => 60   元の t[1,2] と同じ要素を、コピーせず見る
```

代償は**連続性（contiguity）**です。バッファ上を素直に端から端まで並べた
テンソルを**連続**（C 連続／F 連続）と呼び、転置などのビューは非連続に
なりえます。要素を端から舐めるだけの最適化された数値ルーチンや、形を
本当に作り直す `reshape` は、非連続だと使えず、その時だけ**連続コピー**
（`ascontiguousarray`）が走ります。「ふだんはゼロコピーのビュー、どうしても
必要なときだけコピー」── 文字列の章のロープや Erlang のサブバイナリと
同じ、**共有でコピーを避ける**設計がここにも貫かれています。

## ブロードキャスト：ストライド 0 という発明

形の違うテンソル同士の演算（`(3,1)` の行列に `(3,)` のベクトルを足す等）を
支える**ブロードキャスト**も、データ構造の小技で実装されます。足りない
次元や長さ 1 の次元を、**stride 0** にして相手の形へ合わせるのです。

stride 0 の次元は、「その次元をいくら進んでも**同じ要素**を読む」ことを
意味します。つまり**メモリを増やさずに、論理的に値を繰り返す**。`(3,)` の
ベクトルを `(2,3)` に足すなら、ベクトルに stride 0 の行次元を付けて
`(2,3)`・strides `[0, 1]` の**仮想的に 2 行あるビュー**にしてから、要素ごとに
足すだけです。複製は一切作りません。

```ruby
# 長さ3のベクトルを、stride 0 の行次元で (2,3) に「広げる」
v = Tensor.new([1, 2, 3], [3], [1])
bcast = Tensor.new(v.buf, [2, 3], [0, 1])  # 行を進んでも同じ3要素を読む
p [bcast[0, 2], bcast[1, 2]]   # => [3, 3]   2 行に見えるが実体は3要素
```

## 計算グラフ：微分できるデータ構造

ここからが (2) ── テンソル**演算の記録**です。機械学習は「出力の誤差を、
各パラメータで微分した勾配」を必要とします。手で導関数を書くのは非現実的
なので、処理系が**自動微分**（automatic differentiation）します。その鍵は、
**計算を、後からたどれるデータ構造として残す**ことです。

最も使われる**逆モード**（reverse mode）は、こう動きます。順方向に計算
しながら、**実行した演算を 1 つずつ記録した有向グラフ**（歴史的に
**テープ／Wengert list** と呼ぶ）を作ります。各ノードは「どの演算か・
入力ノードは誰か・勾配計算に要る中間値」を持ちます。計算が終わったら、
このグラフを**逆順にたどり**、各演算の**局所的な微分（VJP）**を掛けながら
入力へ勾配を**足し込んで**いきます。構文木の章の「木を後行順にたどって
評価」を、**逆向きに一度たどって微分**しているのです。

```ruby
# 逆モード自動微分の最小実装：演算をテープに記録し、逆順にたどる
class Var
  attr_accessor :val, :grad, :parents, :backward
  def initialize(val, parents = [], backward = ->(_g) {})
    @val, @grad, @parents, @backward = val, 0.0, parents, backward
  end

  def *(o)
    out = Var.new(@val * o.val, [self, o])
    out.backward = ->(g) { @grad += o.val * g; o.grad += @val * g }  # 積の微分
    out
  end
  def +(o)
    out = Var.new(@val + o.val, [self, o])
    out.backward = ->(g) { @grad += g; o.grad += g }                 # 和の微分
    out
  end

  def backprop
    topo, seen = [], {}                      # グラフを後行順に並べる
    visit = ->(n) { return if seen[n]; seen[n] = true
                    n.parents.each { |p| visit.(p) }; topo << n }
    visit.(self)
    @grad = 1.0
    topo.reverse_each { |n| n.backward.(n.grad) }  # 逆順に局所微分を伝播
  end
end

# z = (x * y) + x を x,y で微分する（x=3, y=4）
x = Var.new(3.0); y = Var.new(4.0)
z = (x * y) + x
z.backprop
p z.val           # => 15.0   (3*4 + 3)
p [x.grad, y.grad] # => [5.0, 3.0]   dz/dx=y+1=5, dz/dy=x=3
```

このグラフを **いつ作るか** で流儀が分かれます。**PyTorch** は順計算の
たびにグラフを動的に組み（define-by-run、上のコードがまさにこれ）、
逆伝播が済めば捨てます。初期 **TensorFlow** は先に静的なグラフを組んで
から流す方式でした。代償は**メモリ**で、逆伝播には順方向の中間値
（活性）を取っておく必要があり、これが巨大モデルのメモリを食います ──
一部を捨てて逆伝播時に再計算する**チェックポインティング**は、時間と
メモリのトレードオフの典型です。

## トレースして IR にする：JAX

**JAX** はもう一段踏み込みます。Python の関数を一度**トレース**し、テンソル
演算の列を **jaxpr** という**中間表現（IR）**として取り出すのです。構文木の
章で見た IR が、ここでは「実行を覗いて作る IR」として現れます。いったん IR
にしてしまえば、それは**ただのデータ**なので、`grad`（微分した IR を作る）・
`jit`（コンパイルする）・`vmap`（バッチ次元を足した IR に変換する）と、
**プログラム変換**を重ねられます。自動微分が「微分という名のコンパイラ
パス」になるわけで、Lisp の章で見た「コードをデータとして変換する」発想の、
数値計算版と言えます。

## 双対数：順モードのもう一つの表現

逆モードがグラフを残すのに対し、**順モード**（forward mode）は**数の表現を
拡張する**だけで微分します。**双対数**（dual number）── `a + bε`（`ε² = 0`
とする架空の単位）── を使うと、`f(a + ε) = f(a) + f'(a)ε` となり、ε の係数に
そのまま導関数が出ます。実装は「値と微分の組」を持ち演算子を多重定義する
だけ。数値の章の複素数（実部・虚部の組）とまったく同じ構造で、虚部に
「微分」を載せたものです。

```ruby
Dual = Struct.new(:re, :eps) do          # re=値, eps=微分
  def *(o) = Dual.new(re * o.re, re * o.eps + eps * o.re)  # 積の微分法則
  def +(o) = Dual.new(re + o.re, eps + o.eps)
end
# f(x) = x*x + x を x=3 で微分：x に eps=1 を載せて評価する
x = Dual.new(3.0, 1.0)
f = x * x + x
p [f.re, f.eps]   # => [12.0, 7.0]   f(3)=12, f'(3)=2x+1=7
```

逆モードはグラフ（データ構造）に残してから逆向きに解く、順モードは数の
表現を太らせて前向きに運ぶ ── 入力が多く出力が少ない学習では逆モード、
逆なら順モードが効率的で、**同じ微分を別のデータ表現で実現している**のが
面白いところです。

## まとめ

- テンソルは「一本のバッファ＋ **dtype・shape・strides・offset**」。要素位置は
  添字の一次式で出る。strides の取り方が行優先／列優先を決める。
- **ビュー**（reshape・転置・スライス・ブロードキャスト）は strides・shape・
  offset の操作だけのゼロコピー。転置は strides 入れ替え、ブロードキャストは
  **stride 0**。代償は連続性で、必要なときだけ連続コピーが走る。
- 自動微分は**計算をデータ構造に残す**。逆モードは演算のグラフ（テープ）を
  作り逆順にたどる、順モードは**双対数**で数の表現を拡張する。
- **JAX** はトレースで計算を **IR（jaxpr）** にし、微分・コンパイル・
  バッチ化を IR 変換として重ねる ── 自動微分はコンパイラパスになる。

配列の一次元の列が、strides で多次元へ、グラフで微分可能へと広がりました。
次の章では、添字ではなくキーで引くもう一つの基本コレクション ──
ハッシュへ進みます。
