# APL とその子孫：すべてが配列

## ループのない言語

APL は 1962 年、Iverson が数学の記法として設計したものから生まれた
言語です [](#cite:iverson1962)。その世界観は一つの原則に貫かれて
います —— **すべての値は多次元配列である**。スカラー（単独の数）は
0 次元の配列、ベクトルは 1 次元、行列は 2 次元。そして演算子は
すべて、配列の**全要素に一斉に**作用します。

```apl
⍝ APL: 1〜10 のうち偶数だけの合計（30）
+/ v × 0 = 2 | v ← ⍳10
```

`⍳10` が 1〜10 のベクトルを作り、`2 |` が各要素の剰余を一斉に取り、
`0 =` が**0/1 のベクトル**を作り、それを掛けて `+/` で畳み込む。
どこにもループがありません。第II部で見てきた「コレクションを 1 要素
ずつ処理する」世界とは別の星の言語に見えますが、内部のデータ構造を
覗くと、本書で学んだ部品でできていることが分かります。

## 内部表現：shape と平坦な本体

APL 系処理系の配列は、ほぼ例外なく次の二つ組です。

- **shape**（形）：各次元の長さを並べた小さな整数ベクトル。
  3×4 行列なら `[3, 4]`、スカラーなら空ベクトル。
- **ravel**（本体）：全要素を**一列に平坦に**並べた連続バッファ。

「3 行 4 列の 2 行 3 列目」は `本体[(2-1)×4 + (3-1)]` という添字
計算で引けます（配列の章の「先頭＋添字×サイズ」の多次元版です）。
この表現の妙味は、**多くの操作が本体に触れずに済む**ことです。

- **転置・reshape**：shape（とストライド）を書き換えるだけ。
  100 万要素の行列の転置が O(1)。
- **take / drop（部分配列）**：開始位置と shape の調整だけ。

NumPy の `ndarray` が「shape＋ストライド＋平坦バッファ」で、
転置やスライスをビューとして O(1) で返すのは（配列の章）、
APL のこの伝統の直系です。

## ブール配列：条件分岐をデータにする

APL のもう一つの発明は、**条件をブール（0/1）ベクトルとして
実体化する**ことです。「偶数である」は `if` 文ではなく
`0 0 1 0 1 ...` というデータになり、絞り込みは**圧縮**
（compress、ブールベクトルで選び出す演算）、数え上げは総和、
「最初に条件を満たす位置」は検索になります。

```apl
⍝ 「合格点(60以上)の人数」: if もカウンタ変数も登場しない
+/ scores ≥ 60
```

リスト・タプル・集合の章で見たビット集合の演算が、ここでは言語の
基本語彙です。そしてこの様式は現代のハードウェアと劇的に相性が
良い。分岐（CPU の苦手な予測ミスの源）が消え、連続メモリへの
一様な演算（SIMD 命令の得意技）だけが残るからです。1960 年代の
記法が 2020 年代のベクトル演算器の上で再評価されるという、
めぐり合わせの面白さがあります。

## 行優先か列優先か：レイアウトという見えない仕様

多次元配列を平坦に並べる順序には二流派あります。C・NumPy（既定）は
**行優先**（row-major、行の中身を連続に）、Fortran・MATLAB・R・
Julia は**列優先**（column-major、列の中身を連続に）。数学的には
等価ですが、**キャッシュには等価ではありません**。行優先の配列を
列方向にループすれば、1 要素ごとに離れた番地へ飛び、キャッシュ
ミスの嵐になります。「どちら向きのループが速いか」が言語の
配列レイアウトで決まる —— 論理構造と物理表現の分離（構文木の章）を
怠ると足をすくわれる、古典的な例です。

## K と q：配列が表になり、データベースになる

APL の子孫である K とその問い合わせ言語 q は、金融業界の時系列
データベース kdb+ の中核として現役です。その鍵は**テーブルの表現**に
あります。q のテーブルは「行の配列」ではなく、**列ごとのベクトルを
束ねた辞書**です。

```
trade: ([] time:09:00 09:01 09:02; price:101.2 101.5 101.3)
⍝ 実体は time ベクトルと price ベクトルの 2 本
```

「ある列だけ集計する」分析クエリでは、必要な列のベクトルだけを
端から端まで舐めればよく、行指向のデータベースのように無関係な
データを読み飛ばす無駄がありません。これは現代の列指向データベース
（columnar database）や Apache Arrow、pandas の DataFrame と同じ
構造です。R の `data.frame` も「列ベクトルのリスト」であり
（R の章で再訪します）、**「配列の束としての表」**という APL 系の
発想は、データ分析基盤の標準形になりました。

## 系譜と教訓

APL 系が本書に残す教訓を整理します。

- **データ表現を一種類に絞る**と、すべての操作がその表現への
  最適化に集中できる（shape＋平坦バッファ、ビューによる O(1) 変形）
- **制御構造をデータに変える**（ブールベクトル）と、分岐が演算に
  なり、ハードウェアの並列性に乗る
- **メモリレイアウト（行優先・列優先、行指向・列指向）は性能の
  仕様**であり、論理的な等価性に隠れてはくれない

NumPy・Julia・pandas・Arrow・GPU プログラミング —— 「配列で考える」
APL の精神は、名前を変えて今日の数値計算とデータ分析の全域を
覆っています。次の章では、まったく別の方向に振り切れた言語 ——
変数が「まだ何者でもない」論理型言語 Prolog を訪ねます。
