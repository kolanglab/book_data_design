# GIL を外す：フリースレッド CPython の解剖

並行処理の章で、「ロックを外すとは、ロックが守っていた**すべての
データ構造を作り直す**ことだ」と述べました。本章はその実例を、
一つだけ、徹底的に解剖します。CPython から GIL を取り除く
**フリースレッド化**（free threading、PEP 703 [](#cite:pep703)）です。

題材としてこれほど本書向きの事件はありません。「マルチコアを使いたい」
という要求の変化が、参照カウント・アロケータ・GC・辞書・リスト・
インラインキャッシュと、第I部と第II部で見てきた構造を**軒並み**
作り直させたからです。一つの章で本書全体を復習する、総決算として
読んでください。

## 三十年越しの宿題

GIL は 1990 年代はじめ、CPython がスレッドに対応したときに導入され
ました。撤廃の試みには長い歴史があります。

- **1996 年**：Greg Stein が内部の表を個別のロックで守る実験的
  パッチを作成。動いたものの、シングルスレッドが数十%遅くなり
  不採用に。
- **2016 年**：Larry Hastings の **Gilectomy**（GIL 切除術）。やはり
  シングルスレッド性能の壁を越えられず中断。
- **2021 年**：Sam Gross の **nogil** フォーク。後述するデータ構造の
  総取り替えによって、初めて「シングルスレッドの減速を実用域に
  抑えつつ GIL を外せる」ことを実証し、**PEP 703** として 2023 年に
  受理されました [](#cite:pep703)。
- **2024 年**：Python 3.13 が実験的なフリースレッド版ビルド
  （`python3.13t`）を同梱。**2025 年**：Python 3.14 で「実験的」の
  ラベルが外れ、正式サポートに昇格。ただし本書執筆時点（2026 年）でも
  既定のビルドではなく、移行は段階的に進められています。

30 年かかった理由は、GIL が「一個の大きなロック」ではなく、
「**無数のデータ構造が同期不要でいられるための前提**」だったこと
です。GIL が守っていたものを数え上げてみると ——参照カウンタ、
アロケータのフリーリスト、循環検出 GC の連結リスト、`dict` と `list` の
内部、メソッドや属性のキャッシュ、インターン表（識別子の章）……。
本章では、この一つひとつがどう作り直されたかを順に見ていきます。

## 参照カウントを三つに割る

最大の難所は、メモリ管理の章で見た**参照カウント**です。カウンタの
増減はあらゆる代入について回るので、これを素朴に不可分（atomic）操作に
すると、CPU のキャッシュ間でカウンタの奪い合いが起き、シングル
スレッドでも大幅に遅くなります。歴代の挑戦はみなここで沈みました。

PEP 703 の答えは、「すべてのカウンタを平等に扱うのをやめる」こと
でした。オブジェクトを観察すると、参照のされ方には偏りがあります。

**第一の偏り：ほとんどのオブジェクトは、作ったスレッドだけが触る。**
そこで **biased reference counting**（偏向参照カウント）
[](#cite:choi2018) は、カウンタを**二本に分割**します。オブジェクトの
ヘッダに所有スレッドの ID を記録し、所有スレッドからの増減は同期なしの
**ただの足し算**、他スレッドからの増減だけ不可分操作を使う、という
作戦です。

```ruby
# biased reference counting の概念実装
class BiasedCounter
  def initialize
    @owner  = Thread.current  # 生成したスレッドが「所有者」
    @local  = 1               # 所有者専用カウンタ：同期不要
    @shared = 0               # 他スレッド用カウンタ
    @mu     = Mutex.new       # 本物は CPU の不可分命令。ここでは代用
  end

  def incref
    if Thread.current.equal?(@owner)
      @local += 1                       # 速い道：ただの足し算
    else
      @mu.synchronize { @shared += 1 }  # 遅い道：同期つき
    end
  end
end
```

オブジェクトの寿命は二本の合算が 0 になったとき尽きます（実際には
所有スレッドが手を引くときにカウンタを統合する手続きがあります）。
「**よくある場合を速い道に、まれな場合を遅い道に**」—— JIT の章の
投機と同じ発想が、カウンタ一個のレベルで使われているのです。

> [!NOTE]
> 「カウンタを触るたびに毎回スレッド判定をするのは、それ自体が
> 高くつかないか？」と思うかもしれません。これが安いのです。
> 所有スレッドの ID はオブジェクトヘッダの、**カウンタの隣**に
> 置かれています —— カウンタを増減する以上どのみち読むキャッシュ
> ラインです。比べる相手の「自分の ID」は、多くの環境でスレッド
> ポインタ用レジスタから一命令で読めます。そして分岐はほぼ常に
> 「所有者だった」側に倒れるので、分岐予測が当たり続けます。
> つまり判定の実費は数サイクル。対して不可分操作は空いていても
> 数十サイクル、キャッシュラインの取り合いになれば数百サイクルに
> 跳ねます。数サイクル払って数十〜数百サイクルを回避する、割の
> よい賭けです。とはいえ GIL 版の「ただの足し算」よりは確実に
> 重く、この差が後述するシングルスレッド性能の税金の一部に
> なっています。

**第二の偏り：`None` や `True` は全スレッドが絶えず触る。** 二本に
割っても、世界中から参照されるオブジェクトのカウンタは奪い合いに
なります。そこで **immortal objects**（不死オブジェクト、PEP 683
[](#cite:pep683)）は、`None`・`True`・`False`・小さな整数・インターン
済み文字列（値の表現の章、識別子の章の常連たちです）のカウンタを
**特別な値に固定**し、増減の関数がその値を見たら**何もしない**ように
しました。どうせ死なないオブジェクトは、数えるだけ無駄だからです。
副産物として、`fork` した子プロセスがこれらのページに書き込まなく
なるため、OS のコピーオンライト共有が保たれるという利点もあります
（プリフォーク型サーバでのメモリ削減として、GIL とは独立に
Python 3.12 で先行導入されました）。

**第三の偏り：関数・クラス・モジュールは、頻繁に参照されるが
めったに死なない。** これらには **deferred reference counting**
（遅延参照カウント）が使われます。インタプリタのスタックからの
参照はカウントせず、正確な生死の判定は GC のときにまとめて行う ——
「即時回収」という参照カウントの建前を、一部のオブジェクトについて
放棄する判断です。

一本のカウンタが、偏向・不死・遅延の三種に分かれました。一様で
美しい設計が、並行という要求の前で**場合分けの塊**になる —— その
場合分けの一つひとつが、観測された偏りに対応しているのです。

## GC を作り直す：連結リストを捨てる

メモリ管理の章で、CPython は循環参照を回収するため、コンテナ系
オブジェクトを**双方向連結リスト**でつないでいる、と述べました。
この設計はフリースレッド化と衝突します。オブジェクトの生成・破棄の
たびにグローバルなリストへの挿入・削除が起きるので、スレッドが
増えるとリストの操作自体が同期の集中点になるのです。

フリースレッド版の答えは思い切っています。**リストを捨てました**。
代わりにアロケータを **mimalloc** [](#cite:leijen2019) に統一し、循環
検出の際は mimalloc が管理するページのメタデータを走査して「ヒープ上の
全オブジェクト」を列挙します。アロケータはどのみち「どこに何を割り
当てたか」を知っているのだから、**アロケータの台帳に GC の台帳を
兼ねさせる**わけです。オブジェクトごとに 2 ワードあった連結リスト用
ヘッダも不要になりました。循環検出そのものは、全スレッドを一旦
停止させて（stop-the-world）行います。

## dict と list を作り直す：ロックと楽観のあわせ技

ハッシュの章で見た `dict`、配列型の章で見た `list` は、GIL の下では
「操作の途中で他スレッドに割り込まれない」ことを暗黙の前提にして
いました。フリースレッド版では、各オブジェクトのヘッダに小さな
**オブジェクト単位ロック**（1 バイトのミューテックス）が追加され、
書き込み操作はこれを取って行います。並行処理の章で見た
`ConcurrentHashMap` の細粒度化（表全体 → 区画 → バケット）と同じ
方向の、「処理系全体 → オブジェクト 1 個」という細粒度化です。

しかし読み取りのたびにロックを取っていては、今度は読み取りが
遅くなります。そこで読み取りは**楽観的**（optimistic）に行います。
ロックを取らずに読み、「読んでいる間に書き換えが起きていないか」を
後から確認して、起きていたらやり直すのです。

```ruby
# 楽観的読み取りの骨格：版数を読む → データを読む → 版数を確かめる
def optimistic_read(table)
  loop do
    v1 = table.version            # 書き込みのたびに増える版数
    value = table.lookup_unsafe   # ロックなしで読む
    return value if table.version == v1   # 変わっていない：読めた値は正しい
    # 変わっていた：書き込みと交錯したのでやり直し
    # （本物は数回失敗したらロック取得に切り替える）
  end
end
```

## 解放はみんなの通過を待ってから：QSBR

楽観的読み取りには、やり直しでは済まない問題が一つ残っています。
ロックなしで表を読んでいる最中に、別のスレッドが再ハッシュして
**古いバケット配列を解放**したら？ 版数の確認にたどり着く前に、
読み手は解放済みメモリに触れて落ちてしまいます。

参照カウントは助けてくれません。バケット配列は Python の
オブジェクトではない処理系内部のメモリで、そもそもカウンタを
持たないからです。では持たせればよいかというと、読むたびに共有
カウンタへ書き込むのでは本末転倒です —— biased reference counting が
苦労して避けたのは、まさにこの「読むだけなのに書く」でした。

フリースレッド版 CPython の答えが **QSBR**（quiescent-state-based
reclamation、静止状態に基づく回収）です。発想を一言で言えば、
「**解放を、全員が読み終わったと確信できるまで遅らせる**」。

1. 書き手は古い配列をすぐには解放せず、**現在の世代番号つきで
   墓場リストに置く**だけにする。
2. 各スレッドは、「いま楽観的読み取りの途中ではあり得ない」と
   言える切りのいい地点 —— インタプリタループの境界や、ブロック
   するシステムコールの前後 —— を通過するたびに**検問所に申告**
   する（これが**静止状態**です）。
3. 墓場の配列は、**全スレッドが、それが置かれた時刻より後に検問を
   通過したら**、本当に解放する。読み取りの途中のスレッドは定義
   により検問を通過していないので、足元のメモリが消えることは
   ありません。

```ruby
# QSBR の概念実装：「保留された free」の台帳
class QSBR
  def initialize = (@now = 0; @passed = Hash.new(0); @graveyard = [])

  def quiescent!(thread)        # 各スレッドが切りのいい所で申告
    @passed[thread] = @now
  end

  def retire(buffer)            # free の代わりに墓場へ（時刻つき）
    @graveyard << [@now += 1, buffer]
  end

  def reclaim!(threads)         # 全員が追い越した分だけ本当に解放
    horizon = threads.map { @passed[_1] }.min
    done, @graveyard = @graveyard.partition { |t,| t <= horizon }
    done.each { |_, buf| buf.really_free }
  end
end
```

代償は二つあります。解放が遅れるぶんメモリが膨らむこと。そして
**一人でも検問に来ないスレッドがいると、全員の墓場が溜まり続ける**
ことです（長い計算や C 呼び出しに入ったきりのスレッドが典型で、
CPython はブロックしそうな場面でスレッドが「離席中」を申告して
おく仕組みで緩和しています）。

並行処理の章の最後で、「GC のない言語はロックフリー構造のために
エポックベース回収やハザードポインタといった**ミニ GC** を自作する」
と述べました。Linux カーネルの RCU も同族です。そして参照カウント
という立派な自動メモリ管理を持つはずの CPython さえ、ロックを
外した途端、**カウンタの届かない内部メモリのため**にこのミニ GC を
内蔵する羽目になりました。「生きているとは何か」という GC の問いは、
どこまでも追いかけてくるのです。

## キャッシュを作り直す：投機はスレッドと相性が悪い

構文木の章と JIT の章で見たとおり、CPython 3.11 以降の高速化の柱は
**特殊化適応型インタプリタ**（PEP 659 [](#cite:pep659)）でした。実行
しながらバイトコードを書き換え、インラインキャッシュに観測結果を
書き込む —— つまり**コード自体が実行時に変化する可変データ構造**です。
シングルスレッドなら無害なこの自己書き換えは、複数スレッドが同じ
バイトコードを同時に実行・書き換えすると競合します。

このため 3.13 のフリースレッド版では特殊化が**丸ごと無効化**され、
それが当初の大きな減速の一因になりました。3.14 での復活の手口は、
ロックでも不可分操作でもありません。**スレッドローカル・バイト
コード**（thread-local bytecode）—— 書き換えの対象であるバイト
コードを、インラインキャッシュごと**実行するスレッドの数だけ複製
してしまう**のです。各スレッドは自分の複製を、自分の観測に基づいて
自由に書き換えます。書き手が一人しかいない可変データ構造に競合は
ありません。並行処理の章の三戦略で言えば、「守る」のが難しかった
データ構造を「**複製する**」に乗り換えたわけです。スレッドごとに
観測される型が違えば、同じ関数がスレッドごとに**別の特殊化**を
持つことさえあります。代償はもちろんメモリで、複製は「そのコードを
実行したスレッドの数」だけ増えます。膨らみが問題になる場合に複製を
止めるオプション（`-X tlbc=0`）も用意されていますが、それは特殊化
ごと手放すという意味になります。

## 最大の敵は互換性

ここまでは CPython の**内側**の作り直しでした。しかし最大の難関は
外側、**C 拡張**です。NumPy をはじめとする膨大な C 拡張は、GIL の
存在を暗黙の前提に書かれてきました。「GIL を持っている間は誰も
割り込まない」という性質が、C 拡張にとっての無料のロックだった
のです。

API の規約にも前提が染み込んでいます。たとえば従来の
`PyDict_GetItem` は、カウンタを増やさない**借用参照**（borrowed
reference）を返します。GIL の下では「借りている間に消されることは
ない」と言えましたが、フリースレッドの世界では、借りた瞬間に別の
スレッドが辞書から削除して、参照先が解放されるかもしれません。
このため、強い参照を返す `PyDict_GetItemRef` などの新 API が追加され、
拡張側の書き直しが進められています。フリースレッド版が別の ABI
（バイナリ互換性の単位）になっているため、エコシステム全体が
ライブラリを二種類ビルドして配る移行期が続いています。

直列化の章で「フォーマットは一度公開したら互換性に縛られる」と
述べましたが、同じことがここでも起きています。データ構造の作り直しは
処理系の内側で完結しません。**その構造を前提に積み上がった外部世界
ごと**作り直しになる —— GIL 撤廃に 30 年かかった本当の理由はこれです。

## 代償と、もう一つの道

これだけの大工事を経ても、タダにはなりませんでした。三分割された
参照カウント、オブジェクトごとのロック、楽観的読み取りのやり直し ——
それぞれがわずかなコストを持ち、フリースレッド版のシングルスレッド
性能は通常版より数%〜1 割ほど遅くなります（特殊化が無効だった
3.13 ではさらに大きな差でした）。マルチコアの並列性と引き換えに
全員が少しずつ払う税金をどこまで圧縮できるかが、既定ビルド昇格の
鍵になっています。

同じ問いに、CRuby は別の答えを選んだのでした。並行処理の章で見た
**Ractor** —— GVL に相当するロックを実行単位ごとに分け、共有できる
オブジェクトを不変なものに制限することで、内部データ構造の
シングルスレッド前提を（Ractor の中では）保ったまま並列実行を可能に
する道です。三つの戦略で言えば、CPython は「守る」の徹底
（細粒度化）を、CRuby は「複製・隔離する」を選んだことになります。
どちらが優れているかは一概に言えず、C 拡張の量、典型的な
ワークロード、互換性への許容度といった**エコシステム側の事情**が
答えを左右します。

なお、Java や Go のように最初から「スレッドはいるもの」として設計
された処理系には、この大工事は存在しません。参照カウントではなく
トレーシング GC を選び、コレクションの同期方針を最初に決めて
あったからです。**最初の設計で何を仮定するか**が、30 年後の工事費を
決めるのです。

## まとめ：一個のロックの中身

GIL が守っていたものと、その作り直しを並べます。

| GIL が守っていたもの | フリースレッド版の答え |
|---|---|
| 参照カウンタ | 偏向（二本立て）・不死・遅延の三分割 |
| 循環検出 GC の連結リスト | アロケータ（mimalloc）の台帳を GC が間借り |
| `dict`・`list` の内部 | オブジェクト単位ロック＋楽観的読み取り＋QSBR |
| インラインキャッシュ | バイトコードごとスレッドローカルに複製 |
| C 拡張の暗黙の前提 | 借用参照を返さない新 API と新 ABI への移行 |

「処理系を一個のロックで守る」という一行の決定の裏に、これだけの
データ構造が依存していました。そして要求が変わったとき、変更は
一行では済まず、構造の総取り替えになりました。本書の主張 ——
**要求が構造を決め、構造が性能と互換性を決める** —— の、これ以上
ない実例です。

即値とヒープオブジェクト、連続配列とロープ、可変ハッシュと HAMT、
そしてロックと不変性 —— 主流言語の世界の旅はここまでです。
続く第III部では少し足を伸ばし、独自の世界観を持つ言語たち ——
すべてが配列の APL、束縛を取り消せる Prolog、フレームすら
オブジェクトの Smalltalk たち —— のデータ構造を訪ねます。
主流の設計がなぜ主流なのかは、外から見たときに一番よく分かるのです。
