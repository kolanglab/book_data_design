# 時間型：暦と時計をデータにする

## 一見ありふれたデータ型の、底なしの難しさ

「現在時刻」や「日付」を扱う時間型は、ほぼすべての言語に備わっています。
そして、標準ライブラリの中で**最も設計のやり直しが多い型**でもあります。
Java は `Date` → `Calendar` → `java.time` と三世代を作り直し、
JavaScript は `Date` の失敗を 25 年引きずった末に新 API（Temporal）を
標準化し（TC39 Stage 4。2026 年半ば時点で Firefox・Chrome・Node.js が
実装済み、他のエンジンは追従中）[](#cite:temporal)、Python の
`datetime` には後から `fold` という謎の属性が生えました
[](#cite:pep495)。なぜこれほど難しいのか。
本章では、時間型を**複数のデータ構造の積み重ね**として解剖します。

## 素朴な構造体は、なぜだめか

素朴には「年・月・日・時・分・秒」をフィールドに持つ構造体を
思いつきます。人間の読み書きには便利ですが、**計算**がひどく面倒です。
「3 日後」を求めるには月またぎ・月末の日数（28〜31 日）・うるう年を
場合分けし、「二つの時刻の差」は全フィールドの繰り下がり付き減算に
なります。比較ですら自明ではありません。

そこで多くの処理系は、時刻を**ある基準時点からの経過時間**、すなわち
**一本の数直線上の数**として表します。広く使われる基準は
「**1970 年 1 月 1 日 0 時 0 分 0 秒（協定世界時）**」で、ここからの
経過秒数を **Unix 時刻**（Unix time、エポック秒）と呼びます。

```ruby
t = Time.now
p t.to_i           # => 経過秒数（例: 1781243696）
p (t + 3 * 86400)  # => 3日後（86400 = 1日の秒数を足すだけ）
```

時刻の差は引き算、前後関係は大小比較、「3 日後」は `86400 × 3` を
足すだけ。**複雑な暦の構造を、計算に強い単純な数へ押し込める**——
これが時間型の第一の設計判断です。「年月日時分秒」は、表示のときに
だけ数直線から**変換して作る**見かけの姿、と役割が逆転します。

## 何ビットで、どの精度で数えるか

数直線で表すと決めたら、次は**幅と精度**です。これは数値型の章の
表現範囲の議論が、そのまま時間に応用される場面です。

**32 ビット秒の限界（2038 年問題）**。C の `time_t` を 32 ビット符号付き
整数で実装したシステムは、1970 年から約 68 年後 —— **2038 年 1 月 19 日**
に表現範囲を使い切り、時刻が 1901 年へ巻き戻ります。組み込み機器や
ファイルフォーマットに埋まった 32 ビット時刻は今も残っており、
「整数の幅」という地味な選択が数十年後の社会問題になった実例です。

**ミリ秒を浮動小数点数で（JavaScript）**。JS の `Date` はエポックからの
**ミリ秒を double** で持ちます。数値型の章で見たとおり double の整数
精度は 2⁵³ までなので、±28 万年ほどは 1ms 単位で正確 —— 範囲は十分
ですが、**精度がミリ秒で頭打ち**という制約が言語に焼き付きました。
高精度計測のために `performance.now()` が別途用意されたのは
このためです。

**秒＋ナノ秒の二本立て**。現代の主流は、POSIX の `timespec` に
ならった「秒（64 ビット整数）＋ナノ秒（0〜999,999,999 の整数）」の
**ペア**です。Java の `Instant`、Go の `time.Time`、Rust の
`Duration`/`Instant` がこの族です。「秒を double で持てば 1 本で
済むのに」と思うかもしれませんが、double の精度は値の大きさに依存
します。現在のエポック秒（約 17.8 億 ≈ 2³⁰·⁷）では刻みが
2⁻²² 秒 ≈ 0.24 マイクロ秒となり、**ナノ秒にはまったく届きません**。
「いまに近いほど精度が要る」時刻という量に、「0 に近いほど精度が
高い」浮動小数点数は根本的にミスマッチなのです。

**Ruby の任意精度**。CRuby の `Time` はさらに進んで、内部を整数または
**有理数**（数値型の章の `Rational`）で持ち、ナノ秒どころか任意精度の
秒未満を正確に表せます。`Time.at(1, 1, :nanosecond)` のような生成も、
有理数演算として誤差なく処理されます。「時刻の演算で丸め誤差を出さない」
ことを数値表現の選択で保証した、Ruby らしい割り切りです。

> [!IMPORTANT]
> 時間型の中身は、結局のところ数値型です。整数の幅・浮動小数点数の
> 精度分布・有理数の正確さ —— 数値表現の知識（数値型の章）が、
> そのまま時間型の設計品質を決めます。「時間のバグ」の多くは、
> 実は「数値表現のバグ」なのです。

## 二種類の時計：壁時計と単調時計

ここまでの「数直線」には、隠れた落とし穴があります。**壁時計**
（wall clock、現実世界の時刻を指す時計）は、**巻き戻ることがある**の
です。NTP（ネットワーク経由の時刻合わせ）が進みすぎた時計を戻す、
管理者が時刻を設定し直す、うるう秒の調整が入る —— その瞬間、

```ruby
t1 = Time.now
do_something
elapsed = Time.now - t1   # ← 時計が戻されていたら、負になりうる！
```

のような経過時間の測定が壊れます。タイムアウト処理が永遠に発火しない、
ベンチマークが負の時間を報告する、といった事故は実際に起きてきました。

そこで OS と言語は、もう一本の時計を持ちます。**単調時計**（monotonic
clock）—— 起点はてきとう（起動時など）だが、**決して巻き戻らない**ことだけ
を保証する時計です。経過時間の測定はこちらで行うのが正解です。

```ruby
t1 = Process.clock_gettime(Process::CLOCK_MONOTONIC)
do_something
elapsed = Process.clock_gettime(Process::CLOCK_MONOTONIC) - t1  # 常に正
```

面白いのは Go の設計です。Go 1.9 から、`time.Now()` が返す `time.Time` は
**壁時計とモノトニック時計の両方の読みを 1 つの構造体に同居**させて
います [](#cite:cox2017)。時刻の引き算ではモノトニック部分が自動で
使われ、表示には壁時計部分が使われる。「二種類の時計の使い分け」と
いう人間が間違えやすい判断を、**データ構造に両方持たせることで自動化**
したわけです。

## うるう秒：仕様の歪みをどこで吸収するか

地球の自転は一定ではないので、原子時計ベースの世界時（UTC）には
ときどき**うるう秒**（leap second、23:59:60 という 61 秒目）が挿入
されます。ところが Unix 時刻は「**1 日は必ず 86400 秒**」と定義されて
おり、うるう秒を**なかったことにします**（挿入の瞬間、同じ Unix 時刻が
2 秒間続く）。ほとんどのソフトウェアはこの嘘の上で動いています。

この嘘が許せない場面（うるう秒の瞬間にも単調・連続であってほしい
分散システム）のために、Google や AWS は**うるう秒スメア**（leap
smear）—— うるう秒前後の十数時間、時計をわずかに遅く進めて 1 秒を
塗り広げる —— を運用しています。データ構造の話に見えませんが、
「時刻の数直線と現実の地球の回転のずれを、**どの層の誰が吸収するか**」
という設計問題であり、時間型の API（うるう秒を表せるか？ `60` 秒目を
パースできるか？）に波及します。

## タイムゾーン：数直線から人間の時刻へ

同じ瞬間でも、東京では朝、ロンドンでは真夜中です。内部の数直線
（UTC 基準）から人間向けの「年月日時分秒」へ変換するには、
**タイムゾーン**（time zone、時間帯）の情報が必要です。ここが時間型
最大の難所です。なぜなら、タイムゾーンは物理ではなく**政治**だから
です。国は予告なく夏時間を導入・廃止し、標準時をずらします。

事実上の世界標準が **IANA タイムゾーンデータベース**（tzdata、
通称 Olson database）です [](#cite:tzdb)。`Asia/Tokyo` のような地域名
ごとに、**「この瞬間から UTC との差がこう変わった」という遷移の履歴**を
列挙したデータベースで、年に数回更新され、世界中の OS・言語処理系に
配布されています。

データ構造として見ると、各ゾーンは

- **遷移時刻の整列済み配列**（いつ切り替わったか）と、
- 各区間の**オフセット・略称・夏時間フラグ**、
- 将来の夏時間を予測する**規則**（「3 月の第 2 日曜に +1 時間」など）

からなります。「Unix 時刻 t のときのローカル時刻は？」という問いは、
**遷移配列の二分探索**に帰着します。

```ruby
# 概念図：ゾーン情報の中身と、時刻→オフセットの解決
Zone = Struct.new(:transitions, :offsets)
tokyo = Zone.new(
  [-2587712400, -683802000, -672310800, ...],  # 遷移時刻（エポック秒、昇順）
  [[33539, "LMT"], [32400, "JST"], [36000, "JDT"], ...]  # 各区間の情報
)

def utc_offset_at(zone, t)
  # 「t 以下で最大の遷移時刻」を二分探索 —— 平衡木の章の近傍検索
  i = zone.transitions.bsearch_index { |tr| tr > t }
  case i
  when 0   then zone.offsets[0]      # 最初の遷移より前（既定の区間）
  when nil then zone.offsets[-1]     # 最後の遷移より後
  else          zone.offsets[i - 1]
  end
end
```

シンボルテーブルの章で見た「整列配列＋二分探索」が、こんなところで
現役なのです。配布形式（TZif バイナリ）もほぼこの 2 本の配列の
直列化で、遷移は数十〜百件程度。一度引いたゾーンをキャッシュすれば、
時刻変換のコストは実質この二分探索だけです。

```ruby
# 同じ瞬間を異なるオフセットで表示する（Ruby 2.6+ の Time の in:）
t = Time.at(1780890896, in: "+09:00")
p t  # => 2026-06-08 12:54:56 +0900
p Time.at(1780890896, in: "+01:00")  # => 2026-06-08 04:54:56 +0100
```

ただし、組み込みの `Time` の `in:` に渡せる文字列は `"+09:00"` の
ような**固定オフセット**だけで、`"Asia/Tokyo"` のような**ゾーン名**は
受け付けません（`ArgumentError` になります）。ゾーン名で扱うには、
前述の TZif データベースを引いてくれるライブラリ（tzinfo gem など）が
返すタイムゾーンオブジェクトを `in:` に渡します。標準ライブラリが
オフセットまでしか面倒を見ない —— この線引き自体が、次に述べる
「オフセットとゾーンは別物」という区別の現れです。

> [!WARNING]
> 「タイムゾーン＝UTC からの固定オフセット」という理解は間違いです。
> `+09:00` という**オフセット**は変換結果の一断面にすぎず、
> `Asia/Tokyo` という**ゾーン**は遷移履歴と将来規則を含む別物です。
> 「日本は夏時間がないから同じでしょ」と思っても、過去（1948〜51 年）の
> 日本には夏時間がありました。過去の時刻を正しく扱うには、オフセット
> ではなくゾーン名で持つ必要があります。

## 存在しない時刻、二度ある時刻

夏時間（DST、daylight saving time）の切り替えは、ローカル時刻の
数直線に**穴と重複**を作ります。時計を 1 時間進める春には、その地域に
「2 時 30 分」が**存在しない**日が生まれ、戻す秋には「1 時 30 分」が
**二度ある**日が生まれます。

つまり「ローカルの年月日時分秒 → Unix 時刻」という逆変換は、
**部分関数かつ多価関数**なのです。各言語はこの数学的事実への対応を
迫られました。Python は PEP 495 で `datetime` に **`fold`** という
属性（二度ある時刻の 1 回目か 2 回目かを示す 1 ビット）を追加し
[](#cite:pep495)、Java の `ZonedDateTime` は「重複時は前のオフセットを
優先、欠落時はギャップ分ずらす」という解決規則を仕様化しました。
たかが 1 ビット、されど 1 ビット —— **データ構造に 1 ビット足すことで
しか表せない現実**があるのです。

## 型を分けるという到達点：java.time の設計

歴代の失敗を踏まえて設計された Java 8 の `java.time`（JSR-310）は、
「時間」を**意味の異なる複数の型に分解**しました。現在の時間 API 設計の
事実上の模範解答なので、構造を見ておきましょう。

| 型 | 中身 | 意味 |
|---|---|---|
| `Instant` | 秒＋ナノ秒 | 数直線上の一点（ゾーン無関係） |
| `LocalDate` | 年・月・日 | 「2026-06-08」という暦の日付（時刻なし） |
| `LocalDateTime` | 年月日＋時分秒 | ゾーン未定の「壁の時刻」 |
| `ZonedDateTime` | LocalDateTime＋ゾーン | 特定地域の時刻（変換規則込み） |
| `Duration` | 秒＋ナノ秒 | 物理的な経過時間（「90 分」） |
| `Period` | 年・月・日の組 | 暦の上での期間（「1 ヶ月」） |

注目すべきは `Duration` と `Period` の分離です。「1 ヶ月後」は
**物理的な長さではありません**。1 月 31 日の「1 ヶ月後」は 2 月 28 日
（うるう年なら 29 日）—— 月末への切り詰めという**暦の演算**であり、
2,678,400 秒を足す演算とは別物です。素朴な API はこの二つを混同して
「3 月 3 日」を返したり、夏時間の日に「1 日後」が 23 時間後になったり
します。**意味の違いを型の違いに昇格させる** —— 静的型付けの思想が
時間という題材で最も説得力を持つ瞬間です。

```ruby
# Ruby でも Date と Time は別の型で、暦の演算は Date 側にある
require "date"
p Date.new(2026, 1, 31) >> 1   # => 2026-02-28 （月単位の演算、切り詰めあり）
p Date.new(2026, 1, 31) + 30   # => 2026-03-02 （日数の加算）
```

## 暦そのものもデータである：ユリウス通日

`Date` のような暦の日付型は、内部でも「年・月・日」を持っているとは
限りません。CRuby の `Date` は内部表現として**ユリウス通日**（Julian
Day Number、紀元前 4713 年 1 月 1 日からの通し日数）を使います。
日付を一個の整数にしてしまえば、日数の加算・差・曜日（7 で割った余り）が
ただの整数演算になり、グレゴリオ暦・ユリウス暦・和暦といった**複数の
暦法**も「通日 ↔ 各暦」の変換関数の問題に分離できます。エポック秒の
「数直線化」とまったく同じ発想が、日付のレイヤーでも繰り返されている
わけです。この種の暦変換アルゴリズムの数学は Knuth も論じています
[](#cite:knuth1997)。

変換そのものは、整数演算だけの短い関数です（Fliegel–Van Flandern の
公式として知られる形）。「1・2 月を前年の 13・14 月とみなす」ことで
うるう日を年末に追いやり、「3 月始まりの月の長さ 31,30,31,30,31… を
`(153m+2)/5` という一次式で合計する」のが急所です。

```ruby
# 年月日 → ユリウス通日（グレゴリオ暦）
def civil_to_jdn(y, m, d)
  a  = (14 - m) / 12          # 1,2月なら 1（前年の13,14月として扱う）
  yy = y + 4800 - a
  mm = m + 12 * a - 3
  d + (153 * mm + 2) / 5 + 365 * yy +
    yy / 4 - yy / 100 + yy / 400 - 32045   # うるう年規則がここに見える
end

# ユリウス通日 → 年月日（逆変換）
def jdn_to_civil(j)
  a = j + 32044
  b = (4 * a + 3) / 146097    # 146097 = 400年の総日数
  c = a - 146097 * b / 4
  d_ = (4 * c + 3) / 1461     # 1461 = 4年の総日数
  e = c - 1461 * d_ / 4
  m_ = (5 * e + 2) / 153
  [100 * b + d_ - 4800 + m_ / 10,
   m_ + 3 - 12 * (m_ / 10),
   e - (153 * m_ + 2) / 5 + 1]
end

p civil_to_jdn(2026, 6, 8)               # => 2461200（Date#jd と一致）
p jdn_to_civil(civil_to_jdn(2000, 2, 29)) # => [2000, 2, 29] 往復できる
```

`yy/4 - yy/100 + yy/400` の三項に、グレゴリオ暦のうるう年規則
（4 で割り切れる年、ただし 100 の倍数は除く、ただし 400 の倍数は
含む）がそのまま埋まっています。`Date.new(2026,6,8).jd` と
突き合わせれば検算できます。ただし古い日付で検算するときは
`Date.new(y, m, d, Date::GREGORIAN)` と暦を明示すること ——
Ruby の `Date` は既定で 1582 年の改暦（ユリウス暦→グレゴリオ暦の
切り替え、10 日間の「存在しない日付」つき）を再現するため、
それ以前はこの公式（**先発グレゴリオ暦**、proleptic Gregorian ——
改暦前にもグレゴリオ暦を遡って適用する流儀）とずれるのです。
「どの暦で解釈するか」までがデータの一部、という時間型らしい
落とし穴です。

## 文字列化と交換：ISO 8601

時間型はシステム間でやり取りされるため、**文字列表現**も設計の一部です。
事実上の標準は ISO 8601 / RFC 3339 [](#cite:rfc3339) の
`2026-06-08T12:54:56+09:00` 形式で、(1) 桁が固定で、(2) **文字列の
辞書順がそのまま時刻順になる**よう設計されています。ただし (2) が
成り立つのは、**オフセットと精度（小数秒の桁数）を揃えた場合だけ**です。
オフセットが混在すると、`2026-06-08T09:00:00+09:00` は
`2026-06-08T00:30:00Z` より 30 分**前**の瞬間なのに、文字列としては
**後**に並んでしまいます。ログのファイル名やデータベースのキーには、
UTC（末尾 `Z`）に正規化し桁数を固定した形で使うのが定石で、そうすれば
文字列ソートが時刻ソートを兼ねる —— 表現の設計が、それを格納する
データ構造（整列、索引）にまで波及する好例です。

> [!TIP]
> 「未来の予定」を保存するときは、エポック秒ではなく**ローカル時刻＋
> ゾーン名**で持つのが定石です。「2030 年 3 月 10 日 9:00 @ 東京」を
> いまエポック秒に変換して保存すると、その日までに日本が夏時間を導入
> したら**間違った瞬間**を指してしまいます。過去の確定した瞬間は
> エポック秒、未来の暦上の約束はローカル＋ゾーン。同じ「時刻」でも
> 意味によって最適な表現が違う —— 本書を貫くテーマそのものです。

## まとめ：時間型は表現の積層

時間型を分解すると、こうなります。

- **物理の瞬間**：64 ビット秒＋ナノ秒（または任意精度有理数）の数直線
- **経過時間の測定**：巻き戻らない単調時計（壁時計と分離、Go は同居）
- **人間の時刻**：数直線 ↔ 暦の変換層（tzdata の遷移配列＋二分探索）
- **暦の演算**：通日という第二の数直線と、月・年の非一様な演算
- **交換表現**：辞書順＝時刻順の文字列形式

一見「ただの数値が 1 個入った型」に見える `Time` の中に、これだけの
層が積み重なっています。そしてどの層にも、本書で学んだ道具 ——
整数の幅、浮動小数点数の精度、有理数、整列配列と二分探索、型による
意味の分離 —— が顔を出します。時間型は、データ構造設計の総合演習
問題なのです。

次の章では、もう一つの「内部に複雑な機械を抱えた型」——
**正規表現**を解剖します。
