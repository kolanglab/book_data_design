# おわりに：表現を選ぶということ

## 本書を貫いた一つの教訓

三部にわたる旅でした。第I部では処理系の**内部**を支える構造 ——
名前の表、識別子のインターン、構文木と中間表現、値のビット表現、
スタックとフレーム、GC、JIT の台帳たち、そしてプログラムを組み立てる
ロードの仕組み。第II部では言語が
利用者に**提供**するデータ型たち —— 数値、文字列、配列、ハッシュ、リストと
集合、平衡木、オブジェクト、クロージャ、サンク、時間、正規表現、
直列化、並行のための構造。第III部では個性派言語たちが独自の要求から生んだ
構造を訪ねました。

全体を貫く教訓は、次の一文に尽きます。

> **同じ「データ型」に唯一の正しい実装はない。要求 —— 速度・メモリ・順序・
> 安全・並行性・互換性 —— に応じて表現を選び、ときに切り替え、
> ときにキャッシュする。**

即値とヒープオブジェクト。連続配列とロープ。可変ハッシュと HAMT。
バックトラックとオートマトン。壁時計と単調時計。ロックと不変性。
本書に登場した対立項はすべて、トレードオフの両端でした。そして
優れた処理系は、どちらか一方を選ぶのではなく、**両方を実装して
見えないところで切り替える**ことが多い、というのも繰り返し見た
とおりです（Fixnum と Bignum、埋め込み文字列とヒープ文字列、
ar_table と st_table、単相と多相のインラインキャッシュ）。

## 繰り返し現れた設計パターン

言語も時代も違う処理系たちが、同じ形の解に何度も到達していました。
名前を付けて持ち帰ってもらえるよう、主なものを再掲します。

- **タグとニッチ**：使われないビットパターンに情報を埋める
  （タグ付きポインタ、NaN ボクシング、Rust の `Option<&T>`、R の NA）
- **名前は早く番号にする**：実行時に文字列を引かない
  （ID、スロット番号、vtable のオフセット、定数プール）
- **同じ形は共有する**：構造の地図を一枚にして全員で使う
  （インターン、シェイプ、キー共有辞書、構造共有の永続データ構造）
- **観測してからキャッシュする**：繰り返しに賭け、無効化を設計する
  （インラインキャッシュ、コードレンジ、メソッドキャッシュと世代番号）
- **遅延し、必要なら記憶する**：作るのを先送りする
  （サンク、ConsString、Tcl の二重表現、ALTREP、コピーオンライト）
- **二段構えの篩**：軽い検査で大半を振り落としてから重い処理へ
  （SwissTable のメタデータ、正規表現のリテラル先回り）
- **寿命で分ける**：いっしょに死ぬものはまとめて管理する
  （アリーナ、世代別 GC、スコープのスタック）

## ここからの読み方

本書は「気持ちをつかむ」ことを優先し、実装の細部は各処理系の
ソースコードに譲りました。幸い、本書で名前を挙げた構造はすべて、
公開されたコードの中に実物があります。CRuby の `st.c`（ハッシュ）、
`gc.c`（GC）、`shape.c`（シェイプ）。CPython の `dictobject.c` と
`listobject.c`。V8 と GHC と BEAM の設計文書。読み始めれば、
「あの章のあれだ」という再会が必ずあるはずです。

データ構造は、言語処理系という建物の鉄骨です。表からは見えませんが、
建物の高さも形も寿命も、結局は鉄骨が決めます。あなたが次に
`h[:key]` と書くとき、その一行の下で動く何層もの構造を思い出して
もらえたら —— そして いつか自分の処理系の鉄骨を組みたくなったら、
本書の役目は果たされたことになります。

さらに学ぶための道しるべを、付録にまとめました。
