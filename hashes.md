# ハッシュ（辞書）型：キーで引く表

## キーで引く、という要求

配列は「0 番目、1 番目…」と整数の添字で引きました。しかし現実には、
「名前 `"Alice"` の年齢は？」「設定項目 `:port` の値は？」のように、
**任意の値をキーにして引きたい**場面が大半です。これを実現するのが
**ハッシュ**（hash、連想配列・辞書とも呼ぶ）型です。Ruby の `Hash`、
Python の `dict`、JavaScript の `Map` がこれにあたります。

```ruby
ages = { "Alice" => 30, "Bob" => 25 }
p ages["Alice"]   # => 30
ages["Carol"] = 28
```

ハッシュの理想は、キーが文字列でもオブジェクトでも、**平均 O(1)** で
読み書きできることです。第2章のシンボルテーブルでハッシュ表の骨格は
すでに見ました。この章では、実用のハッシュを支える設計判断を掘り下げます。

## ハッシュ関数とバケット

ハッシュ型の心臓は**ハッシュ関数**（hash function）です。これは任意のキーを
固定範囲の整数（ハッシュ値）に変換する関数で、その整数を配列（バケットの列）の
添字に使います。良いハッシュ関数は、異なるキーをできるだけ**均等に**
散らばらせます。偏ると衝突が増え、O(1) の理想が崩れるからです
[](#cite:knuth1998)。

```ruby
"Alice".hash    # => （実行ごとに変わる大きな整数）
:port.hash      # => ハッシュ値
[1, 2].hash     # => 配列も中身からハッシュ値を計算できる
```

Ruby ではオブジェクトが `hash` メソッドと `eql?` メソッドを持ち、
ハッシュのキーとして使えます。`hash` が「どのバケットに入れるか」を決め、
`eql?` が「同じキーか」を最終判定します。自作クラスをキーにしたいときは、
この二つを正しく定義する必要があります。

```ruby
Point = Struct.new(:x, :y) do
  def hash = [x, y].hash      # 中身からハッシュ値を作る
  def eql?(o) = o.is_a?(Point) && x == o.x && y == o.y
end

h = {}
h[Point.new(1, 2)] = "origin-ish"
p h[Point.new(1, 2)]   # => "origin-ish"  別オブジェクトでも同じキー扱い
```

> [!IMPORTANT]
> 「`hash` が等しいキー同士は `eql?` でなければならない」という規約があります。
> `eql?` で等しい二つのキーが違う `hash` を返すと、ハッシュ表は同じキーを
> 別物として扱ってしまい、`h[k]` が見つからなくなります。`hash` と `eql?` は
> **必ずセットで**正しく定義する、と覚えてください。

## 衝突への二つの対処法

どんなに良いハッシュ関数でも、異なるキーが同じバケットに割り当てられる
**衝突**は避けられません（バケット数より多くのキーを入れれば必ず起きます）。
対処法は大きく二つです。

**チェイン法**（chaining）は第2章で見たとおり、同じバケットに来たキーを
連結リストや配列でつないでいく方式です。実装が素直で、要素数が容量を
超えても破綻しません。

**オープンアドレス法**（open addressing）は、衝突したら**別の空きバケットを
探して**そこに入れる方式です。すべての要素がバケット配列の中に直接収まり、
リストのための追加メモリやポインタ参照が要りません。連続メモリに収まるため
CPU キャッシュとの相性が良く、近年の高速なハッシュ実装で好まれます
[](#cite:knuth1998)。

```ruby
# 概念図：線形探索によるオープンアドレス法
class OpenHash
  def initialize(cap = 8)
    @keys = Array.new(cap)
    @vals = Array.new(cap)
  end

  def []=(key, val)
    i = key.hash % @keys.size
    until @keys[i].nil? || @keys[i] == key
      i = (i + 1) % @keys.size      # 埋まっていたら隣のマスへ（線形探索）
    end
    @keys[i] = key
    @vals[i] = val
  end

  def [](key)
    i = key.hash % @keys.size
    until @keys[i].nil?
      return @vals[i] if @keys[i] == key
      i = (i + 1) % @keys.size
    end
    nil
  end
end
```

埋まっていたら隣を見る、という上の方式を**線形探索法**（linear probing）と
呼びます。実装は簡単ですが、要素が固まって入ると探索が長引く欠点があり、
これを和らげるために探索の進み方を工夫した変種（二次探索、ダブル
ハッシュなど）も使われます [](#cite:knuth1998)。

> [!CAUTION]
> オープンアドレス法では**削除**に注意が必要です。あるキーを単純に消して
> 空マスに戻すと、その先にあった「衝突して押し出されたキー」が探索で
> たどり着けなくなります。これを防ぐため、削除したマスに「ここは消したが
> 通り抜けてよい」という目印（**トゥームストーン**、tombstone、墓石）を
> 置くのが定石です。

## 負荷率と再ハッシュ

ハッシュ表は、要素が増えてバケットが混んでくると衝突が増え、性能が落ちます。
混み具合を表す指標が**負荷率**（load factor）で、「要素数 ÷ バケット数」で
定義されます。負荷率が一定（たとえば 0.7）を超えたら、**より大きな
バケット配列を確保し、全要素を入れ直す**操作を行います。これを
**再ハッシュ**（rehashing）と呼びます。

これは第6章で見た可変長配列の「倍々拡張」と同じ考え方です。再ハッシュは
全要素の入れ直しなので一回は重い（O(N)）のですが、容量を倍々に増やすので
ならせば 1 要素あたり O(1) に収まります [](#cite:cormen2009)。
「平均 O(1)」という、ハッシュの売り文句の裏には、このならし計算量が
あるのです。

> [!NOTE]
> 負荷率の上限をどこに設定するかも設計判断です。低く設定すれば衝突は
> 減って速くなりますが、空きバケットが増えてメモリを食います。高く
> 設定すればメモリは節約できますが衝突が増えます。チェイン法は負荷率 1 を
> 超えても動きますが、オープンアドレス法は負荷率が 1 に近づくと急激に
> 遅くなるため、低めの上限が選ばれます。

## 挿入順を覚えるハッシュ

ハッシュは本来「順序のない」コレクションですが、Ruby の `Hash` は
**挿入した順序を保って**反復します。これは利用者にとって直感的で便利な
性質です（Python の `dict` も 3.7 以降は同様です）。

```ruby
h = {}
h[:c] = 1; h[:a] = 2; h[:b] = 3
p h.keys   # => [:c, :a, :b]   入れた順に並ぶ
```

これをどう実装するのでしょう。素朴なハッシュ表はバケットに散らばるので
順序が壊れます。そこで近年の実装は、**「挿入順に詰めた配列」と「キーから
その配列位置を引くハッシュ表」を組み合わせる**方式を採ります。実データは
挿入順の配列に並べ、ハッシュ表にはその配列の添字だけを入れておくのです。
反復は配列を順になぞるだけ、検索はハッシュ表で添字を引いてから配列を見る、
という二段構えで、順序保持と O(1) 検索を両立させます。

## ハッシュを攻撃から守る

ハッシュには、性能を悪用される弱点があります。攻撃者が**わざと同じバケットに
集まるキー**ばかりを送り込むと、衝突が多発してすべての操作が O(N) に
劣化し、サーバが処理不能に陥ります（**ハッシュ衝突攻撃**、hash-flooding）。

対策として、近年の処理系はハッシュ関数に**乱数の種**（プロセスごとに変わる
秘密の値）を混ぜます。攻撃者は種を知らないため、衝突するキーを意図的に
作れません。Ruby を含む多くの言語が、文字列のハッシュにこの目的で
設計された **SipHash** という関数を採用しています。`"Alice".hash` の値が
実行のたびに変わるのは、この種が毎回異なるためです。

> [!WARNING]
> 自分でハッシュ関数を実装するときは、速さだけでなく**安全性**も考える
> 必要があります。外部から来るデータ（HTTP のパラメータなど）を
> キーにするなら、衝突攻撃への耐性が欠かせません。「速いハッシュ関数」と
> 「攻撃に強いハッシュ関数」は別物で、用途で使い分けます。

## 不変なハッシュ：再び HAMT

第6章の永続配列で登場した HAMT は、もともとは**ハッシュ表**のための
データ構造でした [](#cite:bagwell2001)。キーのハッシュ値を 5 ビットずつに
区切り、その各かたまりを 32 分木の枝をたどる道しるべに使います。
これにより、再ハッシュのための一括コピーを必要とせず、要素を追加しても
木の経路ぶんのノードを作り直すだけで済みます。

この性質は「変更しても元のバージョンが残る」永続性と相性が良く、Clojure の
不変マップや Scala の不変 `Map` の実装基盤になっています。可変ハッシュは
オープンアドレス法＋再ハッシュ、不変ハッシュは HAMT —— 同じ「キーで引く表」
でも、可変か不変かという要求の違いが、まったく異なるデータ構造を呼ぶのです。

見てきました。次の章では、これらを部品として組み立てられる、より高水準な
データ構造 —— オブジェクト指向の**オブジェクト**の実装に進みます。
