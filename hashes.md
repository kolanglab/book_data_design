# ハッシュ（辞書）の表現：キーで引く表

## キーで引く、という要求

配列は「0 番目、1 番目…」と整数の添字で引きました。しかし現実には、
「名前 `"Alice"` の年齢は？」「設定項目 `:port` の値は？」のように、
**任意の値をキーにして引きたい**場面が大半です。これを実現するのが
**ハッシュ**（hash、連想配列・辞書とも呼ぶ）型です。Ruby の `Hash`、
Python の `dict`、JavaScript の `Map`、Java の `HashMap`、Go の `map` が
これにあたります。

```ruby
ages = { "Alice" => 30, "Bob" => 25 }
p ages["Alice"]   # => 30
ages["Carol"] = 28
```

ハッシュの理想は、キーが文字列でもオブジェクトでも、**平均 O(1)** で
読み書きできることです。[シンボルテーブルの章](symbol-table.md)でハッシュ表の骨格は
すでに見ました。この章では、実用のハッシュを支える設計判断を掘り下げます。
ハッシュ表ほど「言語ごとに実装が違い、しかもその違いに理由がある」
データ構造はありません。

## ハッシュ関数とバケット

ハッシュ型の心臓は**ハッシュ関数**（hash function）です。これは任意のキーを
固定範囲の整数（ハッシュ値）に変換する関数で、その整数を配列（バケットの列）の
添字に使います。良いハッシュ関数は、異なるキーをできるだけ**均等に**
散らばらせます。偏ると衝突が増え、O(1) の理想が崩れるからです
[](#cite:knuth1998)。

```ruby
"Alice".hash    # => （実行ごとに変わる大きな整数）
:port.hash      # => ハッシュ値
[1, 2].hash     # => 配列も中身からハッシュ値を計算できる
```

Ruby ではオブジェクトが `hash` メソッドと `eql?` メソッドを持ち、
ハッシュのキーとして使えます。`hash` が「どのバケットに入れるか」を決め、
`eql?` が「同じキーか」を最終判定します。自作クラスをキーにしたいときは、
この二つを正しく定義する必要があります。

```ruby
Point = Struct.new(:x, :y) do
  def hash = [x, y].hash      # 中身からハッシュ値を作る
  def eql?(o) = o.is_a?(Point) && x == o.x && y == o.y
end

h = {}
h[Point.new(1, 2)] = "origin-ish"
p h[Point.new(1, 2)]   # => "origin-ish"  別オブジェクトでも同じキー扱い
```

> [!IMPORTANT]
> 「`eql?` で等しいキー同士は、同じ `hash` を返さなければならない」という
> 規約があります（逆は要求されません。`hash` が同じでも別キーなのは、
> ただの衝突として正常に扱われます）。
> `eql?` で等しい二つのキーが違う `hash` を返すと、ハッシュ表は同じキーを
> 別物として扱ってしまい、`h[k]` が見つからなくなります。`hash` と `eql?` は
> **必ずセットで**正しく定義する、と覚えてください。同じ規約は Java の
> `hashCode`/`equals`、Python の `__hash__`/`__eq__` にもあります。
> どの言語でも、ハッシュ表という**データ構造の要請**が、オブジェクトの
> 振る舞いの契約として言語仕様に昇格しているのです。

関連して、**可変なオブジェクトをキーにする**問題があります。キーを
表に入れた後で中身を書き換えると、ハッシュ値が変わって「入れたのに
見つからない」状態になります。Python は土俵際でこれを防ぎ、リストなどの
可変オブジェクトを**そもそもキーにできなく**しました（`unhashable`）。
Ruby は許した上で、壊れたら `rehash` で再構築する道具を渡し、ただし
最頻出の文字列キーだけは**登録時に凍結した複製**を作って守る、という
折衷を採っています。

## 衝突への二つの対処法

どんなに良いハッシュ関数でも、異なるキーが同じバケットに割り当てられる
**衝突**は避けられません（バケット数より多くのキーを入れれば必ず起きます）。
対処法は大きく二つです。

**チェイン法**（chaining）は[シンボルテーブルの章](symbol-table.md)で見たとおり、同じバケットに
来たキーを連結リストや配列でつないでいく方式です。実装が素直で、要素数が
容量を超えても破綻しません。Java の `HashMap` はチェイン法ですが、
一つのバケットの鎖が 8 本を超えると、そのバケットだけ連結リストを
**赤黒木**（平衡二分探索木の一種、「平衡木」の章で詳述します）
に組み替え、最悪ケースを O(n) から O(log n) に抑えるという念の入れ
ようです。

**オープンアドレス法**（open addressing）は、衝突したら**別の空きバケットを
探して**そこに入れる方式です。すべての要素がバケット配列の中に直接収まり、
リストのための追加メモリやポインタ参照が要りません。連続メモリに収まるため
CPU キャッシュとの相性が良く、近年の高速なハッシュ実装で好まれます
[](#cite:knuth1998)。

```ruby
# 概念図：線形探索によるオープンアドレス法
class OpenHash
  def initialize(cap = 8)
    @keys = Array.new(cap)
    @vals = Array.new(cap)
  end

  def []=(key, val)
    i = key.hash % @keys.size
    until @keys[i].nil? || @keys[i] == key
      i = (i + 1) % @keys.size      # 埋まっていたら隣のマスへ（線形探索）
    end
    @keys[i] = key
    @vals[i] = val
  end

  def [](key)
    i = key.hash % @keys.size
    until @keys[i].nil?
      return @vals[i] if @keys[i] == key
      i = (i + 1) % @keys.size
    end
    nil
  end
end
```

埋まっていたら隣を見る、という上の方式を**線形探索法**（linear probing）と
呼びます。実装は簡単ですが、要素が固まって入ると探索が長引く欠点があり、
これを和らげるために探索の進み方を工夫した変種（二次探索、ダブル
ハッシュなど）も使われます [](#cite:knuth1998)。

> [!CAUTION]
> オープンアドレス法では**削除**に注意が必要です。あるキーを単純に消して
> 空マスに戻すと、その先にあった「衝突して押し出されたキー」が探索で
> たどり着けなくなります。これを防ぐため、削除したマスに「ここは消したが
> 通り抜けてよい」という目印（**トゥームストーン**、tombstone、墓石）を
> 置くのが定石です。

## 負荷率と再ハッシュ

ハッシュ表は、要素が増えてバケットが混んでくると衝突が増え、性能が落ちます。
混み具合を表す指標が**負荷率**（load factor）で、「要素数 ÷ バケット数」で
定義されます。負荷率が一定（たとえば 0.7）を超えたら、**より大きな
バケット配列を確保し、全要素を入れ直す**操作を行います。これを
**再ハッシュ**（rehashing）と呼びます。

これは[配列の章](arrays.md)で見た可変長配列の「倍々拡張」と同じ考え方です。再ハッシュは
全要素の入れ直しなので一回は重い（O(N)）のですが、容量を倍々に増やすので
ならせば 1 要素あたり O(1) に収まります [](#cite:cormen2009)。
「平均 O(1)」という、ハッシュの売り文句の裏には、このならし計算量が
あるのです。

> [!NOTE]
> 負荷率の上限をどこに設定するかも設計判断です。低く設定すれば衝突は
> 減って速くなりますが、空きバケットが増えてメモリを食います。高く
> 設定すればメモリは節約できますが衝突が増えます。チェイン法は負荷率 1 を
> 超えても動きますが、オープンアドレス法は負荷率が 1 に近づくと急激に
> 遅くなるため、低めの上限が選ばれます（Java HashMap は 0.75、
> 後述の SwissTable は 0.875 前後）。

なお、Go の `map` は再ハッシュを**一括では行わない**設計を一貫して
採っています。Go 1.23 までの実装は、新旧二つの表をしばらく併存させて
操作のたびに少しずつ引っ越す**漸進的な成長**でした。Go 1.24 で内部が
一新された後も（後述の SwissTable）、map 全体を多数の小さな表に分割し、
成長を**表単位**で行うことで、一回の操作が突然 O(N) になる「再ハッシュの
瞬間」を避けています [](#cite:pratt2025)。ならし O(1) ではなく
**最悪でも小さい**ことを優先する —— レイテンシ重視の設計です。

## 挿入順を覚えるハッシュ：二つの配列に分解する

ハッシュは本来「順序のない」コレクションですが、Ruby の `Hash` は
**挿入した順序を保って**反復します（Python の `dict` も 3.7 以降は
言語仕様として同様です）。

```ruby
h = {}
h[:c] = 1; h[:a] = 2; h[:b] = 3
p h.keys   # => [:c, :a, :b]   入れた順に並ぶ
```

これをどう実装するのでしょう。素朴なハッシュ表はバケットに散らばるので
順序が壊れます。現代の実装が採るのは、**表を二つの配列に分解する**方式です。

1. **エントリ配列**：「キー・値・ハッシュ値」の組を、挿入順に詰めて
   並べた密な配列。
2. **索引配列**：ハッシュ値から「エントリ配列の何番目か」を引くための
   オープンアドレス表。中身は小さな整数だけ。

反復はエントリ配列を頭から舐めるだけ（キャッシュに優しい！）、検索は
索引配列で位置を引いてからエントリ配列を見る、という二段構えです。
索引配列に入るのは添字だけなので、表が小さいうちは 1 マス 1 バイトで
済み、メモリも大きく節約できます。

この二段構えも、骨格だけなら短く実装できます。

```ruby
# 挿入順ハッシュ（索引＋エントリ配列）の最小実装
class OrderedDict
  def initialize
    @index   = Array.new(8)   # ハッシュ表側：エントリ配列の添字だけが入る
    @entries = []             # 実データ [ハッシュ値, キー, 値] を挿入順に
  end

  def find_slot(key, h)       # 線形探索のオープンアドレス法
    i = h % @index.size
    until @index[i].nil?
      e = @entries[@index[i]]
      return i if e[0] == h && e[1].eql?(key)   # まずハッシュ値で粗く比較
      i = (i + 1) % @index.size
    end
    i                         # 空きスロット
  end

  def []=(key, value)
    h = key.hash
    i = find_slot(key, h)
    return @entries[@index[i]][2] = value if @index[i]   # 上書き
    if (@entries.size + 1) * 4 > @index.size * 3         # 負荷率 3/4 で拡張
      @index = Array.new(@index.size * 2)                # 索引だけ作り直し！
      @entries.each_with_index { |e, n| @index[find_slot(e[1], e[0])] = n }
      i = find_slot(key, h)
    end
    @index[i] = @entries.size
    @entries << [h, key, value]
  end

  def [](key)
    n = @index[find_slot(key, key.hash)]
    n && @entries[n][2]
  end

  def each_pair = @entries.each { |_, k, v| yield k, v }  # 挿入順そのまま
end

d = OrderedDict.new
d[:c] = 1; d[:a] = 2; d[:b] = 3; d[:a] = 20
d.each_pair { |k, v| print k, "=", v, " " }   # => c=1 a=20 b=3
```

再ハッシュ時に**エントリ配列にはいっさい触らず、索引だけを作り
直している**点が、この設計の利得の凝縮です（エントリにはハッシュ値が
記録済みなので再計算も不要）。削除は、エントリ配列に「空き」の印を
残して索引から外すだけにして、空きが溜まったら詰め直す ——
オープンアドレス法のトゥームストーンと同じ扱いになります。

この設計は CPython の dict（Hettinger の提案によるコンパクト辞書
[](#cite:hettinger2012)）と、CRuby 2.4 以降の `st_table`（Makarov に
よる全面改修 [](#cite:makarov2016)）がそれぞれ採用したもので、
**「挿入順の保持」が最初から目的だったのではなく、密な配列に詰めた
副産物として順序が手に入った**、というのが面白いところです。Python では
「実装の都合で順序が保たれる」状態が一版だけ続いた後、3.7 で言語仕様に
昇格しました。データ構造の選択が言語仕様を書き換えた例です。

さらに CRuby は、**8 要素以下の小さなハッシュにはハッシュ表を使わ
ない**という割り切りも持っています（`ar_table`）。キーと値をただ配列に
並べ、検索は線形比較。要素が一桁なら、ハッシュ計算より 8 回の比較の
ほうが速く、メモリも最小だからです。キーワード引数など小さなハッシュが
大量に作られる Ruby らしい最適化です。

## 高速化の最前線：SwissTable

オープンアドレス法の進化は今も続いています。近年の代表が、Google の
**SwissTable** 系の設計です [](#cite:kulukundis2017)。Rust の標準
`HashMap`（hashbrown）、C++ の `absl::flat_hash_map`、そして Go 1.24 の
`map` [](#cite:pratt2025) が採用しています。

鍵は**メタデータ配列**です。各バケットの「空き／削除済み／使用中＋
ハッシュ値から取った 7 ビット」を 1 バイトに詰めた配列を、本体とは
別に持ちます（どの 7 ビットかは実装ごとに違い、Abseil や Go 1.24 は
バケット選択に使わなかった**下位** 7 ビット、Rust の hashbrown は
上位 7 ビットを使います）。検索ではまずこのメタデータを**グループ
単位でまとめて**照合し —— Abseil は 16 マスを **SIMD 命令**（一度に
16 バイトを比較できる CPU 命令）で、Go 1.24 は 8 マス分の制御バイトを
1 個の 64 ビット整数とみなしてビット演算で [](#cite:pratt2025) ——
7 ビットが一致したマスだけ本体のキー比較に進みます。ほとんどの
不一致は 1 バイトの比較で振り落とされるので、キー本体に触れる回数が
激減します。

「ハッシュ値の一部をキャッシュに優しい別配列に複製しておく」——
情報の**置き場所**を変えただけでアルゴリズムは古典のままなのに、
現代の CPU では数倍速くなる。データ構造の性能がメモリ階層で決まる
時代を象徴する設計です。なお一世代前には、挿入時に「家から遠い者
優先」で席を譲り合うことで探索距離のばらつきを抑える **Robin Hood
ハッシュ法** [](#cite:celis1986) が Rust などで使われており、
こちらも有名です。

## ハッシュを攻撃から守る

ハッシュには、性能を悪用される弱点があります。攻撃者が**わざと同じバケットに
集まるキー**ばかりを送り込むと、衝突が多発してすべての操作が O(N) に
劣化し、サーバが処理不能に陥ります（**ハッシュ衝突攻撃**、hash-flooding）。
2011 年に主要な Web 言語が軒並みこの攻撃に脆弱だと公表され、各処理系が
一斉に対策した事件は、データ構造の選択がセキュリティ問題になることを
広く知らしめました。

対策として、近年の処理系はハッシュ関数に**乱数の種**（プロセスごとに変わる
秘密の値）を混ぜます。攻撃者は種を知らないため、衝突するキーを意図的に
作れません。Ruby・Python・Rust などが、この目的で設計された **SipHash**
[](#cite:aumasson2012) を文字列ハッシュに採用しています。`"Alice".hash` の
値が実行のたびに変わるのは、この種が毎回異なるためです。

> [!WARNING]
> 「安全なハッシュ関数」はタダではありません。SipHash は乱雑さの保証と
> 引き換えに、単純な乗算ベースの高速ハッシュより遅いのです。攻撃に
> さらされない内部用途（コンパイラの内部表など）では速い関数を使い、
> 外部入力をキーにする表では安全な関数を使う、という使い分けが
> 行われます。Rust が `HashMap` のハッシュ関数を型パラメータとして
> 差し替え可能にしているのは、この使い分けを利用者に開放するためです。

ちなみに Go は、ハッシュの**反復順序を意図的に毎回ランダム化**して
います。順序に依存したコードを最初から書けなくするためで、「実装の
都合（順序不定）を仕様（順序保証）に昇格させた」Python とちょうど
逆向きの判断です。

## 不変なハッシュ：再び HAMT

[配列の章](arrays.md)の永続配列で登場した HAMT は、もともとは**ハッシュ表**の
ためのデータ構造でした [](#cite:bagwell2001)。キーのハッシュ値を 5 ビット
ずつに区切り、その各かたまりを 32 分木の枝をたどる道しるべに使います。
これにより、再ハッシュのための一括コピーを必要とせず、要素を追加しても
木の経路ぶんのノードを作り直すだけで済みます。

HAMT を実装可能にしている急所は、**ビットマップと popcount による
ノードの圧縮**です。32 分木のノードに 32 本の子ポインタを常に
持たせたら、まばらなノードはほぼ空欄でメモリの無駄です。そこで
各ノードは「**どの枝が存在するか**」を 32 ビットのビットマップで
持ち、子の配列には**存在する枝だけを詰めて**並べます。「枝 i は
配列の何番目か」は、**ビットマップの i 番目より下位に立っている
ビットの数**——CPU の popcount（立っているビットを数える）1 命令——
で求まります。

```ruby
# HAMT のルックアップ（1 段ぶん）。実装の核心は 3 行
def hamt_get(node, hash, shift, key)
  idx = (hash >> shift) & 0b11111          # ハッシュ値から 5 ビット取り出す
  bit = 1 << idx
  return nil if node.bitmap & bit == 0     # その枝は存在しない
  pos = (node.bitmap & (bit - 1)).to_s(2).count("1")  # popcount で詰め位置
  child = node.children[pos]
  child.is_a?(Leaf) ? (child.key == key ? child.val : nil)
                    : hamt_get(child, hash, shift + 5, key)
end
```

枝が 1 本のノードは 1 要素ぶんの配列しか持たない —— 疎でも密でも
無駄が出ません。「疎な固定長配列を、ビットマップ＋詰めた配列で表す」
このイディオムは HAMT の発明の半分であり、単体でも使い道の多い
道具です。

この性質は「変更しても元のバージョンが残る」永続性と相性が良く、Clojure の
不変マップや Scala の不変 `Map` の実装基盤になっています。可変ハッシュは
オープンアドレス法＋再ハッシュ、不変ハッシュは HAMT —— 同じ「キーで引く表」
でも、可変か不変かという要求の違いが、まったく異なるデータ構造を呼ぶのです。

## 言語ごとのハッシュ実装・早見表

本章のまとめとして、主要処理系のハッシュ実装を並べてみます。

| 処理系 | 方式 | 特徴 |
|---|---|---|
| CRuby (`Hash`) | 小：線形配列 ／ 大：索引＋エントリ配列 | 挿入順保持、SipHash |
| CPython (`dict`) | 索引＋エントリ配列 | 挿入順は言語仕様、キー共有辞書（[オブジェクトの章](objects.md)） |
| Java (`HashMap`) | チェイン法＋赤黒木化 | 負荷率 0.75、最悪 O(log n) |
| Go (`map`) | SwissTable（1.24〜） | 表単位の漸進的成長、反復順ランダム化 |
| Rust (`HashMap`) | SwissTable (hashbrown) | ハッシュ関数差し替え可、既定 SipHash |
