# Lisp：コードがデータである

Lua が「すべてはテーブル」だったように、Lisp にも一枚岩の答えがあります。
ただし Lisp の凄みは、その一つの構造 ── **S 式** ── が、**データであると
同時にプログラムそのものでもある**点にあります。構文木の章で見た「木」を、
Lisp は利用者の手に握らせてしまう。これを **homoiconicity（同図像性、
コードとデータが同じ形）** と呼びます。本章はこの一点 ── プログラムが
自分自身をデータとして扱える、という設計 ── を掘ります。

## かっこの中の世界：S 式

Lisp のソースは、**S 式（S-expression）** だけでできています。S 式は
たった二種類です。

- **アトム（atom）**：これ以上分解しない値 ── 数、文字列、そして
  **シンボル**（`+`、`foo` のような名前。識別子の章で見たインターン済みの
  名前そのものです）。
- **リスト（list）**：`(a b c)` のように、S 式をかっこで並べたもの。
  その実体は、リスト・タプル・集合の章で見た **cons セルの連なり**です。

`(+ 1 (* 2 3))` という式は、`+` と `1` と「`*` と `2` と `3` のリスト」を
要素に持つ、ただのリストです。プログラムの構造が、そのまま木（入れ子の
リスト）として目に見えている ── 構文という飾りがほとんどなく、
**書いた通りの形が構文木**なのです。ほかの言語が `1 + 2 * 3` という
平らな文字列を構文解析の章の苦労を経て木に変えるのに対し、Lisp は
最初から木を書かせます。

## リーダ：テキストから S 式へ

Lisp 処理系の入口は **リーダ（reader）** です。`read` はソースの文字列を
読み、対応する S 式 ── すなわち cons セルとアトムのデータ構造 ── を
作って返します。その逆が **プリンタ**（`print`）で、S 式を文字列へ戻します。
有名な **REPL**（Read-Eval-Print Loop）は、この `read` → `eval` →
`print` をただ繰り返す輪です。

ここで決定的なのは、**`read` の出力（S 式）と、`eval` の入力（プログラム）が
同じものだ**ということです。構文解析の章では「テキスト → トークン →
AST」と段を踏み、AST は処理系の内部だけの構造でした。Lisp ではその AST が、
利用者がいつでも作れて触れる**第一級のリスト**になっています。
パーサの出力が、そのまま言語のデータ型なのです。

## quote：評価を止めると、データになる

同じ S 式が「コード」にも「データ」にもなる、その切り替えが **quote** です。

```lisp
(+ 1 2)        ; => 3        評価される：コードとして
'(+ 1 2)       ; => (+ 1 2)  評価されない：3 要素のリストというデータ
(car '(+ 1 2)) ; => +        その先頭はシンボル + （データとして取り出せる）
```

クォート `'`（`quote` の略記）は「この S 式を評価せず、データとして
よこせ」という指示です。`(+ 1 2)` はふだんは「1 と 2 を足す呼び出し」
ですが、クォートすると「`+`・`1`・`2` という 3 要素のリスト」という
**ただのデータ**になり、`car`/`cdr`（リスト・タプル・集合の章）で
中身を分解できます。逆に、データとして組み立てたリストを `eval` に
渡せば、それはコードとして走ります。**コードとデータのあいだに壁が
なく、`quote` と `eval` で自由に行き来できる** ── これが homoiconicity の
正体です。

## マクロ：S 式を受け取り、S 式を返す関数

homoiconicity が最も威力を発揮するのが **マクロ**です。Lisp のマクロは、
「**S 式を受け取って、別の S 式を返す関数**」にすぎません。ただし
ふつうの関数と違い、**コンパイル時**（評価される前）に呼ばれ、その戻り値の
S 式が元の場所に置き換わってから評価されます。つまりマクロは
「プログラムを書くプログラム」を、言語の中のただのリスト操作として書ける
のです。

```lisp
;; unless を if のマクロとして定義する
(defmacro unless (cond body)
  (list 'if cond nil body))   ; (unless C B) を (if C nil B) という S 式に変換

(unless nil 'ok)   ; => ok    展開後: (if nil nil 'ok)
```

`unless` は `(if cond nil body)` という**リストを組み立てて返す**だけです。
手で `(list 'if ...)` と組むのは煩雑なので、Lisp は **準クォート**
（quasiquote、`` ` ``）と **アンクォート**（unquote、`,`／`,@`）という
テンプレート記法を用意します。`` `(if ,cond nil ,body) `` は「`if` と
`nil` はそのまま、`,cond` と `,body` のところだけ値を埋める」という、
穴あきの S 式テンプレートです。

マクロが生成した一時変数の名前が利用者の変数と衝突する問題と、その解
（**gensym** と**衛生的マクロ**）は、識別子の章で見たとおりです ──
名前のデータ構造を「綴り＋出自」にリッチ化して衝突を消す、あの話が、
ここで実際に使われます。

## 構文木が第一級になるということ

ふつうの言語で「プログラムを生成・変換するプログラム」を書くのは大仕事
です。構文木の章の AST は処理系の内部にあり、利用者は文字列を
こねるか、専用のマクロ機構を学ぶしかありません。Lisp はその AST を
**言語の基本データ型（リスト）そのもの**にしたので、メタプログラミングが
「リストを組み替える」という日常の操作になります。

```lisp
;; コードをデータとして組み立て、eval で走らせる
(let ((expr (list '+ 1 (list '* 2 3))))   ; (+ 1 (* 2 3)) というデータを作る
  (eval expr))                            ; => 7  データをコードとして実行
```

「構文木をデータとして持ち、変換し、実行する」── 本書の構文木の章・
クロージャの章（継続）・JIT の章で別々に見た「コードを値として扱う」
発想の、最も純粋な原型がここにあります。プログラムとデータの境界を
取り払うと何ができるか、を Lisp は 1960 年から示し続けてきました
[](#cite:mccarthy1960)。

## 系譜と教訓

- **homoiconicity** は、Scheme・Common Lisp・Clojure・Racket に受け継がれ、
  さらに Elixir（quote/unquote で AST を操る）や Rust（手続きマクロが
  トークン列を受け取り返す）など、Lisp 直系でない言語のマクロ機構にも
  形を変えて生きています。「コードを値として変換する」道具立ては、
  いまや主流言語のメタプログラミングの土台です。
- **S 式というデータ形式**そのものも生き延びました。設定やデータ交換に
  使われる入れ子の表記 ── ある意味で JSON すら ── は、「アトムとリストの
  入れ子で何でも表す」という S 式の子孫と見ることができます。直列化の
  章で見た木＋後方参照の話も、S 式の `read`/`print` がその古典でした。
- Lua が「**一つのデータ構造（テーブル）で全部**」だったのに対し、Lisp は
  「**一つのデータ構造（S 式）でプログラムすら表す**」まで踏み込みました。
  どちらも構造を一つに絞る思想ですが、Lisp はその一つを**言語自身の
  記述**にまで使い切った点で際立ちます。
- 教訓は明快です。**構文木を第一級のデータにすると、言語は自分自身を
  拡張できる**。新しい制御構造も、ドメイン特化言語も、ライブラリとして
  リスト操作で書ける ── 「言語機能か、ライブラリか」の境界を、データ
  構造の選択一つで溶かしてみせたのが Lisp でした。

これで個性派言語の旅は本当におしまいです。すべてが配列の APL から、
コードすらデータの Lisp まで ── どの言語も「何を一つの構造に賭けたか」が、
そのまま言語の個性になっていました。次章では本書全体を振り返り、ここまで
見てきた数々のデータ構造に通底する設計の原理を、改めて言葉にします。
