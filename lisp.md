# コードがデータである Lisp

Lua が「すべてはテーブル」だったように、Lisp も一つの構造に賭けました。
ただしその構造（**S 式**）は、データであると同時にプログラムでもあります
（**homoiconicity、同図像性**）。これ自体は言語機能の話ですが、本書の問いは
その先です。その「コード兼データ」を、メモリではどんな形で
持っているのか。Lisp の凄みは、思想ではなく表現にあります。

## cons セルとタグ付きの語

S 式は二種類しかありません。**アトム**（数やシンボルなど、これ以上
分解しない値）と、**リスト**（S 式を並べたもの）です。そして、その実体は
本書ですでに見た二つの道具に尽きます。

- **アトムは、タグ付きの 1 語**です（[値の表現の章](values.md)）。小さな整数は即値、
  `nil` は特別な定数、シンボル `+` は**インターン済み**のシンボルオブジェクト
  への参照です（[識別子の章](identifier.md)。同じ綴りは必ず同一になります）。
- **リストは、cons セルの鎖**です（[リスト・タプル・集合の章](lists.md)）。cons セルは
  `car` と `cdr` の 2 語だけの箱。`car` に要素、`cdr` に「残りのリスト」を
  指させ、末尾は `nil` で閉じます。

つまり `(+ 1 (* 2 3))` は、構文でも木でもなく、メモリ上に散らばった
cons セルとタグ付きの語が、ポインタでつながったグラフです。

```text
(+ 1 (* 2 3)) のメモリ表現（□=cons セル[car|cdr]、→=ポインタ）

  □[+ | →]──□[1 | →]──□[• | nil]
                          │
                          └─□[* | →]──□[2 | →]──□[3 | nil]
```

外側のリストは 3 つの cons セル、内側の `(* 2 3)` はさらに 3 つです。各 `car` は
シンボル（`+` か `*`）か即値（`1`、`2`、`3`）か、別の cons セルへの参照
（入れ子）です。`cdr` が次のセルを指し、最後だけ `nil` です。この
ポインタグラフこそが「プログラム `(+ 1 (* 2 3))`」の正体であり、同時に
「6 個のアトムからなるデータ」の正体でもあります。木を文字列で書いて
あとで構文解析するのではなく、最初から木そのものを cons セルで書いて
いるのです。

> [!NOTE]
> `cdr` が必ず `nil` で終わるリストを **真リスト**、終わらないもの
> （`(1 . 2)` のような単独の cons）を **ドット対**と呼びます。`(a b c)` は
> `(a . (b . (c . nil)))` の糖衣にすぎません。リストとは「cons セルの
> 連なり」という一つの表現の見え方であり、構文上の特別扱いではないのです。

## テキストを cons セルに組み上げるリーダ

Lisp 処理系の入口 **リーダ（`read`）** の仕事は、文字列を上の cons グラフへ
組み上げることです。`(` を読んだらリストの構築を始め、要素を読むたびに
cons セルを一つ作って鎖を伸ばし、`)` で `nil` を結んで閉じる。数字なら
即値アトムに、名前ならシンボル表を引いて（無ければ登録して）インターン済み
シンボルにします。逆向きの `print` は、この cons グラフをたどって文字列へ
戻すだけです。

構文解析の章では「テキスト → トークン → AST」と段を踏み、AST は処理系の
内部だけの構造でした。Lisp が違うのは、`read` が吐くその cons グラフが、
**そのまま言語の基本データ型**だという一点です。パーサの出力が、利用者の
手に渡るふつうの値になっています。だから次の話が成り立ちます。

## データとコードを兼ねる cons セル

`eval`（ツリーウォーク型インタプリタ）は、この cons グラフを**ポインタを
たどって**評価します。先頭の `car`（`+`）を見て演算を決め、`cdr` をたどって
引数を順に評価します。[構文木の章](syntax-tree.md)の「木を再帰でたどる評価」そのものですが、
その木が専用の AST ノードではなくふつうの cons セルだ、というのが
Lisp なのです。

**quote** は、この同じセル群を「評価せずそのまま渡す」指示にすぎません。

```lisp
(+ 1 2)        ; => 3        eval が cons をたどって計算する（コード）
'(+ 1 2)       ; => (+ 1 2)  同じ cons 鎖を、たどらず手渡す（データ）
(car '(+ 1 2)) ; => +        その先頭 cons の car（シンボル +）を取り出す
```

`(+ 1 2)` を `eval` するのも、`'(+ 1 2)` を `car` で分解するのも、**触って
いるメモリ上の cons セルは同じ**です。違うのは「たどって計算するか、その
まま値として扱うか」だけです。homoiconicity とは思想以前に、評価器が歩く
構造と、利用者がいじれるデータ構造が、物理的に同一の cons 表現である
という事実なのです。[構文木の章](syntax-tree.md)で AST が内部専用型だったのと、ちょうど
裏返しです。

## cons を組み替える関数としてのマクロ

ここまで来ると、マクロは特別な機構ではなく**cons セルを組み替えて返す
関数**だと分かります。コンパイル時に呼ばれ、受け取った cons グラフから
別の cons グラフを作って、元の場所に差し替えます。

```lisp
(defmacro unless (cond body)
  (list 'if cond nil body))   ; (unless C B) の cons を (if C nil B) の cons へ

(unless nil 'ok)   ; => ok    cons を組み替えてから eval される
```

`(list 'if ...)` は新しい cons セルを確保して鎖を作る操作そのものです。
手で組むのは煩雑なので、**準クォート** `` ` `` と **アンクォート** `,` や `,@`
が「穴あきの cons テンプレート」を用意します。`` `(if ,cond nil ,body) ``
は、`if` と `nil` のセルはそのまま、`,cond` と `,body` の位置だけ別の構造を
継ぎ込んだ cons グラフを組み立てます。マクロが作った一時シンボルが利用者の
名前と衝突する問題と、その解（**gensym**、つまり表に載せないシンボルと、**衛生的
マクロ**、つまり識別子を「綴り＋出自」で表す手法）は、[識別子の章](identifier.md)で見た表現の工夫
そのものです。

## 表現の代償と、その先

「コードを cons で持つ」表現には、はっきりした**代償**があります。cons セルは
1 個 2 語で、リストは要素ごとに別のセルへポインタをたどる連結リストです
（[リストの章](lists.md)）。メモリに散らばったセルを次々追うので、配列のような連続
アクセスに比べてキャッシュに優しくありません。素直に cons グラフを
ツリーウォークする評価は、書きやすいが遅いのです。

だから本格的な Lisp 処理系（SBCL など）は、`read` で得た cons 形を
**そのまま実行せず、機械語へコンパイル**します。cons 表現は「ソースや
マクロ展開の表現」として保ち、実行には別の compact な表現を使います。
[構文木の章](syntax-tree.md)で見た「AST は書きやすい入力表現、実行は命令列」という二段構えが、
ここでも効いています。Clojure に至っては、コードを表すのに cons リストだけ
でなく**永続ベクタ `[]` やマップ `{}`**（配列や[ハッシュの章](hashes.md)の不変版）も
第一級の構文として使い、「コード＝データ」を cons 以外の表現へ広げました。
Rust の手続きマクロが操るのが cons ではなくトークン列であるのも、
「同じ homoiconicity 的発想を、別の表現で実現した」例と言えます。

シンボルの側にも表現の工夫があります。Common Lisp のシンボルは、名前の
ほかに**値セルと関数セルを別々に持ち**（同じ綴りを変数にも関数にも使える
Lisp-2）、加えて属性リスト（plist）まで備えた小さなレコードです。
[識別子の章](identifier.md)のインターンが、ここでは「多目的のスロットを持つ常駐オブジェクト」
として実体化しています。

## 系譜と教訓

- **一つの表現を、コードにもデータにも使う**という選択は、Scheme・Common
  Lisp・Clojure・Racket に受け継がれました。表現の中身は進化し（cons から
  ベクタ・マップへ、トークン列へ）ても、「評価器が歩く構造を、利用者にも
  値として渡す」という表現上の決断は変わりません。
- **S 式という入れ子表現**そのものも生き延びました。設定やデータ交換の
  入れ子記法（ある意味で JSON すら）は「アトムとリストの入れ子で何でも
  表す」S 式の子孫で、その `read`/`print` は[直列化の章](serialize.md)の木⇄テキスト変換の
  古典でした。
- 教訓は表現の言葉で言えます。**AST を専用の内部型にせず、言語の基本
  データ型（cons とタグ付きアトム）で表すと、評価器の入力と利用者のデータ
  構造が一致し、メタプログラミングが「ふつうのリスト操作」になる**。
  その代償（連結リストの遅さ）は、実行用の表現を別に持つことで支払います。
  「入力に優しい表現」と「実行に速い表現」を分ける、本書を貫く構図です。

これで個性派言語の旅は本当におしまいです。すべてが配列の APL から、
コードすら cons セルの Lisp まで、どの言語も「何を一つの表現に賭けたか」が、
そのまま個性になっていました。次章では本書全体を振り返り、ここまで見てきた
数々のデータ構造に通底する設計の原理を、改めて言葉にします。
