# Tcl：すべてが文字列、ただし二重表現

## EIAS：過激な一貫性

Tcl の設計原則は **EIAS**（Everything Is A String、すべては文字列で
ある）と呼ばれます [](#cite:ousterhout1994)。数値も文字列、リストも
文字列、辞書も、そして**プログラムのコードさえも**文字列です。

```tcl
set x 42            ;# 42 という文字列
set lst {1 2 3}     ;# リスト？ いや「1 2 3」という文字列
eval $lst           ;# 文字列はいつでもコードとして実行できる
```

この一貫性の配当は大きいものでした。値とコードの区別がないから
`eval` やコールバックが自明になり、すべてが文字列だからプロセス間・
ネットワーク越しの**シリアライズがタダ**（そのまま送ればよい）に
なります。Tcl が組み込み・テスト自動化・GUI（Tk）の貼り合わせ言語と
して成功した土台は、この「交換可能性」でした。JSON や YAML で
「データを文字列で持ち回る」ことが常識になった現代から見ると、
30 年早い思想だったとも言えます。

問題は性能です。`incr x` のたびに文字列 `"42"` を整数 42 に
パースし、結果の 43 を文字列に戻していたら、ループは壊滅的に
遅くなります。実際、初期の Tcl はそう動いていました。

## Tcl_Obj：二重表現という解

Tcl 8.0（1997 年）は、意味論を一切変えずにこの問題を解決しました。
値の実体 `Tcl_Obj` に**二つの置き場**を持たせたのです。

- **文字列表現**（bytes）：その値の文字列としての姿
- **内部表現**（internalRep）：整数・倍精度・リスト構造・コンパイル
  済みコードなど、**直近の使われ方に最適な**姿と、その型へのポインタ

どちらか一方は**空でもよく**、要求されたときに他方から生成して
キャッシュします。整数として使い続ける限り文字列表現は作られず
（`incr` は内部表現の整数を直接更新し、文字列表現を**無効化**する
だけ）、表示が要求された瞬間に初めて `"43"` が生成されます。

```ruby
# 概念図：Tcl_Obj の二重表現
class TclObj
  def initialize(str) = (@str = str; @int = nil)

  def as_int                 # 整数として使う
    @int ||= Integer(@str)   # 必要なら文字列からパースしてキャッシュ
  end

  def incr!                  # 整数演算は内部表現を直接更新
    v = as_int + 1
    @int = v
    @str = nil               # 文字列表現は「古くなった」ので無効化
  end

  def to_s = (@str ||= @int.to_s)   # 要求されたら再生成
end
```

これは[遅延評価の章](lazy.md)のサンク（必要になるまで作らず、作ったら記憶
する）の応用であり、CRuby のコードレンジ（[文字列の章](strings.md)）や V8 の
ConsString の平坦化と同じ「**遅延＋キャッシュ**」ファミリーです。
リストも同様で、`{1 2 3}` は最初にリスト操作された時点でパース
されて内部の配列になり、以後のリスト操作は配列を直接触ります。
コードも `eval` された時点でバイトコードにコンパイルされ、内部
表現として値にぶら下がります —— **「コンパイル結果のキャッシュ」が
値の属性になっている**わけです。

## shimmering：二重表現の病理

二重表現には固有の病があります。**shimmering**（シマリング、
ちらつき）です。内部表現の置き場は**一つ**しかないので、同じ値を
整数として使い、次にリストとして使い、また整数として…と交互に
使うと、そのたびに内部表現が**破棄され、文字列経由で再構築され**
続けます。

```tcl
# x をリストとして触り、次の行で数値として触る……を繰り返すと、
# 内部表現が list ⇄ int で往復し、毎回パースが走る（shimmering）
```

[配列の章](arrays.md)で見た V8 の要素種別は、表現の遷移を**一方向**（より一般
へ）に制限することで、この往復を構造的に封じていました。Tcl は
EIAS を守るために往復可能にしたので、性能の落とし穴として残った
のです。「複数の表現を持つ値」を設計するときの遷移図 —— 一方向か、
往復か、格子か —— が性能特性を決める、という一般法則の好例です。

## 文字列が単位だからこその設計

EIAS は他の構成要素の設計も規定しました。Tcl の連想配列や `dict` は
**キーも値も文字列**のハッシュ表で、[ハッシュの章](hashes.md)で見た文字列ハッシュ
関数の品質がそのまま全体の性能を左右します。また「コードも文字列」
である以上、構文木を介さず**文字列の置換**（変数展開・コマンド置換）が
評価の基本操作になり、クォート規則（`{}` は展開抑制、`""` は展開）が
言語の文法のほぼすべてを占めます。[構文木の章](syntax-tree.md)で見た「プログラムを
木として持つ」道をあえて捨てると、評価器は「文字列の分割と置換の
規則」に還元される —— shell（POSIX sh）もこの同族で、shell の
クォート地獄と Tcl のクォート規則が似ているのは偶然ではありません。

## 系譜と教訓

- **二重表現（遅延＋キャッシュ付きの複数表現）**は、CRuby の
  コードレンジ、V8 の文字列の動物園、そして「表示用と計算用の
  二つの顔を持つ値」一般の設計パターンへ
- **すべてを文字列で交換する**思想は、JSON/HTTP の時代の
  「テキストプロトコル」文化、設定言語、シェルへ
- **shimmering** は、複数表現を持つ値の遷移設計（一方向 vs 往復）と
  いう教訓へ

「意味論の美しさ（EIAS）を、実装のデータ構造（Tcl_Obj）で買い戻す」
—— 意味論と実装を分離して両立させた手並みは、動的言語の実装史でも
指折りの鮮やかさです。次の章では、評価の規則そのものが変わっている
言語 —— 失敗が制御構造の Icon を訪ねます。
