# 正規表現型：型の中に住む小さな処理系

## パターンはプログラムである

**正規表現**（regular expression）は、`/\d+/`（1 文字以上の数字）の
ようなパターンで文字列を検索・照合する道具で、ほぼすべての言語に
組み込まれています。

```ruby
p "order-42" =~ /\d+/        # => 6   （数字が 6 文字目から始まる）
p "2026-06-08".scan(/\d+/)   # => ["2026", "06", "08"]
```

正規表現が本書の締めくくり近くに置かれているのには理由があります。
`/\d+/` という**パターンそれ自体が小さなプログラム**であり、正規表現型の
実装は**言語処理系のミニチュア**だからです。パターンの文字列は字句解析・
構文解析されて**構文木**になり、**コンパイル**されて内部表現になり、
**実行器**（マッチングエンジン）がそれを動かす。第I部で見た流れが、
そっくりそのまま型の中に入っています。

実際、Ruby で正規表現オブジェクトを作る `Regexp.new(str)` は、
小さなコンパイラの起動にほかなりません。だからこそ、ループの中で
毎回 `Regexp.new` する（毎回コンパイルする）のは無駄で、リテラル
`/.../o` や定数に切り出してコンパイル結果を再利用せよ、という
定番の最適化助言が生まれます。

## パターンの構文木と、機械への二つの道

`a(b|c)*d` というパターンは、まず構文木になります。連接・選択（`|`）・
繰り返し（`*`）を内部ノードに、文字を葉に持つ、構文木の章で見たままの
木です。問題は、この木を**どう実行するか**。ここで道が二つに分かれ、
その分岐が現代の正規表現エンジンの勢力図を決めています。

一つ目の道が**有限オートマトン**（finite automaton、有限状態機械）です。
いくつかの**状態**と、入力文字に応じた状態間の**遷移**からなる機械で、
文字列を 1 文字ずつ読みながら状態を移り、**受理状態**に到達すれば
マッチです。

```mermaid
graph LR
    S0((開始)) -->|a| S1((1))
    S1 -->|b| S2(((受理)))
```

オートマトンには二種類あります。

- **NFA**（非決定性有限オートマトン）：同じ文字で複数の遷移先があり
  うる、または文字を読まずに移れる（ε遷移）もの。「非決定性」とは
  「次にどこへ行くか一通りに決まらない」という意味です。
- **DFA**（決定性有限オートマトン）：どの状態でも、各文字に対する
  遷移先がちょうど一つに決まるもの。

正規表現から NFA への変換は、Thompson の古典的な構成法で機械的に
できます [](#cite:thompson1968)。連接・選択・繰り返しという構文木の
各ノードを、対応する小さな NFA の部品に変換し、ε遷移でつなぎ合わせる ——
「部品から再帰的に全体を組む」、構文木の章と同じ構図です。

```ruby
# Thompson 構成法のイメージ：状態と遷移を表すノードをつなぐ
State = Struct.new(:transitions, :accepting)  # transitions: 文字 => 次状態

def literal_nfa(char)            # 1 文字を読む最小の NFA
  accept = State.new({}, true)
  start  = State.new({ char => accept }, false)
  [start, accept]
end
# 選択 a|b は「両方の開始へ ε で分岐する新しい開始状態」を作る、など
```

NFA は「ありうる状態の集合」を同時に追えば実行でき（後述）、
**部分集合構成法**で DFA に変換すれば 1 状態だけ追う高速実行も
できます。ただし DFA は状態数が最悪で指数的に爆発します。

二つ目の道が**バックトラック**（backtracking、後戻り）です。構文木を
深さ優先で素直にたどり、「選択肢があったら一つ目を試し、失敗したら
戻って二つ目を試す」を繰り返します。実装は再帰関数一本で書ける
ほど素直で、しかも**正規言語を超えた機能**——後方参照（`\1`、前に
捕捉した文字列の再出現）や先読み（`(?=...)`）——も自然に実現できます。

## 実用エンジンの内部：バックトラックVM

Perl・Python・Java・JavaScript（V8 の既定）・Ruby と、主要言語の
正規表現エンジンの多数派はバックトラック方式です。そしてその実装は、
木を直接たどるのではなく、**パターンを専用のバイトコードにコンパイル
して仮想機械で実行**します [](#cite:cox2009)。Ruby のエンジン
Onigmo も、パターンを「この文字列に一致せよ」「ここに選択肢の分岐
ポイントを積め」「N 回繰り返せ」といった命令列に変換し、マッチング
VM がそれを解釈します。

```
# /a(b|c)*d/ のバイトコードのイメージ
0: char 'a'          # 'a' に一致せよ
1: push_alt 5        # 失敗したら 5 へ戻る、という分岐点を積む
2: char 'b'          # まず 'b' を試す
3: jump 1            # 繰り返しの先頭へ
5: ...               # 'c' 側の選択肢…
9: char 'd'
```

「失敗したら戻る」ためのデータ構造が**バックトラックスタック**です。
分岐点を通るたびに「入力のこの位置・命令のこの位置から再開できる」と
いう記録を積み、失敗したら直近の記録まで巻き戻します。スタックと
フレームの章で見た「実行状態の保存と復元」が、ここでは 1 つの型の
内部で行われているわけです。捕捉グループ（`(...)` が何文字目から
何文字目に一致したか）の記録も、このスタックと一緒に保存・復元される
**位置の配列**として実装され、マッチ成功時には `MatchData`
オブジェクト（位置の配列＋元文字列への参照）として利用者に返されます。

## 破滅的バックトラックと ReDoS

バックトラックの代償は、最悪計算量です。選択肢の組み合わせは入力長に
対して**指数的**に増えうるのです。悪名高い例が `/(a+)+$/` です。

```ruby
# 危険な例：「a の繰り返し」の繰り返し、にマッチしない入力を与える
# /(a+)+$/ =~ "a" * 30 + "b"
# 「どこで a+ を区切るか」の全組み合わせ（2^30 通り）を試してしまう
```

`"aaa...b"` は最終的にマッチしないのですが、エンジンは「a の列を
内側の `+` と外側の `+` でどう区切るか」のあらゆる組み合わせを
試してから失敗を報告します。これを**破滅的バックトラック**
（catastrophic backtracking）と呼び、悪意の入力でサービスを停止させる
攻撃を **ReDoS**（Regular expression Denial of Service）と呼びます。
ハッシュ衝突攻撃（ハッシュの章）と並ぶ、「データ構造・アルゴリズムの
最悪ケースを突く攻撃」の代表です。

対策は三つの系統に分かれます。

1. **時間で打ち切る**：Ruby 3.2 の `Regexp.timeout`、.NET の
   `MatchTimeout`。根治ではないが確実な安全網です [](#cite:ruby32)。
2. **記憶して刈る**：同じ「入力位置×命令位置」の組を二度試すのは
   無駄なので、試した組を**メモ化**（ビットセットで記録）して
   二度目を即座に失敗させます。Ruby 3.2 はこの最適化を Onigmo に
   組み込み、後方参照などを含まない大半のパターンで照合時間が
   入力長に対して線形になりました [](#cite:ruby32)。指数的な探索木が、
   メモ化という表一枚で多項式に潰れる —— 動的計画法の教科書例の
   ような改善です。
3. **そもそもオートマトンで走る**：次節の線形エンジンです。

## 線形時間エンジン：Thompson の遺産

Cox は 2007 年の論文記事で、「Thompson 流の NFA 実行なら、どんな
パターンでも入力長に比例した時間で照合できるのに、主要言語は指数的な
バックトラックを使い続けている」と指摘し、大きな影響を与えました
[](#cite:cox2007)。

NFA の線形実行（**Pike VM** と呼ばれます）の核は、「ありうる実行状態の
**集合**を、入力を 1 文字進めるごとにまとめて更新する」ことです
[](#cite:cox2009)。状態集合のサイズはパターンの長さで抑えられるので、
全体は O(パターン長 × 入力長)。さらに各「スレッド」（NFA 上の実行枝）に
捕捉位置の配列を持たせることで、バックトラックなしで捕捉グループも
扱えます。

もっと速くしたければ DFA ですが、事前の部分集合構成は状態爆発の
危険があります。そこで Google の **RE2** や Rust の `regex` クレートは
**遅延 DFA**（lazy DFA）を使います。DFA の状態（＝NFA 状態の集合）を
**実際に遷移した分だけ実行時に作ってキャッシュ**し、キャッシュが
あふれたら捨てて作り直す。メモリ上限を保証しながら、実用入力では
DFA の速度で走る、という折衷です。

Pike VM は短く実装できるので、骨格を載せておきます。「スレッド」とは
ここでは NFA 上の実行枝のことで、命令位置（pc）だけを持ちます。

```ruby
# Pike VM の骨格：プログラムは [op, 引数...] の配列
# op は :char(1文字一致), :split(二方向に分岐), :jmp, :match
def pike_match?(prog, input)
  threads = add_thread([], prog, 0)        # 現在の状態集合
  input.each_char do |c|
    next_threads = []
    threads.each do |pc|
      op, a, b = prog[pc]
      if op == :char && a == c             # 一致したスレッドだけが
        add_thread(next_threads, prog, pc + 1)  # 次の文字へ進む
      end
    end
    threads = next_threads
    return false if threads.empty?
  end
  threads.any? { |pc| prog[pc][0] == :match }
end

def add_thread(list, prog, pc)             # ε遷移(split/jmp)を畳み込む
  return list if list.include?(pc)         # 重複排除が線形性の要！
  op, a, b = prog[pc]
  case op
  when :jmp   then add_thread(list, prog, a)
  when :split then add_thread(add_thread(list, prog, a), prog, b)
  else list << pc
  end
  list
end
```

急所は `add_thread` の重複排除です。状態集合の要素数はプログラム長を
超えられないので、入力 1 文字あたりの仕事量が O(プログラム長) で
頭打ちになり、全体で線形が保証されます。捕捉グループ対応は、pc に
加えて「捕捉位置の配列」をスレッドに持たせるだけです（同じ pc に
複数スレッドが来たら、優先順位の高い方だけ残します）。

残る部品は「パターンの構文木 → この命令列」のコンパイラですが、
これも Thompson 構成をそのまま命令の発行に置き換えるだけです。
構文木の章で予告した**バックパッチ**（飛び先をあとから埋める）が、
ここで実際に登場します。

```ruby
# 構文木 → Pike VM 命令列（Thompson 構成のバイトコード版）
def rcompile(re, prog = [])
  case re
  in [:char, c]   then prog << [:char, c]
  in [:cat, a, b] then rcompile(a, prog)
                       rcompile(b, prog)
  in [:alt, a, b]
    sp = prog.size
    prog << [:split, nil, nil]      # 飛び先は未定のまま置く
    rcompile(a, prog)
    jp = prog.size
    prog << [:jmp, nil]
    prog[sp][1] = sp + 1
    prog[sp][2] = prog.size         # ← b の開始位置が分かった瞬間に埋める
    rcompile(b, prog)
    prog[jp][1] = prog.size         # ← 合流点をあとから埋める（バックパッチ）
  in [:star, a]
    sp = prog.size
    prog << [:split, sp + 1, nil]
    rcompile(a, prog)
    prog << [:jmp, sp]
    prog[sp][2] = prog.size         # ループ脱出先を埋める
  end
  prog
end

def compile_regex(re) = rcompile(re) << [:match]

re = [:cat, [:char, "a"],
      [:cat, [:star, [:alt, [:char, "b"], [:char, "c"]]], [:char, "d"]]]
p pike_match?(compile_regex(re), "abcbd")   # => true
```

構文木（パース）、コンパイル（バックパッチ付き命令生成）、実行器
（Pike VM）—— 100 行に満たないこの三点セットは、そのまま本書
第I部のミニチュアです。手元で拡張する（`?` や `+` を足す、捕捉を
足す）には最適の演習になります。

| 方式 | 時間 | 後方参照等 | 採用例 |
|---|---|---|---|
| バックトラック VM | 最悪指数 | ○ | Perl, Python, Java, Onigmo, V8 |
| バックトラック＋メモ化 | ほぼ線形 | ○ | Ruby 3.2+ |
| Pike VM（NFA 集合実行） | 線形 | ×（捕捉は○） | RE2/Rust の捕捉系 |
| 遅延 DFA | 線形・高速 | × | RE2, Rust regex, Go |

## もう一つの伝統：正規表現の微分

オートマトン構成には、Thompson 構成と並ぶもう一つの古典があります。
Brzozowski の**微分**（derivative）です [](#cite:brzozowski1964)。
正規表現 r の文字 c による微分 ∂c(r) とは、「**r に一致する文字列の
うち、c で始まるものの、c を取り除いた残り**」に一致する正規表現の
ことです。定義は再帰的な書き換え規則そのもので、驚くほど実装が
簡単です。

```ruby
# 正規表現の微分（抜粋）。re は構文木（本章冒頭のノード）
def deriv(re, c)
  case re
  in [:char, ch]    then ch == c ? [:eps] : [:empty]   # 1文字
  in [:cat, a, b]   then                                # 連接 ab
    d = [:alt, [:cat, deriv(a, c), b],
               nullable?(a) ? deriv(b, c) : [:empty]]   # a が空を許すなら b 側も
    simplify(d)
  in [:alt, a, b]   then simplify([:alt, deriv(a, c), deriv(b, c)])
  in [:star, a]     then simplify([:cat, deriv(a, c), re])  # a* は a の微分 + a*
  in [:eps] | [:empty] then [:empty]
  end
end

def match?(re, str)
  str.each_char { |c| re = deriv(re, c) }   # 1 文字ずつ「微分」していき
  nullable?(re)                              # 最後に空文字列を許せば一致
end
```

マッチングは「1 文字読むごとに正規表現自体を書き換えていく」操作に
なり、NFA もスタックも登場しません。さらに、微分して現れる正規表現の
種類は（簡約をきちんとすれば）有限なので、**「正規表現 → 微分結果」を
ノードとする DFA がそのまま生成**できます。部分集合構成より直接的な
DFA 構成法です。

この 1964 年の手法は長く忘れられていましたが、2009 年の再評価論文
[](#cite:owens2009) を機に復権しました。実装の簡潔さと正しさの
証明しやすさから、検証済み正規表現エンジンや、否定・積（`a` かつ
`b`）のような拡張演算を扱うエンジン（微分なら否定も積も自然に
定義できます —— オートマトン構成では厄介な演算です）で使われて
います。「データ構造（オートマトン）を作る」のではなく「式を
書き換え続ける」という、いかにも関数型的な別解です。

「機能（後方参照）を取るか、最悪時間の保証を取るか」という
トレードオフがエンジンの設計を二分しており、近年は V8 の線形モードや
.NET 7 の `NonBacktracking` オプションのように、**一つの言語が両方の
エンジンを併載して切り替える**折衷が広がっています。

## 文字クラスと Unicode：表の中の表

`/[a-z]/` のような**文字クラス**は、データ構造としては**文字集合**です。
ASCII の範囲ならビットセット（リスト・タプル・集合の章で見た 256 ビットの
表）で 1 命令判定ができます。しかし Unicode の `\p{Letter}`（あらゆる
言語の「文字」）は約 14 万コードポイントを含み、素朴なビットセットでは
17KB を超えます。実装は **コードポイントの区間の整列済み配列**
（`[a-z]` を `(97, 122)` のような区間として持ち、判定は二分探索）や、
**多段のテーブル**（上位ビットで一段目を引き、下位ビットで二段目を
引く、ページテーブルに似た構造）で、サイズと速度を両立させます
[](#cite:unicode2023)。大文字小文字を無視する照合（`/i`）も、Unicode の
ケース対応表（`ß` ↔ `SS` のような多対多の対応！）を引く立派な
データ構造問題です。

## エンジンの外側の高速化：先回りの検索

最後に、見落とされがちな実戦テクニックを一つ。実用の正規表現検索の
速さは、オートマトンの理論だけでは決まりません。**「マッチが始まり
そうな位置」へ一気に跳ぶ前処理**が効くのです。

たとえば `/error: \d+/` を 1GB のログから探すとき、賢いエンジンは
まず「必ず `error: ` という固定文字列を含む」ことをパターンから抽出し、
SIMD 命令で高速化された部分文字列検索（`memchr` など）で候補位置だけを
拾い、そこだけ正規表現エンジンを起動します。Rust の regex クレートが
極端に速い主因はこの**リテラル先回り**にあります。
「重い機械（オートマトン）の手前に、軽い篩（文字列検索）を置く」——
ハッシュの章の SwissTable（重いキー比較の前の 1 バイト篩）と同じ
**二段構えの篩**パターンです。

## まとめ：データ構造の選択が、機能・速度・安全を決める

正規表現型の中には、

- パターンの**構文木**
- コンパイル結果の**バイトコード**（または NFA/DFA の遷移表）
- 実行時の**バックトラックスタック**／**状態集合**／**遅延 DFA
  キャッシュ**／**メモ化表**
- 文字クラスの**区間配列・多段テーブル**
- 捕捉位置の**配列**（`MatchData`）

という、本書で学んだデータ構造の縮図が詰まっていました。そして
「どの実行用データ構造を選ぶか」が、後方参照という**機能**、照合の
**速度**、ReDoS への**安全性**を同時に決めてしまう —— 「型の中に
小さな処理系が住んでいる」とは、設計判断ごと住んでいるという意味
なのです。

次の章では、ヒープの中で生きてきたデータ構造たちを**プロセスの外へ
持ち出す**技術 —— 直列化と永続化を見ていきます。
