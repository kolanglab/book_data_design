# JITコンパイラのデータ構造：観測、投機、後戻り

## インタプリタの限界と JIT

本章で扱うのは、現代の高速処理系の心臓部 —— **JIT コンパイラ**
（Just-In-Time compiler、実行時コンパイラ）です。バイトコードの解釈
実行（構文木と中間表現の章）は、1 命令ごとに「命令を読む→分岐する」
というディスパッチの固定費を払い続けます。JIT はよく実行される部分を
実行中に機械語へコンパイルし、この固定費を消します。V8・JVM の
HotSpot・PyPy、そして Ruby の YJIT がこの系譜です。

JIT は「コンパイラの話」に見えますが、その実、独特のデータ構造の
塊です。**何をコンパイルすべきかを知る構造**（プロファイル）、
**型が分からないのに速いコードを出すための構造**（投機とガード）、
**賭けに負けたとき安全に戻るための構造**（デオプティマイズ情報）。
本章はこの三つを軸に解剖します。

## 観測する：カウンタとフィードバックベクタ

JIT の第一の問いは「**どこを**コンパイルするか」です。コンパイルは
高くつくので、元が取れる場所 —— ホットスポット —— に限る必要が
あります。答えは単純で、**数える**ことです。

- **呼び出しカウンタ**：メソッドごとに呼ばれた回数を数え、閾値
  （HotSpot のデフォルトは数千〜1万回）を超えたらコンパイル対象に
  します。
- **ループ（バックエッジ）カウンタ**：後ろ向きのジャンプを数えます。
  「1 回しか呼ばれないが中で 100 万回回るループ」を救うためです。

次の問いは「**何を前提に**コンパイルするか」です。動的言語では
`a + b` の意味が型で変わるので、型が分からなければ遅いコードしか
出せません。そこでインタプリタ実行の間に、**呼び出し箇所ごとの
観測記録**を残します。V8 はこれを**フィードバックベクタ**
（feedback vector）と呼びます —— 関数ごとに「この呼び出し箇所の
レシーバはいつも同じ形（シェイプ —— オブジェクトの隠れた構造情報。
オブジェクト型の章で詳述します）だった」「この加算は常に SMI 同士
だった」を記録するスロットの配列で、オブジェクト型の章で詳しく見る
**インラインキャッシュ**（呼び出し箇所ごとに前回の探索結果を覚えて
おく仕掛け）を、**コンパイラへの報告書**として再利用したものです。CPython 3.11 の
特殊化適応型インタプリタも同じ思想で、バイトコード列に埋め込んだ
キャッシュ領域に観測結果を書き込み、命令自体を特殊化版
（`BINARY_OP_ADD_INT` など）に**置き換えて**いきます
[](#cite:pep659)。

quickening（構文木と中間表現の章）の核 ——「命令が観測結果で
自分自身を書き換える」—— は、玩具で再現できます。

```ruby
# 自己書き換えする加算命令（quickening の玩具）
def execute(code, stack)
  pc = 0
  while (insn = code[pc])
    case insn[0]
    when :push then stack << insn[1]
    when :add                            # 汎用版：型を調べてから足す
      b, a = stack.pop, stack.pop
      code[pc] = [:add_int] if a.is_a?(Integer) && b.is_a?(Integer)
      stack << a + b                     # ↑ 観測した型で命令列を書き換える
    when :add_int                        # 特殊化版：整数前提の高速経路
      b, a = stack.pop, stack.pop
      code[pc] = [:add] unless a.is_a?(Integer) && b.is_a?(Integer)
      stack << a + b                     # ↑ 賭けが外れたら汎用版へ戻す
    end
    pc += 1
  end
  stack.pop
end

code = [[:push, 1], [:push, 2], [:add]]
p execute(code, [])    # => 3
p code[2]              # => [:add_int]  命令列が書き換わっている！
```

本物（CPython 3.11 など）の `add_int` は型検査の後に**タグ付きの
まま加算する**ような専用機械語級の経路を通るので利得が出ます。
ここで見るべきは仕掛けのほう —— **命令列という配列が、実行統計を
記録する可変のデータ構造を兼ねている**ことです。

> [!NOTE]
> ここに美しい再利用があります。インラインキャッシュ
> [](#cite:deutsch1984) はもともと「2 回目以降を速くする」ための
> 仕掛けでしたが、その中身は「ここに何が来たか」の記録でもあり
> ます。PIC（複数クラスぶんの結果を覚える多態版インラインキャッシュ）
> [](#cite:holzle1991) に貯まった数件のクラスは、JIT に
> とって最高の型プロファイルです。**キャッシュは観測装置を兼ねる**
> —— SELF の時代に発見されたこの二重性が、現代のすべての JIT の
> 土台にあります。

## 投機する：ガード付きの賭け

観測の結果「この加算は 99% 整数同士」と分かったとします。JIT は
**整数前提の機械語**を出したい。しかし 1% の例外（巨大数や文字列が
来る）を無視はできません。答えが**投機**（speculation）と**ガード**
（guard）です。

```
# 「a + b は Fixnum 同士」と投機したコードのイメージ
  test  a, 1        ; a の最下位ビットは 1 か？（値の表現の章のタグ検査）
  jz    side_exit   ; 違ったら脱出
  test  b, 1
  jz    side_exit
  lea   r, [a + b - 1]  ; タグ付きのまま加算（数値型の章のトリック）
  jo    side_exit   ; あふれたら脱出
```

本線は最速の数命令、想定外はすべて `side_exit` へ。問題は、
**脱出した先でどう実行を続けるか**です。ここからが JIT の
データ構造の真骨頂になります。

## 後戻りする：デオプティマイズという発明

脱出先で待っているのが**デオプティマイズ**（deoptimization、脱最適化）
—— コンパイル済みコードの実行を捨てて、**同じ地点のインタプリタ実行に
乗り換える**機構です。SELF の Hölzle らが「最適化コードのデバッグ」の
ために発明し [](#cite:holzle1992)、いまや動的言語 JIT の基盤技術に
なっています。

難しさを考えてみましょう。最適化コードの中の値は、レジスタに散らばり、
タグを外され（生の整数として）、ときには**オブジェクトの生成自体が
省略**されています（**エスケープ解析** —— 値がフレームの外へ
逃げ出すかどうかの静的解析。「クロージャと関数」の章で再登場します ——
で「逃げない」と分かったオブジェクトは、フィールドだけがレジスタに
載っています）。一方、
インタプリタが再開するには、スタックとフレームの章で見た**バイト
コード用フレーム** —— ローカル変数表と値スタック —— が、正しい
中身で**存在しなければなりません**。

橋渡しをするのが、コンパイラが脱出点ごとに残す**デオプト情報**
（deoptimization metadata、HotSpot では scope descriptor、V8 では
deoptimization data）です。その実体は対応表です。

```
# 脱出点 #17 のデオプト情報（概念図）
バイトコード位置: 0042
ローカル変数 x  : レジスタ r8 に生の整数として（タグを付け直せ）
ローカル変数 y  : スタックの [sp+16]
ローカル変数 p  : 省略された Point オブジェクト
                  （@x = レジスタ r9, @y = 定数 0 から再構成せよ）
値スタック[0]   : 定数 42
```

脱出時には、この表に従ってフレームを**その場で組み立て直し**
（オブジェクトの再構成まで行います —— 投機で消したものを実体化
するわけです）、インタプリタへ制御を渡します。インライン展開
されていた場合は、**一つの機械語フレームから複数のバイトコード
フレームを復元**します。デオプト情報は「機械語の世界とバイト
コードの世界の対応を、脱出可能なすべての点について記録した辞書」
であり、JIT のメモリ消費の大きな部分を占めるため、各処理系は
差分圧縮などで小さくする工夫を重ねています。

> [!IMPORTANT]
> デオプティマイズがあるからこそ、JIT は**大胆に**なれます。
> 「ガードに失敗したら全部やり直せる」という安全網が、タグ外し・
> インライン展開・アロケーション除去という攻撃的な最適化を
> 正当化するのです。オブジェクト型の章で詳しく見るキャッシュ無効化
> （メソッド再定義の世代番号）も、JIT の世界では「該当する前提で
> コンパイルされたコードを**まとめてデオプト対象にする**」という
> 形で実装されます。前提と、それに依存するコードの集合を結ぶ
> **依存性の表**（HotSpot の dependency tracking）も、JIT の
> 重要な台帳の一つです。

## OSR：走っているループの乗り換え

逆方向の乗り換えもあります。**OSR**（On-Stack Replacement、
スタック上置換）—— インタプリタで回っている**最中の**ループを、
コンパイルできた瞬間に機械語版へ乗り換える機構です。デオプトの
逆向きで、インタプリタのフレームから機械語フレームを組み立てます。
「100 万回ループの 1 万回目でコンパイルが終わったのに、今回の
呼び出しでは使われない」という間抜けを防ぐための仕組みで、
カウンタ（ループの回数）とフレーム対応表という、ここまでに出た
部品の組み合わせでできています。

## 段階を持つ：ティアリング

観測の精度とコンパイルの速さは両立しません。そこで現代の VM は
**多段の実行エンジン**（tiering）を持ちます。V8 なら、
インタプリタ（Ignition）→ 簡易 JIT → 最適化 JIT（TurboFan）という
階段で、上の段ほど起動が遅く実行が速い。各段が**同じフィードバック
ベクタを共有**し、下の段が貯めた観測を上の段が使います。
HotSpot も C1（速いコンパイラ）と C2（賢いコンパイラ）の二段構えです。

ここで「コンパイル済みコード自体の置き場」も問題になります。
**コードキャッシュ**（code cache）—— 実行可能メモリ領域に機械語を
並べ、世代やティアごとに区画を分け、デオプトで死んだコードを
回収する、**機械語のためのヒープ**です。メモリ管理の章の知識が
そのまま、コードという特殊なオブジェクトに適用されます（移動は
できないので、フリーリストと区画管理が主役です）。GC との連携も
要ります。機械語の中に埋め込まれたオブジェクト参照（定数など）を
GC が見つけ・書き換えられるよう、**コード内の参照位置の表**を
コードごとに添えておくのです。

## もう一つの流儀：トレーシングJIT

ここまで暗黙に「メソッド（関数）単位でコンパイルする」前提で話して
きました（**メソッド JIT**）。これと並ぶもう一つの流儀が
**トレーシング JIT**（tracing JIT）です。LuaJIT と PyPy が代表で、
コンパイルの単位を関数ではなく**実際に実行された経路（トレース）**に
します。

仕組みはこうです。ループのカウンタが閾値を超えたら、インタプリタが
**記録モード**に入り、次の 1 周で実行した操作を**分岐のない一本の
命令列**として書き取ります。途中の条件分岐は「こちら側に進んだ」と
いう**ガード**に変わります。メソッド呼び出しは記録の中に**自動的に
インライン展開**されます（呼び出しの先まで含めて「通った道」を
記録するだけだからです）。出来上がるのは、分岐も合流もない線形の
IR —— 最適化が劇的に簡単な形です。ガードが失敗したらサイドへ抜けて
インタプリタに戻り（デオプトと同じ仕掛けです）、同じガードで何度も
抜けるようなら、そこから先を**別のトレース**として継ぎ足します
（trace tree）。

PyPy はこれをさらに一段抽象化しました。**メタトレーシング**
（meta-tracing）[](#cite:bolz2009) —— ユーザープログラムではなく、
**インタプリタ自身の実行をトレースする**のです。RPython という制限
付き Python で「素直な」インタプリタを書けば、トレーシング JIT は
フレームワークが自動付与してくれる。「言語ごとに JIT を書く」を
「JIT 付きのインタプリタ生成器を一つ書く」に置き換えた、実装コスト
の構造を変えた研究です。トレースという線形のデータ構造を選んだ
からこそ成立した自動化、と言えます。

メソッド JIT（広く効くが分岐の多いコードもまるごと相手にする）と
トレーシング JIT（ホットなループ経路には無類に強いが、経路が
散らばるコードではトレースが爆発する）の優劣は長く論争になり、
現在の主流（V8・JVM・YJIT）はメソッド JIT 側に寄りました。YJIT の
LBBV（次節）は、その中間 —— ブロック単位で「通った道」に追従する ——
に位置づけられます。

## YJIT：Lazy Basic Block Versioning

Ruby の YJIT は、この章の道具立てを独自の角度で組み合わせた設計
です。土台は **Lazy Basic Block Versioning**（LBBV）という手法で
[](#cite:chevalier2015)、「メソッド全体を一気にコンパイルする」
代わりに、**基本ブロック（分岐のない命令の並び）を、実行が
到達した瞬間に、その時点で分かっている型情報の文脈つきで**
コンパイルします。

- ブロックの出口に「次のブロックへのスタブ」を置く。実行が
  スタブに到達して初めて、続きをコンパイルする（**遅延**）
- 同じブロックでも、入ってくる型文脈が違えば**別バージョン**を
  生成する（`x` が Fixnum と分かって入る版と、不明のまま入る版）

つまり LBBV は、**プロファイル段階を省略**し、「実行が通った
経路と、そのとき確定していた型」そのものを観測として使います。
フィードバックベクタを貯めてから賭けるのではなく、走りながら
配牌を見て手を作っていく麻雀のような方式で、ウォームアップが
速いのが持ち味です。バージョンの管理表（ブロック×型文脈 →
機械語）と、未コンパイル領域への**スタブ**（呼ばれたらコンパイラを
起動する番人）が中核のデータ構造になります。

## まとめ：JIT は台帳でできている

本章で出てきた構造を並べます。

| 構造 | 役割 |
|---|---|
| 呼び出し・ループカウンタ | どこをコンパイルするか |
| フィードバックベクタ／IC | 何を前提にするか（観測の記録） |
| ガード付き機械語 | 投機の本線と脱出口 |
| デオプト情報 | 機械語世界 → バイトコード世界の対応表 |
| 依存性の表 | 前提が崩れたとき無効化すべきコードの集合 |
| コードキャッシュ＋参照位置表 | 機械語のためのヒープと GC 連携 |
| ブロックバージョン表とスタブ | LBBV の遅延コンパイル |

JIT とは、コンパイラ技術である以上に、**「観測 → 賭け → 清算」の
サイクルを回すための台帳システム**です。そして台帳の中身は、
カウンタ、配列、辞書、対応表 —— 本書でおなじみの顔ぶれに
ほかなりません。

第I部の締めくくりとなる次の章では、視点をプログラム一枚の内側から
外へ広げます。数百のファイルに分かれたプログラムを、処理系はどう
探し、どう順序づけ、どう組み立てるのか —— モジュールとロードの
話です。
