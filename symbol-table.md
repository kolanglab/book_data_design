# シンボルテーブル：名前から情報を引く表

## 表（テーブル）という考え方

処理系がまず必要とするのは、「名前から情報を引く」しくみです。
たとえば次の Ruby のコードを処理系の立場で考えてみましょう。

```ruby
x = 10
y = x + 1
```

2行目を処理するとき、処理系は `x` という名前が「すでに宣言されている変数で、
整数を持つ」ことを知っていなければなりません。この「名前 → 情報」の
対応を覚えておくデータ構造を**シンボルテーブル**（symbol table、記号表）と
呼びます。名前から値を引く構造を一般に**連想配列**（associative array）や
**テーブル型**（table）と呼び、シンボルテーブルはその代表例です。

シンボルテーブルに格納される「情報」は、名前ごとの**属性**（attribute）の
集まりです。変数なら型・スコープ・メモリ上の位置、関数なら引数の個数と型・
戻り値の型、といった具合です。これら一件ぶんの情報をまとめたものを
**エントリ**（entry）と呼びます。コンパイラの各フェーズはこの表を共有し、
名前について知っていることを少しずつ書き加えていきます [](#cite:aho2006)。

```ruby
# シンボルテーブルのエントリを表す素朴な構造体
Entry = Struct.new(:name, :kind, :type, :location)

table = {}
table["x"] = Entry.new("x", :variable, :integer, 0)
table["y"] = Entry.new("y", :variable, :integer, 1)

p table["x"].type   # => :integer
```

ここでは Ruby の `Hash` をそのまま表として使いました。実務の処理系でも、
シンボルテーブルの中身は結局のところ**ハッシュ表**（hash table）であることが
多いです。では、そのハッシュ表はどう実装されているのでしょうか。
本書ではハッシュの内部を第7章で詳しく扱いますが、ここではシンボルテーブルを
理解するのに必要な最小限を押さえておきます。

## ハッシュ表で表を引く

ハッシュ表は、キー（ここでは名前の文字列）を**ハッシュ関数**
（hash function）で整数に変換し、その整数を配列の添字として使うことで、
平均的に**一定時間** O(1) で読み書きできるデータ構造です [](#cite:knuth1998)。
配列の各マスを**バケット**（bucket）と呼びます。

```ruby
class SymbolTable
  def initialize(size = 16)
    @buckets = Array.new(size) { [] }  # 各バケットは [名前, 値] の配列
  end

  def hash_index(name)
    # 文字コードを足し合わせる素朴なハッシュ関数
    name.bytes.sum % @buckets.size
  end

  def []=(name, value)
    bucket = @buckets[hash_index(name)]
    pair = bucket.find { |k, _| k == name }
    if pair
      pair[1] = value          # 既存エントリの更新
    else
      bucket << [name, value]  # 新規エントリの追加
    end
  end

  def [](name)
    bucket = @buckets[hash_index(name)]
    pair = bucket.find { |k, _| k == name }
    pair && pair[1]
  end
end
```

異なる名前がたまたま同じバケットに割り当てられることを**衝突**
（collision）と呼びます。上の実装では、同じバケットに複数の `[名前, 値]` を
並べた配列を持たせることで衝突に対処しています。これを**チェイン法**
（chaining、連鎖法）と呼びます。衝突したエントリを連結リストでつなぐ
イメージです。

> [!NOTE]
> 衝突の対処にはもう一つ**オープンアドレス法**（open addressing）があります。
> 衝突したら別の空きバケットを探して入れる方式で、配列だけで完結するため
> メモリ効率とキャッシュ効率に優れます。詳しくは第7章で扱います。

ハッシュ関数の質は表の性能を大きく左右します。上の `name.bytes.sum` は
理解のための例で、`"abc"` と `"cba"` が同じ値になるなど衝突しやすい
欠点があります。実用のハッシュ関数は、入力のわずかな違いが出力全体に
散らばるよう設計されます。Knuth はハッシュ法と各種探索法を体系的に
分析しています [](#cite:knuth1998)。

## スコープを表す：入れ子になった表

シンボルテーブルが単なる辞書では済まない理由は、**スコープ**（scope、
名前が有効な範囲）の存在です。次の例を見てください。

```ruby
x = 1
def foo
  x = 2   # これは foo の中だけの x
  x + 10
end
```

メソッド `foo` の中の `x` と、外側の `x` は**別物**です。処理系は
「いま自分はどのスコープにいるか」を意識して名前を引かなければなりません。
これを実現する典型的な方法が、シンボルテーブルを**入れ子**（nested）に
することです。スコープに入るたびに新しい表を作り、外側の表へのリンクを
持たせます。

```ruby
class Scope
  def initialize(parent = nil)
    @table = {}
    @parent = parent   # 外側のスコープ（なければ nil）
  end

  def define(name, value)
    @table[name] = value
  end

  # まず自分の表を見て、なければ外側へたどっていく
  def lookup(name)
    if @table.key?(name)
      @table[name]
    elsif @parent
      @parent.lookup(name)
    else
      nil   # どこにも見つからない＝未定義
    end
  end
end

global = Scope.new
global.define("x", :global_x)

local = Scope.new(global)
local.define("x", :local_x)

p local.lookup("x")   # => :local_x （内側が優先される）
p local.lookup("y")   # => nil
```

この「自分になければ外側を見る」という探索を**スコープチェーン**
（scope chain）と呼びます。内側のスコープが外側の名前を**覆い隠す**ことを
**シャドーイング**（shadowing）と呼び、上の例で `local.lookup("x")` が
`:local_x` を返すのはまさにそれです。コンパイラの教科書では、この
入れ子の表をブロックの入退場に合わせて管理する手法が詳しく論じられています
[](#cite:aho2006)。

## スタックとして使うシンボルテーブル

入れ子の表は、**スタック**（stack、後入れ先出しのデータ構造）として
扱うと実装が素直になります。スコープに入るときに新しい表をスタックに
**プッシュ**（積む）し、抜けるときに**ポップ**（降ろす）します。
名前を引くときはスタックの上から順に探します。

```ruby
class ScopeStack
  def initialize
    @stack = [{}]   # 最初はグローバルスコープだけ
  end

  def enter = @stack.push({})   # スコープに入る
  def leave = @stack.pop        # スコープを抜ける

  def define(name, value)
    @stack.last[name] = value
  end

  def lookup(name)
    @stack.reverse_each do |table|   # 内側＝末尾から探す
      return table[name] if table.key?(name)
    end
    nil
  end
end
```

この方式の利点は、スコープを抜けた瞬間に、その内側で定義された名前が
まとめて消える点です。`pop` 一回で内側スコープの表ごと捨てられるため、
不要になったエントリの後始末を個別に考えなくて済みます。
ブロック構造を持つ言語のコンパイラは、おおむねこの考え方で局所変数を
管理します [](#cite:aho2006)。

## 表の実装は一つではない

シンボルテーブルの「表」は必ずしもハッシュ表とは限りません。
要件によっては別のデータ構造が向いていることもあります。

- **整列配列＋二分探索**：エントリが少なく、名前順に並べて表示したい
  場合などに有効です。探索は O(log n) ですが、実装が単純でメモリも
  小さく済みます。
- **平衡二分探索木**：常に名前順を保ちたい、範囲検索したい場合に向きます。
- **トライ**（trie、接頭辞木）：名前を一文字ずつ木の枝としてたどる構造で、
  共通の接頭辞を持つ識別子が多いときに効率的です。
- **スキップリスト**（skip list）：確率的にバランスを取る整列構造で、
  平衡木より実装が簡単なまま O(log n) の探索を実現します
  [](#cite:pugh1990)。

> [!TIP]
> 「シンボルテーブル＝ハッシュ表」と短絡せず、**何件くらいの名前を、
> どんな順序で、どれだけの頻度で引くのか**を考えてデータ構造を選ぶのが、
> 処理系設計者の腕の見せどころです。多くの名前を一度だけ引くならハッシュ表、
> 順序が大事なら木構造、というふうに要件から逆算します。

次の章では、この表に格納する「名前」そのものを、処理系がどう扱うと
速くなるのかを見ていきます。文字列としての名前を、軽い「印」へと
変える技法 —— **インターン**の話です。
