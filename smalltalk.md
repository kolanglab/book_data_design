# Smalltalk のイメージと become: と実体化されたフレーム

## 動的オブジェクト指向の源流

Smalltalk-80 [](#cite:goldberg1983) は、本書がこれまで何度も
引用してきた発明（インラインキャッシュ [](#cite:deutsch1984)、
そのマップ／シェイプへの発展（SELF 経由 [](#cite:chambers1989)）、
バイトコード VM、世代別 GC の実用化）の故郷です。Ruby の
`method_missing` の原型である `doesNotUnderstand:` も Smalltalk の
ものです。この章では、主流に転生した発明ではなく、**Smalltalk に
しかない過激な機能**を支えたデータ構造を三つ見ます。どれも
「すべてはオブジェクトである」という原則を一切妥協しなかった結果です。

## ヒープ丸ごとを永続化するイメージ

Smalltalk にはソースファイルとコンパイルという日常がありません。
開発者は**イメージ**（image、クラスも、メソッドも、ワークスペースの
ウィンドウの位置も、デバッグ途中のオブジェクトも含む、すべての
オブジェクトを含むヒープのスナップショット）の中で暮らし、
終了時にイメージをファイルへ保存し、次回はそこから再開します。

データ構造の視点では、イメージとは**ヒープのシリアライズ**
（直列化、メモリ上の構造をバイト列に写し取ること）です。ポインタ
だらけのオブジェクトグラフをファイルに書き、別のアドレスに読み戻して
も全参照が正しくつながる必要があります。Smalltalk 系の実装は、
読み戻し時のポインタ再配置（relocation）を高速に行う形式を磨き上げ
ました。この技術は、Emacs の dump、Android のヒーププロファイル、
V8 の**スナップショット**（起動高速化のため、組み込みオブジェクトを
構築済みヒープイメージとして埋め込む）、各種言語の「事前コンパイル
済みヒープ」へと受け継がれています。**プロセスの状態を持ち運べる
データにする**という、永続化の一つの極北です。

## 世界中の参照を一瞬で差し替える become:

Smalltalk には `a become: b` という驚愕のメソッドがあります。
実行すると、システム中に存在する a への参照すべてが、その瞬間から
b を指すようになります（相互版では入れ替わります）。「世界中の
ポインタの一斉書き換え」が、なぜ実装できたのでしょうか。

初期の Smalltalk 処理系の鍵は**オブジェクトテーブル**（object table）
です。オブジェクトへの参照を直接のポインタではなく、全オブジェクトの
所在を並べた表の添字で持つのです。すべての参照が表を経由するなら、
`become:` は**表のエントリを書き換えるだけ**、O(1) です。

```mermaid
graph LR
    R1["参照（添字 #42）"] --> OT["オブジェクトテーブル"]
    R2["参照（添字 #42）"] --> OT
    OT -->|"エントリ 42"| A["旧オブジェクト a"]
    OT -.become: 後.-> B["新オブジェクト b"]
```

仕組みは数行で確かめられます。

```ruby
# オブジェクトテーブルがあれば become: は表の 2 マス交換
class ObjectTable
  def initialize = @slots = []
  def register(obj) = (@slots << obj; @slots.size - 1)  # 「参照」＝表の添字
  def deref(handle) = @slots[handle]
  def become!(h1, h2)
    @slots[h1], @slots[h2] = @slots[h2], @slots[h1]     # これだけ。O(1)
  end
end

table = ObjectTable.new
a = table.register("old object")
b = table.register("new object")
ref1, ref2 = a, a            # プログラムのあちこちに散らばった a への参照
table.become!(a, b)
p table.deref(ref1)          # => "new object"
p table.deref(ref2)          # => "new object"  全参照が一斉に切り替わった
```

「値の表現」の章で触れた**ハンドル**（一段の間接参照）の威力が
最も劇的に現れた例です。間接参照は読みのたびに 1 ホップの税金を
取りますが、「全参照の一斉更新」というありえない操作を自明にします。
直接ポインタを使う現代の実装（多くの Smalltalk 子孫も含む）では、
`become:` のためにヒープ全体を走査するか、**転送ポインタ**
（forwarding pointer、「本体はあちら」と書いた札）を旧オブジェクトに
置いて GC に清算させるか、ずっと高い代償を払います。

`become:` は何に使うのでしょう。代表例は**遅延ロードのプロキシ**
（データベースから読む前の身代わりオブジェクトを、読み込み完了時に
本物と入れ替える）と、**クラスの再定義**です。インスタンス変数を
増やしたクラスを再コンパイルすると、既存インスタンスは旧レイアウト
のままです。Smalltalk は新レイアウトのオブジェクトを作って一斉に
`become:` し、生きているオブジェクトのスキーマ移行をやってのけ
ます。[オブジェクトの章](objects.md)のシェイプ遷移が「これから作るオブジェクト」の
話だったのに対し、こちらは「もう存在するオブジェクト」の形を変えます。
イメージの中で何年も生き続けるオブジェクトたちならではの要求です。

## スタックフレームをオブジェクトにする thisContext

Smalltalk の擬似変数 `thisContext` は、いま実行中のメソッドの
フレーム（[スタックとフレームの章](frames.md)の活性レコード）をオブジェクト
として返します。フレームは `MethodContext` クラスのれっきとした
インスタンスで、送り手（呼び出し元フレーム）、命令位置、一時変数を
インスタンス変数として持ち、読むだけでなく書けます。

これで何ができるか。**デバッガがただのライブラリになります**。
例外で止まったスタックを上下に歩き、変数を書き換え、命令位置を
戻して「そこからやり直す」。Smalltalk 名物の「デバッガの中で
プログラムを書く」開発スタイルは、フレームがオブジェクトである
ことの直接の帰結です。継続（[クロージャの章](closures.md)）も例外機構も、VM に
頼らずライブラリとして書けます。

代償も明確です。フレームを本当に毎回ヒープのオブジェクトとして
確保したら遅すぎます。実装は「ふだんはスタックに置き、`thisContext`
で触られた瞬間にオブジェクト化する」という、Python 3.11 のフレーム
遅延生成（[スタックとフレームの章](frames.md)）と同じ二段構えを、40 年早く
編み出していました。

## 系譜と教訓

- **インラインキャッシュ、JIT、世代別 GC、シェイプ**：本書の
  第I部と第II部で「現代の主流」として紹介した技術の多くが、
  Smalltalk とその子 SELF の遺産です（SELF の実装者たちは、のちに
  Java の HotSpot VM や JavaScript の V8 を作りました）
- **イメージ** → V8 スナップショット、事前構築ヒープ
- **オブジェクトテーブルと転送ポインタ** → ムービング GC の
  基盤技術（[メモリ管理の章](memory.md)）
- **実体化されたフレーム** → デバッガ、ジェネレータ、継続の実装
  パターン

「すべてはオブジェクト」を限界まで貫くと、ヒープも、参照も、
実行状態すらもデータ構造として手の中に落ちてきます。Smalltalk は
その実証実験でした。次の章では、対極の禁欲として、データ構造を
二つしか持たない言語、Forth を訪ねます。
