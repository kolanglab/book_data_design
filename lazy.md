# 遅延評価とサンク：まだ計算していない値

## 計算を先送りするという発想

ほとんどの言語は、式をその場で計算します（**正格評価**、strict
evaluation／eager evaluation）。`f(1 + 2)` なら、まず `3` を計算してから
`f` を呼ぶ。当たり前に見えますが、別の流儀もあります。**遅延評価**
（lazy evaluation）—— 値が**実際に必要になるまで**計算を先送りし、
必要になった瞬間に初めて計算する方式です。Haskell はこの方式を言語全体の
既定とした代表的な言語です。

遅延評価をデータ構造として支えるのが**サンク**（thunk）です。サンクとは
「**計算のレシピと、その材料（環境）を包んだ箱**」——
前章のクロージャの特殊形で、引数のないクロージャだと思えばほぼ正解です。

```ruby
# Ruby でサンクを手作りする
class Thunk
  def initialize(&recipe)
    @recipe = recipe      # 計算のレシピ（クロージャ）
    @done   = false
    @value  = nil
  end

  def force                # 「必要になった」ときに呼ぶ
    unless @done
      @value = @recipe.call   # 初回だけ実際に計算し…
      @done  = true           # 結果を覚えて、レシピは捨てる
      @recipe = nil
    end
    @value
  end
end

t = Thunk.new { puts "計算した！"; 6 * 7 }
puts "まだ計算していない"
p t.force   # ここで初めて「計算した！」→ 42
p t.force   # 二度目は表示なしで 42（記憶済み）
```

`force`（強制、サンクの中身を実際に計算させる操作）の二度目が一瞬で
返ることに注意してください。**一度計算したら結果で置き換え、二度と
計算しない** —— この記憶付きの遅延を**必要呼び**（call-by-need）と
呼び、記憶しない単純な先送り（**名前呼び**、call-by-name、必要になる
たびに毎回計算し直す）と区別します。Haskell は必要呼びです。同じ式を
二度評価しない、という保証が遅延評価を実用にしています。

## Haskell：すべての値がサンクかもしれない世界

Haskell では、あらゆる式が既定で遅延されます。「値の表現」の章で
触れたとおり、GHC の `Int` ですらポインタの先にあるのは「もう計算
された 42」かもしれないし「まだのサンク」かもしれません。この一様性を
支えるのが GHC の実行モデル **STG 機械**（Spineless Tagless G-machine）
です [](#cite:peytonjones1992)。

STG の世界では、ヒープ上のあらゆるものが**クロージャ**という同じ形を
しています。先頭に**情報表**（info table）へのポインタがあり、情報表が
「これは計算済みのデータか、未評価のサンクか、関数か」と、評価の
ためのコードを教えてくれます。値が必要になったら、とにかくその
クロージャに「自分を評価せよ」と命じる（コードへ跳ぶ）だけ。計算済みの
データなら即座に自分を返し、サンクなら本体を計算します。
タグの分岐を**コードへの間接ジャンプ**に置き換えた、徹底して一様な
設計です。

サンクの評価が終わると、サンクは結果を指す**間接ノード**
（indirection）に書き換えられ、以後の参照は結果へ素通りします
（GC が間接ノードを刈り取って短絡します —— OCaml の `Lazy.t` も
同様に、強制の瞬間にヒープブロックのタグを書き換え、GC が短絡する
設計です）。また、評価を始めたサンクは**ブラックホール**（blackhole）と
いう印に書き換えられます。評価中に自分自身の値が必要になったら ——
つまり `x = x + 1` のような自己依存なら —— ブラックホールに突き当たる
ので、無限ループではなく「<<loop>>」というエラーとして検出できます。
並行 Haskell では「他のスレッドが評価中」の印としても使われ、同じ
サンクの二重評価を防ぎます。

> [!NOTE]
> 「まだ計算していない値」をデータとして持つ言語では、**評価が
> どこまで進んでいるか**にも名前が要ります。Haskell の値は、一番外側の
> 構成子（リストなら「cons か nil か」）だけが確定した状態 ——
> **弱頭部正規形**（WHNF、weak head normal form）—— まで評価される
> のが基本単位です。リストの先頭だけ確定していて、尻尾はまだサンク、
> という「半分だけ計算されたデータ構造」が当たり前に存在するのです。

## 無限のデータ構造：遅延が可能にするもの

サンクの何がうれしいのか。最大の配当は、**無限のデータ構造**が
書けることです。リストの尻尾がサンクでよいなら、「すべての自然数の
リスト」が作れます。必要な分だけ force すればよいのですから。

```haskell
-- Haskell：無限リストとフィボナッチ数列
nats = [0 ..]                          -- 全自然数
fibs = 0 : 1 : zipWith (+) fibs (tail fibs)  -- 自分自身を使って定義！
take 8 fibs   -- => [0,1,1,2,3,5,8,13]  必要な 8 個だけ計算される
```

`fibs` の定義は自分自身を参照していますが、サンクのおかげで循環定義が
そのまま動きます（**結び目を作る**、tying the knot と呼ばれる技法です）。
Hughes は古典論文「なぜ関数型プログラミングは重要か」で、遅延評価が
**生成器と消費者の分離**を可能にする —— 「全部生成してから絞り込む」と
素直に書いても、実際には必要な分しか生成されない —— ことを、関数型
プログラミングの本質的な強みとして挙げました [](#cite:hughes1989)。
探索木を「全部展開してから刈る」プログラムが、実行時には刈った分しか
展開しない、というのが彼の有名な例です。

この思想は正格な言語にも輸入されています。SICP が Scheme の上に
`delay`/`force` で構築する**ストリーム**（先頭は計算済み・尻尾は
サンクのリスト）はその古典的な教材です [](#cite:abelson1996)。
冒頭の `Thunk` があれば、Ruby でも 15 行で作れます。

```ruby
# 先頭は計算済み・尻尾はサンク、の無限ストリーム
class Stream
  def initialize(head, &tail) = (@head = head; @tail = Thunk.new(&tail))
  attr_reader :head
  def tail = @tail.force                      # 触られて初めて伸びる

  def self.from(n) = new(n) { from(n + 1) }   # 全自然数

  def map(&f) = Stream.new(f.(head)) { tail.map(&f) }
  def take(k) = k.zero? ? [] : [head] + tail.take(k - 1)
end

p Stream.from(0).map { |x| x * x }.take(5)  # => [0, 1, 4, 9, 16]

def fibs(a = 0, b = 1) = Stream.new(a) { fibs(b, a + b) }
p fibs.take(8)                              # => [0, 1, 1, 2, 3, 5, 8, 13]
```

`map` が尻尾の計算を**新しいサンクに包み直す**だけで返っている
ことに注目してください。無限のストリームを map しても一切計算は
走らず、`take` が触った分だけが実体化します。Thunk が記憶を持つ
ので、同じストリームを二度なめても二度目は計算なし —— Haskell の
リストの挙動が、この 15 行に再現されています。実用言語では、
Clojure の遅延シーケンス（`lazy-seq`）がこの直系です
（Clojure は性能のため、要素を 32 個ずつまとめて評価する
**チャンク化**も行います）。Python のジェネレータや Ruby の
`Enumerator::Lazy` も「必要な分だけ生成する列」を提供します。

```ruby
# Ruby の遅延列挙：無限列から必要な分だけ取る
p (1..Float::INFINITY).lazy.map { |x| x * x }.select(&:even?).first(3)
# => [4, 16, 36]   無限に map してから select している…ように見えて、
#                  実際は first(3) が満たされるまでしか計算しない
```

ただし実装はサンクとは別物です。Ruby の `lazy` は「ブロックの
パイプラインを記録しておき、要素を 1 個ずつ通す」**反復子の合成**で
あり、結果の**記憶はしません**。同じ遅延列を二度なめれば二度計算します。
「必要になるまで計算しない」は同じでも、**記憶付きのサンク**
（Haskell・`Lazy.t`・Clojure の lazy-seq）と**記憶なしの反復子**
（Ruby lazy・Python ジェネレータ）は、二度目の挙動とメモリ消費が
まったく違う —— 遅延系 API を使うときに必ず意識すべき区別です。

## 遅延の代償：スペースリーク

遅延評価はタダではありません。まず、**あらゆる値にサンクの生成・
判定・書き換えのコストがかかり**ます。GHC が**正格性解析**
（strictness analysis、「この引数はどうせ必ず使われる」と証明して
サンクを作らずに済ませる最適化）に多大な努力を注ぐのはこのためです。

そして悪名高いのが**スペースリーク**（space leak）です。たとえば
リストの合計を素朴な畳み込みで書くと：

```haskell
foldl (+) 0 [1..1000000]
-- 内部では ((((0+1)+2)+3)+...) という「足し算のサンクの塔」が
-- 100 万段、ヒープに積み上がってから、最後に一気に崩される
```

合計は最後まで「必要」にならないので、足し算は一度も実行されず、
**未評価のサンクが要素数ぶん積み上がる**のです。メモリを食い尽くした
あげく、評価の段階でスタックも溢れかねません。対策は、各ステップで
強制的に評価する正格版 `foldl'` や、`seq`・バン記法（`!x`）による
明示的な正格化です。「いつ計算するか」を制御できることは力ですが、
**「いつ計算されるか」を見失うとメモリの形が読めなくなる** —— 遅延
評価の光と影は表裏一体です。

> [!TIP]
> スペースリークは Haskell 固有の話ではありません。Ruby や Python でも、
> 「あとで評価するブロック／コールバック」を溜め込む設計は、前章で見た
> クロージャ経由の捕獲と相まって、同じ形のメモリ膨張を起こします。
> 「遅延されたままの計算が、いまヒープに何個あるか」という見方は、
> どの言語でも有効です。

## 一度だけ初期化：遅延のミニチュア

遅延評価の最小単位は、実は日常のコードに遍在しています。
**遅延初期化**（lazy initialization）です。

```ruby
def config
  @config ||= load_config_file   # 初回だけ読み込み、以後は記憶を返す
end
```

Ruby の `||=` イディオムは、まさに記憶付きサンクの手動展開です。
Scala の `lazy val` はこれを言語機能にしたもので、初期化済みかどうかの
フラグと同期処理をコンパイラが生成します。Java の `static` 初期化や
各種シングルトンの「ダブルチェックロッキング」も同族で、マルチ
スレッド下では「二つのスレッドが同時に force したらどうするか」という、
Haskell のブラックホールと同じ問題に行き着きます。遅延とは「計算の
状態（未／実行中／済）を持つデータ構造」であり、状態があるところには
必ず並行性の問題が付いてくる —— 次々章「並行処理とデータ構造」への
伏線です。

## まとめ：時間をデータ構造に変える

サンクは「**まだ起きていない計算**」をヒープ上の値として実体化する
データ構造でした。整理すると：

- **サンク**＝レシピ（コード＋環境）＋記憶領域＋状態（未評価／評価中／
  評価済み）
- 必要呼び（記憶あり）と名前呼び（記憶なし）の違いは、二度目の
  コストとして現れる
- 無限リスト・生成と消費の分離・結び目 —— 遅延は表現力を与える
  [](#cite:hughes1989)
- 代償はサンク管理のコストと、スペースリークという「見えない
  メモリの形」

クロージャが「空間（環境）を持ち歩く値」だとすれば、サンクは
「時間（いつ計算するか）を持ち歩く値」です。Okasaki が示したとおり、
この二つを組み合わせると、純粋関数型の世界でならし計算量を保証する
データ構造（「リスト・タプル・集合」の章のキューなど）まで作れます
[](#cite:okasaki1998)。計算そのものをデータとして扱う —— 言語処理系の
データ構造の中でも、もっとも思想的な一章でした。

次の章からは、打って変わって現実世界との接点 —— **時間**という、
扱いを間違えやすい型の実装を見ていきます。
