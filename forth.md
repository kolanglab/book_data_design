# 二本のスタックと一冊の辞書で動く Forth

## 極小の言語

Forth は 1970 年前後に Chuck Moore が作った言語で、今日も
ブートローダ（Open Firmware）や組み込み機器、宇宙機で生き続けて
います。その理由は徹底した小ささです。Forth の処理系は、本書が
これまで章を割いてきたデータ構造のほとんどを持ちません。
構文木なし、シンボルテーブルのスコープなし、GC なし、オブジェクトなし。
あるのは実質、**二本のスタックと一冊の辞書**だけです。それでこれだけの
ことができる、という下限の見本として、本章はあります。

## 構文木を捨てるデータスタック

Forth のプログラムは**逆ポーランド記法**（RPN、演算子を後に書く）で、
すべての語（**ワード**と呼びます）は**データスタック**を介して
値をやり取りします。

```forth
3 4 + 2 * .     \ (3+4)*2 を計算して表示 → 14
```

`3` はスタックに 3 を積むワード、`+` は上の 2 つを足して結果を積む
ワード。これは[構文木の章](syntax-tree.md)で見たスタック機械のバイトコードと
完全に同じものです。つまり Forth とは、**人間がバイトコードを直接
書く言語**です。構文解析も優先順位も構文木も不要です。ソースの語順が
そのまま実行順だからです。中間表現を人間の表記に昇格させるこの
倒立した設計で、処理系から構文解析フェーズが丸ごと消えます。

二本目の**リターンスタック**は戻り先（[スタックとフレームの章](frames.md)の
コールスタック）専用です。多くの言語が一本のスタックに「データと
戻り先」を混載するのに対し、Forth は分離しました。おかげで
プログラマはリターンスタックを一時置き場として直接操作でき
ます（`>r` `r>`）。これは危険と引き換えの自由です。フレームという抽象を
作らない代わりに、その構成部品を生のまま渡す設計だからです。

## 時間順に積もるシンボルテーブルとしての辞書

Forth の「メモリ」は**辞書**（dictionary）と呼ばれる一本の連続領域
です。新しいワードを定義すると、辞書の末尾に「名前・前のワードへの
リンク・本体」が追記されます。

```forth
: square ( n -- n^2 ) dup * ;   \ 新しいワード square を辞書に追記
5 square .                       \ → 25
```

辞書は実装上、新しい定義から古い定義へ向かう連結リストです。
ワードの検索は新しい方から線形に（実用処理系はハッシュを併用）。
ここに Forth 特有の意味論が生まれます。同名のワードを再定義しても、
古い定義は**消えずに埋もれる**だけ。再定義より前にコンパイル
された呼び出しは、古い定義を指したまま動き続けます。

[シンボルテーブルの章](symbol-table.md)で見たスコープチェーンが「空間の入れ子」
だったのに対し、Forth の辞書は時間の積み重ねとしてのスコープ
です。そして「どこまで定義したか」を覚えておけば、`MARKER` 一発で
それ以降の定義を全部忘れる（辞書の末尾を切り戻す）ことが
できます。[構文木の章](syntax-tree.md)で見たアリーナの一括解放そのものです。
GC を持たない Forth のメモリ管理は、この「積んで、まとめて切り戻す」
だけで成立しています。

スタックと辞書しかない言語は、インタプリタも一画面で書けます。

```ruby
# ひと画面 Forth：データスタック＋辞書
class MiniForth
  def initialize
    @stack = []
    @dict = {
      "+"   => -> { @stack << @stack.pop + @stack.pop },
      "*"   => -> { @stack << @stack.pop * @stack.pop },
      "dup" => -> { @stack << @stack.last },
      "."   => -> { print @stack.pop, " " },
    }
  end
  attr_reader :stack

  def make_word(src) = -> { run(src) }   # 本体を「値で」捕まえる（後述）

  def run(src)
    words = src.split                 # 字句解析は split だけ！
    until words.empty?
      w = words.shift
      if w == ":"                     # コロン定義：; までを本体として登録
        name = words.shift
        body = words.take_while { |t| t != ";" }
        words.shift(body.size + 1)
        @dict[name] = make_word(body.join(" "))    # 定義＝辞書への追記
      elsif (impl = @dict[w])
        impl.call
      else
        @stack << Integer(w)          # 辞書になければ数とみなす
      end
    end
  end
end

f = MiniForth.new
f.run(": square dup * ;   5 square 3 +")
p f.stack   # => [28]
```

構文解析も構文木もなく、`split` と辞書引きだけで言語が立ち上がる
ことを確かめてください。

> [!CAUTION]
> 地味な顔をした `make_word` が、実はバグ除けです。素直に
> `@dict[name] = -> { run(body.join(" ")) }` と書くと、ループ内の
> **同じ変数 `body`** をすべての定義のラムダが共有してしまい、
> 二つ目のワードを定義した瞬間に一つ目のワードの本体まで
> すり替わります（実際、本書の初稿はこのバグを踏みました）。
> メソッド呼び出しの引数は呼び出しごとに新しい束縛なので、
> `make_word(…)` に通すことで「値で」捕まえられます。
> 「クロージャと関数」の章で見たループ変数の捕獲問題の、
> 生きた実例です。

ただしこの玩具は本物と一つ意味論が違います。本体を文字列のまま保存して実行時に再解釈するので、参照する
語をあとから再定義すると挙動が変わる（遅束縛）のです。本物の
Forth は定義の時点で辞書を引き、本体を**アドレスの列にコンパイル**
するので、再定義前にコンパイルされた語は旧定義を使い続けます
（早束縛）。「埋もれるが消えない」という先述の辞書の意味論は、
このコンパイル時解決から来ています。玩具と本物の差分そのものが
教材になる好例です。

## コンパイラはたった一つのフラグ

Forth には「実行モード」と「コンパイルモード」があり、その切り替えは
**STATE という変数 1 個**です。`:` がコンパイルモードに入れ、以後の
ワードは実行されずに辞書へコンパイル（呼び出しの追記）され、
`;` が戻します。

ここに Forth 最大の発明があります。**immediate ワード**は、
コンパイルモード中でも即座に実行されるワードです。`if` や
`loop` といった制御構造は、文法ではなく immediate ワードとして
ライブラリで実装されています。`if` は「あとで埋める分岐の
飛び先」をスタックに積みながらコンパイルを進め、`then` がそれを
埋める。つまりコンパイラの内部状態（未解決の分岐先）を、ユーザーの
見えるスタックで管理しているのです。コンパイラを拡張する言語
機能（Lisp のマクロに相当）が、データ構造 2 つの世界では「コンパイル
中に走るただのワード」に還元されます。言語の自己拡張の最小実装です。

## 間接スレッデッドコードによるディスパッチ

コンパイルされたワードの本体は、伝統的に**スレッデッドコード**
（threaded code、呼び出すワードのアドレスをただ並べた配列）
です。実行器（インナーインタプリタ）は、アドレスを一つ読んでは
そこへ跳ぶ、を繰り返すだけ。

```
# ワード square の本体（間接スレッデッドコードのイメージ）
[ dup のアドレス, * のアドレス, exit のアドレス ]
```

これは[構文木の章](syntax-tree.md)のバイトコードディスパッチの一変種で、命令の
列挙値を switch で分岐する代わりに、**アドレスそのものを命令にする**
ことで分岐を間接ジャンプ 1 回に削った形です。同じ技法は GCC の
computed goto を使った高速インタプリタ（CPython や CRuby の
ディスパッチループ）に今も生きています。メモリが数 KB の機械で
インタプリタを成立させるための 1970 年代の倹約術が、最新の VM の
内側で現役なのです。

## 並べることが合成になる連結プログラミング

Forth のプログラムをもう一度見てください。**引数の名前がどこにも
ありません**。`: square dup * ;` に仮引数はなく、既存のワードを
並べただけです。すべてのワードは「スタックを受け取り、スタックを
返す関数」なので、二つのワードを連結して並べることが、そのまま
**関数合成**になります（`square` は `dup` のあとに `*` を適用する
合成関数です）。

```forth
: square  dup * ;          \ square = 「dup して掛ける」の合成
: quad    square square ;  \ 並べる＝さらに合成する
```

この性質は後年**連結プログラミング**（concatenative programming）と
名付けられ、Joy という言語で理論化されました。スタック効果さえ
合っていれば、プログラムのどの部分列も切り出して名前を付けられます。
リファクタリング（Forth 用語で **factoring**）が「列の切り貼り」に
なるのが実用上の旨味です。

データ構造の眼で見ると、これは本書の議論と裏表の関係にあります。
引数を**名前**で受け渡す言語は、名前を解決するための環境やフレーム
（[シンボルテーブルの章](symbol-table.md)、[クロージャの章](closures.md)）を必要としました。並べる
だけの言語では、値の受け渡し場所は**スタックただ一つ**。だから
Forth は環境というデータ構造を丸ごと省略できたのです。代償として、
スタック上の値の段取り（`dup` `swap` `over` といったシャッフル
語による整列）がプログラマの仕事になります。名前で渡すか、
位置で渡すか。「変数」というおなじみの機能の下にも、データ構造の
選択が隠れていたわけです。

## 系譜と教訓

- **スタック機械の表記**は PostScript（プリンタの中の Forth 的
  言語）、Java バイトコードや WebAssembly のスタック型命令セット、
  Bitcoin Script へ
- **スレッデッドコード**は高速インタプリタのディスパッチ技法へ
- **連結プログラミング**（前節）は Joy で理論化され、Factor などの
  後継言語へ

Forth の教訓は逆説的です。**データ構造を減らせば減らすほど、残った
データ構造にすべての意味が宿る**。スタックは式も制御もコンパイラの
作業領域も兼ね、辞書は名前空間もメモリ管理も歴史も兼ねる。豊富な
専用構造を持つ現代言語と対極の、しかし一貫した設計哲学です。

次の章では、別の方向の「一種類しかない」言語、すべてが文字列の
Tcl を訪ねます。
