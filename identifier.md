# 識別子の管理：インターンとシンボル

## 名前を何度も比較する問題

前章のシンボルテーブルでは、名前を**文字列**として比較していました。
`name == "x"` のような比較です。ところが処理系は、同じ名前を
**おそろしい回数**比較します。ソースコードに `x` が100回現れれば、
解析の過程で `x` という文字列の照合が何度も走ります。

文字列の比較は、最悪の場合、文字数に比例した時間 O(n) がかかります。
`"variable_with_a_long_name"` と `"variable_with_a_long_namf"` を比べると、
最後の一文字まで読まないと違いが分かりません。識別子（identifier、
変数名・関数名などの名前）の比較が処理系の至るところで起きることを
考えると、これは無視できないコストです。

そこで処理系は、**同じ名前は一度だけ覚え、二度目以降は軽い印で扱う**という
工夫をします。これを**インターン**（interning）と呼びます。

## インターン：同じ文字列を一つにまとめる

インターンとは、内容が等しい文字列を処理系の中で**ただ一つの実体**に
統一する操作です。同じ綴りの名前が何度ソースコードに現れても、
処理系の内部ではそれらを唯一の代表オブジェクトに対応づけます。
こうしておくと、名前の比較が**実体が同じかどうか**の比較、すなわち
**参照（ポインタ）の比較**だけで済みます。参照の比較は、中身の長さに
関係なく一定時間 O(1) で終わります。

インターンの実装は、これまで作ってきたハッシュ表そのものです。
「すでに同じ文字列を登録済みか？」をハッシュ表で調べ、あればそれを返し、
なければ登録して返します。

```ruby
class Interner
  def initialize
    @pool = {}   # 文字列 => 代表オブジェクト
  end

  def intern(str)
    @pool[str] ||= str.dup.freeze   # 初登場なら凍結して保存
  end
end

interner = Interner.new
a = interner.intern("counter")
b = interner.intern("counter")

p a.equal?(b)   # => true （まったく同じオブジェクト）
```

`equal?` は Ruby で**オブジェクトの同一性**（同じ実体か）を調べるメソッドで、
内容の等価を調べる `==` とは区別されます。インターン後は `a.equal?(b)` が
`true` になるので、名前の照合を `equal?`（実質ポインタ比較）で行えます。
この「文字列をいったん登録し、以後は同一性で比べる」考え方は、動的型付け
言語の値表現を論じた Gudeman の技術報告でも、識別子表現の基本として
触れられています [](#cite:gudeman1993)。

> [!NOTE]
> インターンには副次的な利点が二つあります。第一に**メモリの節約**。
> 同じ綴りの名前を何度書いても文字列の実体は一つしか存在しません。
> 第二に**ハッシュ値の再利用**。インターン時に一度ハッシュ値を計算して
> 代表オブジェクトに添えておけば、以後その名前をハッシュ表のキーに
> 使うとき、ハッシュ計算そのものを省けます。名前は何度もハッシュ表の
> キーになるので、この効果は絶大です。

## シンボル：言語機能になったインターン

インターンの考え方を、言語の利用者が直接使える機能として表に出したものが
**シンボル**（symbol）型です。Ruby の `:name` や、Lisp のシンボル、
Erlang/Elixir の**アトム**（atom、`:ok` のような定数名）がその代表です。
シンボルは「インターン済みの名前」であり、同じ綴りのシンボルは常に
同一のオブジェクトになります。

```ruby
p :counter.equal?(:counter)   # => true  （シンボルは常に同一）
p "counter".equal?("counter") # => false （文字列リテラルは別オブジェクト）

p :counter.object_id          # 何度評価しても同じ整数
```

Ruby の `Hash` のキーにシンボルがよく使われるのは、この性質のおかげです。
キーの比較が高速で、しかもハッシュ値の計算も一度きりで済みます。
シンボルは「人間に読める名前」と「機械にとって軽い印」を兼ね備えた、
処理系にとって理想的な識別子なのです。

```ruby
# シンボルキーのハッシュは比較が速い
config = { host: "localhost", port: 8080 }
p config[:port]   # => 8080
```

歴史をさかのぼると、シンボルは Lisp が 1950 年代から持っていた機能です
[](#cite:mccarthy1960)。Lisp 処理系はソースを読み込む段階で識別子を
すべてシンボルへインターンし、**obarray** などと呼ばれる表（まさに
インターンのためのハッシュ表）で管理してきました。Ruby のシンボルは
この direct な子孫です。

## インターンはいたるところに

「言語機能としてのシンボル」を持たない言語でも、処理系の内部では
インターンが多用されています。サーベイしてみましょう。

- **Java**：コンパイル済みのクラスファイルは**定数プール**（constant
  pool）という表を持ち、クラス名・メソッド名・文字列リテラルなどを
  番号で参照します [](#cite:lindholm2014)。バイトコード中に名前の
  文字列は現れず、「定数プールの 17 番」のような参照だけが並びます。
  さらに実行時には、文字列リテラルが VM 全体で共有の**文字列プール**へ
  インターンされます（`String.intern()` で明示的にも使えます）。
- **Python**：識別子に使われる文字列は自動的にインターンされ、
  `sys.intern()` で任意の文字列も登録できます。辞書（`dict`）の検索は
  「まずポインタ比較、だめなら内容比較」の二段構えなので、
  インターン済みの属性名は高速に引けます。
- **JavaScript エンジン**：V8 や SpiderMonkey は、プロパティ名を
  **アトム**（atom）と呼ばれるインターン済み文字列にします。
  「オブジェクト型」の章で見る隠しクラスやインラインキャッシュは、
  プロパティ名がポインタ比較できることを前提に組まれています。
- **Erlang/Elixir**：アトムは BEAM 仮想機械のグローバルな**アトム
  テーブル**に登録され、ワード 1 個の即値（表の添字）として扱われます
  [](#cite:stenman2024)。

> [!WARNING]
> インターンした名前を「いつ捨てるか」は設計上の急所です。
> 古い Ruby（2.1 以前）ではシンボルは一度作ると回収されない実装
> だったため、外部入力から無制限にシンボルを生成するとメモリを
> 食い潰されました（現在はシンボルも GC 対象です）。Erlang のアトム
> テーブルは今も既定で回収されず、上限（既定で約 100 万個）を超えると
> **仮想機械ごと停止**します。「ユーザー入力をアトムに変換しない」は
> Erlang プログラミングの鉄則です。インターンとは「永続的な表に
> 登録すること」であり、登録の権限を外部に渡すと資源枯渇攻撃に
> なる —— この構図は覚えておく価値があります。

## ID：整数に振り直す

実装をさらに突き詰めると、インターンした名前に**整数の通し番号**を
割り当てる方式に行き着きます。Ruby の処理系（CRuby）内部では、この通し番号を
**ID** と呼びます。名前を登録するたびに「0番、1番、2番…」と番号を振り、
以後は文字列でも参照でもなく、この整数だけを持ち回ります。

整数の比較は CPU が最も得意とする操作で、文字列比較とは比較にならないほど
高速です。配列の添字としても使えるため、「ID番号 → 名前」「ID番号 → 属性」と
いった対応を、ハッシュ表ではなく**ただの配列**で引けるようになります。

```ruby
# 名前 <-> ID の相互変換を行う最小の実装
class IdTable
  def initialize
    @id_of   = {}   # 名前   => ID
    @name_of = []   # ID番号 => 名前（添字がそのまま ID）
  end

  def to_id(name)
    @id_of[name] ||= begin
      @name_of << name
      @name_of.size - 1   # 末尾の添字を ID として返す
    end
  end

  def to_name(id) = @name_of[id]
end

ids = IdTable.new
p ids.to_id("if")      # => 0
p ids.to_id("while")   # => 1
p ids.to_id("if")      # => 0  （二度目も同じ ID）
p ids.to_name(1)       # => "while"
```

`@id_of` は「名前 → ID」を引くためのハッシュ表ですが、これは
**登録時に一度だけ**使います。いったん ID が分かれば、あとは整数として
持ち回り、名前に戻したいときだけ `@name_of` 配列を添字で引きます。
配列の添字アクセスは O(1) で、ハッシュ計算すら不要です。

この「名前 → ID → 配列で属性を引く」という三段構えは、シンボルテーブルの
高速化として広く使われます。たとえばキーワード（`if`・`def`・`while` など、
言語に予約された語）の判定を、文字列比較ではなく ID 比較や整数の
範囲チェックで済ませる、といった最適化が可能になります [](#cite:aho2006)。

## 印そのものに情報を埋め込む

最後に、もう一歩進んだ技法に触れておきます。名前に振った ID（整数）を
持ち回るとき、その整数の**ビットの一部に種類の情報を埋め込む**ことが
あります。たとえば下位数ビットを「これは局所変数」「これはインスタンス変数」
「これは定数」といった区別に使うのです。

Ruby の処理系では実際に、ID の下位ビットを見るだけでその識別子が
ローカル変数なのか、インスタンス変数（`@x`）なのか、グローバル変数（`$x`）
なのか、定数（`Foo`）なのかを判別できるようになっています。文字列の
先頭文字を調べる代わりに、整数のビット演算一発で種類が分かるわけです。

```ruby
# 概念図：ID の下位 3 ビットに「種類」を埋め込む
LOCAL    = 0b000
INSTANCE = 0b001
GLOBAL   = 0b010

def make_id(serial, kind) = (serial << 3) | kind
def kind_of(id)           = id & 0b111
def serial_of(id)         = id >> 3

id = make_id(42, INSTANCE)
p kind_of(id) == INSTANCE   # => true
p serial_of(id)             # => 42
```

さらに CRuby では、よく使われるシンボルはこの ID を「値の表現」の章で
見る即値としてポインタの中に直接埋め込みます（静的シンボル）。つまり
`:foo` という値の実体は、ヒープ上のオブジェクトですらなく、ID を
タグ付きでエンコードした 1 ワードの整数なのです。一方、実行中に
動的に作られたシンボルはヒープ上のオブジェクト（動的シンボル）として
表現され、GC で回収できます。「永続的な名前は即値、一時的な名前は
オブジェクト」という、寿命に応じた二段構えになっています。

このように「値そのものにタグ（種類の印）を埋め込む」発想は、
本書の後半で繰り返し登場します。整数や `nil` や `true` をオブジェクトとして
表現するときの**タグ付き表現**（「値の表現」の章）も、まったく同じ
アイデアの応用です [](#cite:gudeman1993)。識別子の管理は、その最初の
出会いの場所といえます。

## 名前を作る、名前を汚さない：gensym とマクロ衛生

最後に、識別子管理の最前線に触れます。**プログラムがプログラムを
生成する**とき（マクロ）、名前は新しい難問になります。

Lisp 系のマクロは、コードの断片（構文木）を組み立てて返します。
このとき、マクロが内部で使う一時変数の名前が、**展開先にたまたま
ある同名の変数と衝突**したら？ 古典的な対症療法が **gensym**
（generate symbol）——「絶対に誰とも衝突しない新品のシンボル」を
発行する機能です。実装は本章の ID 発行そのもので、インターン表に
**載せない**（だから誰も同じ名前を書けない）シンボルを作ります。

Scheme はこれを言語の保証に高めました。**衛生的マクロ**（hygienic
macro）—— マクロが導入した名前と利用者の名前は、同じ綴りでも
**自動的に別物として**扱われる、という保証です。実装の中心は、
識別子を「ただの文字列」ではなく**「綴り＋出自（どの展開段階で
生まれたか）」の組**として表すことです。つまり識別子のデータ構造を
リッチにすることで衝突を原理的に消す。その現代的な到達点が
Racket の **scope set** 方式 [](#cite:flatt2016) で、識別子に
**スコープの集合**を持たせ、束縛の解決を「スコープ集合の包含関係で
最大のものを選ぶ」操作として定義し直しました。リスト・タプル・
集合の章で見た「集合」が、名前解決の意味論そのものになっている
わけです。

「名前とは綴りである」という素朴な前提は、インターン（綴り→一意な
印）で一度目の修正を受け、マクロ衛生（綴り＋文脈→束縛）で二度目の
修正を受けました。識別子は、見かけによらず進化し続けている
データ構造なのです。

次の章では、こうして整理した名前を使って組み立てる、プログラムの構造
そのものを表すデータ構造 —— **構文木**を見ていきます。
