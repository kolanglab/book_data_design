# R の内部データ構造

## 統計の現場が言語を変形させた

R は S 言語（1976 年、ベル研）の子孫で、統計解析の事実上の標準語
です。「データ分析の対話的な使い心地」と「巨大なベクトルの効率」と
いう、相反する要求に数十年さらされた結果、R の内部には他の言語で
見られない独特のデータ構造がいくつも育ちました。本章では、プロミス、
コピーオンモディファイ、ALTREP の三つを見ます。

## 引数はすべてプロミス

R の関数の引数は、呼び出し時に評価されません。各引数は**プロミス**
（promise）として渡され、関数の中で最初に使われた瞬間に評価されます。
プロミスは[遅延評価の章](lazy.md)で見たサンクの R 版で、「式（構文木）、
それを評価すべき環境、評価結果のキャッシュ」の三つ組です。

Haskell と違うのは、プロミスが**式の構文木を保持していて、しかも
プログラムから取り出せる**ことです。

```r
f <- function(x) substitute(x)   # x の「値」ではなく「式」を取り出す
f(height + weight)
# => height + weight   ← 評価される前の構文木そのもの！
```

この仕組み（**非標準評価**、non-standard evaluation）が、R の
使い心地の核心です。`plot(height ~ weight)` のグラフ軸に変数名が
自動で付くのも、dplyr の `filter(df, age > 20)` が `age` を
「いまは存在しない変数」としてエラーにせず**データフレームの列名**
として解釈できるのも、ライブラリが引数の構文木を覗いて独自に評価
し直しているからです。[構文木の章](syntax-tree.md)で「コンパイル時の構造」として
紹介した AST が、R では実行時の第一級の値として日常的に受け渡される。
ここはサンクとメタプログラミングの合流点です。

> [!NOTE]
> 代償は予測可能性です。引数がいつ評価されるか（そもそも評価される
> のか）が呼び出し側から見えないため、副作用のある式を引数に書くと
> 挙動が読めなくなります。また既定値引数が他の引数を参照できる
> （`function(n, m = n * 2)`）のもプロミスの遅延ゆえですが、循環
> させると実行時にエラーになります。[遅延評価の章](lazy.md)で見たブラックホール検出と
> 同じ問題が、引数リストの中で起きるのです。

## 値意味論を偽装するコピーオンモディファイ

R の代入は**値意味論**です。`b <- a` のあとで `a` を変更しても
`b` は変わりません。統計処理では「元のデータフレームを壊さない」
ことが絶対の安心感につながるため、これは譲れない仕様です。しかし
1 億行のベクトルを代入のたびにコピーしていたら使い物になりません。

実装は**コピーオンモディファイ**（copy-on-modify）です。`b <- a` は
ポインタを共有するだけ。各オブジェクトは「自分が何ヶ所から共有
されているか」の印（古くは NAMED フラグ、現在は参照カウント）を
持ち、**書き込みのときだけ**それを調べて、共有されていれば複製、
独占していればその場で書き換えます。

```r
a <- runif(1e8)   # 巨大ベクトル
b <- a            # コピーは起きない（共有の印が付くだけ）
b[1] <- 0         # ここで初めて b の実体が複製される
a[1] <- 0         # a はもう独占状態なので、その場で書き換え（複製なし）
```

[メモリ管理の章](memory.md)で「解放のため」に使った参照カウントを、ここでは
「コピーを省いてよいか」の判定に使っています。まったく同じ
設計が Swift の配列や辞書（`isKnownUniquelyReferenced` で内蔵
バッファの独占を確認し、共有時のみ複製）と PHP の配列にもあり、
「値意味論を見せて、共有を隠す」は一つの設計パターンとして
確立しています。Go のスライス（[配列の章](arrays.md)）が共有を見せる選択を
したのと対照的です。

## ベクトルのふりをする別表現 ALTREP

R 3.5 で入った **ALTREP**（alternative representations）は、
「ベクトルという顔の裏に、別のデータ構造を差し込む」枠組みです。
たとえば `1:1e9`（1 から 10 億の整数列）は、10 億個の整数を確保
しません。「開始 1、終わり 10⁹ の等差数列」という 2 ワードの
記述子だけを持ち、`x[i]` が来たら `i` を計算で返します。総和も
`n(n+1)/2` で済みます。実体化（materialize）は、本当に要素の
書き換えが起きるまで起きません。

```r
x <- 1:1e9        # メモリはほぼゼロ（compact representation）
sum(x)            # 公式で計算、依然としてゼロ
x[3] <- 0         # ここで初めて 10 億要素が実体化される（4GB!）
```

ファイルから読んだ文字列ベクトルの「数値への変換を遅延する」
表現や、「ソート済みであることを覚えている」表現（min/max が O(1)
になる）も ALTREP で実現されています。[配列の章](arrays.md)の「要素種別」、
[遅延評価の章](lazy.md)の「サンク」、[文字列の章](strings.md)の「ロープ」を一般化した、
「同じ型の中の多態な内部表現」のための公式な差し込み口です。

## 欠損値というビットパターン

統計言語として、R はあらゆる型に**欠損値 NA** を持ちます。問題は
表現です。`double` のベクトルに「欠損」をどう埋め込むか。R の
答えは、NaN の特定のペイロード（仮数部に 1954 という値を入れた
特別な NaN）を NA として予約することでした[^na1954]。「値の表現」の章で
見た NaN ボクシングと同じ「NaN の空き地」を、たった 1 つの番外値の
ために使っているのです。整数ベクトルでは最小値 `INT_MIN` を NA に
予約します。ニッチ（決して現れないビットパターン）に意味を割り当てる、
本書で繰り返し見た技法の、統計分野での現れです。

[^na1954]: 1954 は S 言語の作者にちなんだ遊び心です。

## 列の束としての表 data.frame

R の中心的データ構造 `data.frame` は、[APL の章](apl.md)で見た q のテーブルと
同じく「列ベクトルのリスト」です。1 列だけの集計は 1 本の
ベクトル走査で済み、列の追加はリストへの追加だけ。行の追加が遅い
（全列を作り直す）のも同じ理由です。pandas や Polars、Apache Arrow と
いった現代のデータフレーム実装はみなこの列指向であり、R はその
普及の立役者でした。

## 系譜と教訓

- **プロミス＋構文木**：サンク（[遅延評価の章](lazy.md)）にメタプログラミング
  を接木した独自進化。「式の保存と選択的評価」は Julia のマクロや
  各種クエリ DSL に通じる
- **コピーオンモディファイ**：値意味論の偽装。Swift や PHP と収斂進化
- **ALTREP**：内部表現の多態化を公式 API にした例。V8 の要素種別の
  「ユーザー拡張可能版」と読める
- **NA のビットパターン**：ニッチ最適化の応用

R は「言語設計の教科書」からは外れた道を歩きましたが、その内部は
本書の道具立て（サンク、参照カウント、遅延実体化、タグの隙間）の
応用問題集です。最後の章では、Web の現場で似た進化を遂げた
二つの言語、PHP と Perl を訪ねます。
