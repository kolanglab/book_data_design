# 並行処理とデータ構造：共有か、複製か、不変か

## 並行性はデータ構造の問題である

本書の最後の主題は**並行処理**（concurrency、複数の実行の流れが
同時に進むこと）です。なぜデータ構造の本の締めくくりが並行処理なのか。
それは、並行処理の困難の正体が「**同じデータ構造に複数の実行が同時に
触る**」ことだからです。

ハッシュの章で見た再ハッシュを思い出してください。全要素を新しい
バケット配列へ移している**途中**の状態は、表として壊れています。
一人で使うなら途中状態は誰にも見えませんが、別のスレッドが
その瞬間に検索したら？ ——これが**データ競合**（data race）です。
本章では、言語処理系がこの問題にどんなデータ構造で答えてきたかを、
大きく三つの戦略 ——**守る・複製する・不変にする**—— に分けて巡ります。

## 戦略1：ロックで守る —— そして大きすぎたロック

素朴な解は、データ構造を**ロック**（lock、同時に一人しか入れなくする
仕組み）で囲うことです。その極端な形が、処理系全体を一個のロックで
囲ってしまう **GIL**（Global Interpreter Lock、グローバルインタプリタ
ロック）です。CPython の GIL、CRuby の GVL がこれにあたります。

なぜそんな大胆なことをするのか。理由はまさに本書で見てきたデータ構造に
あります。CPython の参照カウンタ（メモリ管理の章）は、あらゆる代入で
更新されます。これを複数スレッドから安全にするには、すべてのカウンタ
操作を不可分（atomic）にする必要があり、シングルスレッド性能が大きく
落ちます。インタプリタの内部構造（フリーリスト、メソッドキャッシュ、
シンボル表…）も同様です。「全部を一個のロックで守れば、内部データ構造は
シングルスレッド前提のまま速くできる」—— GIL は怠慢ではなく、
トレードオフの選択でした。

その代償が「スレッドを増やしても CPU を 1 個分しか使えない」ことです。
CPython はこの壁を壊す挑戦を続けており、PEP 703（フリースレッド化）は
参照カウントを**所有スレッド用の高速カウンタと、他スレッド用の不可分
カウンタに分ける**（biased reference counting）など、データ構造の
作り直しによって GIL 撤廃を可能にしました [](#cite:pep703)。
ロックを外すとは、ロックが守っていた**すべてのデータ構造を作り直す**
ことなのです。

ロックの粒度を細かくする方向の到達点として、Java の
`ConcurrentHashMap` を見ておきましょう [](#cite:goetz2006)。初期の
実装は表を 16 個の区画（セグメント）に分けて区画ごとにロックする
**ロックストライピング**でしたが、現在の実装はさらに細かく、
**バケット単位**で同期し、空きバケットへの挿入はロックすら使わない
CAS（compare-and-swap、「値が期待どおりなら入れ替える」CPU 命令）で
行います。「どの単位で守るか」の設計が 20 年かけて配列全体 →
区画 → バケット 1 個へと細粒度化してきた歴史です。

## 戦略2：複製する —— Erlang の世界観

二つ目の戦略は、**そもそも共有しない**ことです。Erlang の仮想機械
BEAM は、軽量プロセスごとに**完全に独立したヒープ**を持たせます
[](#cite:stenman2024)。プロセス間のやり取りは**メッセージ送信**だけで、
送られたデータは相手のヒープへ**深いコピー**をします。

- 共有がないので、ロックも競合も原理的に存在しない
- GC（メモリ管理の章）も**プロセスごとに独立**して走らせられる。
  数十万プロセスがいても、止まるのは小さなヒープ一つずつ ——
  「世界を止めない GC」を、アルゴリズムではなく**ヒープの区画割り**で
  実現している
- 代償はコピーのコスト。そのため 64 バイトを超える大きなバイナリ
  だけは例外的に、プロセス間で共有される参照カウント付き領域に置く
  （refc binary）という抜け道が用意されている

「共有しない」を原則とする Erlang にも、実は公認の例外があります。
**ETS**（Erlang Term Storage）—— どのプロセスのヒープにも属さない
共有メモリ上のテーブル（ハッシュ表または平衡木）で、プロセスは
そこへ項を**コピーで**出し入れします。GC の対象外なので巨大な
テーブルが GC 時間に響かず、読み書きは VM が同期します。
「原則は隔離、しかし大きな共有状態には専用の置き場を」という、
理念と実用の折り合いの付け方として知られています。

メッセージを受け取る側の**メールボックス**（mailbox、届いた
メッセージの待ち行列）も興味深い構造です。Erlang の `receive` は
「条件に合う最初のメッセージ」を取り出せる（**選択的受信**）ため、
合わないメッセージが残り続けてメールボックスが肥大化する、という
この言語特有の性能問題があります。キューという単純な構造に
「選択」を許した瞬間、走査コストが生まれる —— ここにも構造と性能の
直結が見えます。

Erlang にはもう一つ、「動かしたまま入れ替える」文化の極めつけと
いえる機能があります。**ホットコードロード**（実行を止めずに
モジュールを更新する機能）です。BEAM は各モジュールのコードを**「現行版」と「旧版」の
二世代まで同時に保持**し [](#cite:stenman2024)、実行中のプロセスは
旧版を走り続け、モジュール名経由の呼び出しをした瞬間に新版へ移り
ます。電話交換機を止められないという要求が、**コード自体を二重保持の
データ構造**にしてしまった例です。

Ruby の **Ractor** も同じ思想の系譜です。Ractor 間で共有できるのは
**凍結された（深く不変な）オブジェクトだけ**で、可変なオブジェクトは
コピーするか、所有権ごと**移動**（move、送り主からは見えなくなる）して
渡します。「不変なら共有してよい、可変なら一人しか持てない」——
Rust の所有権・`Send`/`Sync` がコンパイル時に検査する規律を、
実行時の規則として導入したものと見ることができます。

## 戦略3：不変にする —— 永続データ構造の出番

三つ目の戦略が、本書で何度も顔を出してきた**不変性**です。
**決して書き換わらないデータ構造は、何の同期もなしに共有できます**。
読み手は途中状態を絶対に見ないからです。

Clojure はこの戦略を言語の中心に据えました。コレクションはすべて
永続データ構造（配列とハッシュの章で見た HAMT [](#cite:bagwell2001)）で、
「変更」は構造共有による新版の作成です。では状態の変化はどう表すのか。
Clojure の答えは **atom** です。

```clojure
;; Clojure: 不変なマップを atom（1 マスの可変な箱）に入れる
(def state (atom {:count 0}))
(swap! state update :count inc)   ; CAS で差し替える
```

`swap!` の内部は「現在の中身を読む → 関数を適用して新版を作る →
**CAS で差し替える**（誰かが先に差し替えていたらやり直し）」という
ループです。可変な場所を**ポインタ 1 個**に集約し、データ本体は
すべて不変 —— ロックなしで一貫したスナップショットが読める、
シンプルで強力な構成です。同じパターンは、読み取りが圧倒的に多い
場面の定石として他言語にもあります。Java の `CopyOnWriteArrayList`
（書き込みのたびに配列を丸ごと複製して差し替える）、Linux カーネルの
RCU、そして CRuby が内部で使う「メソッドテーブルの差し替え」も
この family です。

## ケーススタディ：Go のチャネル

「共有メモリで通信するな、通信でメモリを共有せよ」を掲げる Go の
**チャネル**（channel）は、戦略 1 と 2 の合いの子です。内部構造
（`hchan`）は教科書のように整っています。

- **リングバッファ**（環状の配列）：バッファ付きチャネルの本体。
  先頭・末尾の添字をぐるぐる回す、固定長キューの定番構造です。
- **送信待ちキューと受信待ちキュー**：バッファが満杯／空のときに
  寝て待つゴルーチンの連結リスト。
- 全体を守る小さな**ミューテックス**（ロック）。

妙味は**直接手渡し**（direct handoff）です。受信者が待っている状態で
送信すると、データはバッファを経由せず、**送信者から受信者のスタック
変数へ直接書き込まれ**ます。キューという抽象の下で、実装は「できる
ときは構造を素通りする」のです。

ゴルーチンを CPU に割り当てるスケジューラにも有名なデータ構造が
います。各 OS スレッドが自分専用の実行可能キューを持ち、手すきの
スレッドが他人のキューの**尻から半分奪っていく**——**ワーク
スティーリング**（work stealing）です。自分は先頭から、泥棒は末尾から
触るので競合が起きにくい、という両端キュー（Chase-Lev deque
[](#cite:chase2005)）の性質が並行スケジューリングの土台になっています。

## ケーススタディ：メモリを誰がいつ解放するか

並行とデータ構造の交差点で、最後に GC の話に戻ります。ロックフリーの
（ロックを使わない）データ構造を C++ や Rust で書くとき、最大の難問は
「ノードを取り外したあと、**まだ読んでいる人がいるかもしれないのに、
いつ解放するか**」です。GC のある言語ではこの問題が**消えます** ——
読んでいる人がいる限り GC が生かしてくれるからです。Java に世界有数の
並行コレクション群がそろっているのは偶然ではなく、**GC が並行データ
構造の隠れた基盤**だからです。GC なしの世界では、エポックベース回収や
ハザードポインタといった「ミニ GC」を自作する必要があり、Rust の
`crossbeam` などがそれを提供しています。メモリ管理の章で見た
「生きているとは何か」という問いが、並行の世界では一段と切実になる
わけです。

## まとめ：三つの戦略

並行処理への三つの戦略は、本書で学んだデータ構造の知識の上に
立っていました。

- **守る**：ロックの粒度設計（全体 → 区画 → バケット → CAS）。
  GIL は参照カウントという内部構造の保護でもあった
- **複製する**：プロセスごとのヒープ（Erlang）、コピーと移動
  （Ractor）。GC の独立という配当つき
- **不変にする**：永続データ構造＋CAS の 1 マス（Clojure）、
  コピーオンライトの差し替え

即値とヒープオブジェクト、連続配列とロープ、可変ハッシュと HAMT、
そしてロックと不変性 —— 主流言語の世界の旅はここまでです。
続く第III部では少し足を伸ばし、独自の世界観を持つ言語たち ——
すべてが配列の APL、束縛を取り消せる Prolog、フレームすら
オブジェクトの Smalltalk たち —— のデータ構造を訪ねます。
主流の設計がなぜ主流なのかは、外から見たときに一番よく分かるのです。
