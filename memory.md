# メモリ管理とGC：オブジェクトの生と死

## なぜ処理系がメモリを管理するのか

前章までで、値はヒープ上のオブジェクトになることがあると分かりました。
では、そのオブジェクトは**いつ、誰が捨てる**のでしょうか。C のように
プログラマが `free` で手動管理する言語もありますが、Ruby・Python・Java・
Go・Haskell など大半の言語は、**ガベージコレクション**（garbage
collection、**GC**、ごみ集め）—— もう使われないオブジェクトを処理系が
自動的に見つけて回収する仕組み —— を備えています。GC は 1960 年の
Lisp とともに生まれた、言語処理系最古の発明の一つです
[](#cite:mccarthy1960)。

GC は「アルゴリズムの話」と思われがちですが、実際には**データ構造の
かたまり**です。ヒープをどう区切るか、空き領域をどう覚えるか、
「生きている」印をどこに付けるか、世代をどう分けるか —— 本章では
GC をデータ構造の観点から解剖します。体系的な全体像は GC ハンドブック
[](#cite:jones2011) と Wilson のサーベイ [](#cite:wilson1992) が
決定版です。

## ヒープの区画整理：ページとサイズクラス

まず、オブジェクトを置く土地であるヒープの構造から始めます。
汎用の `malloc` をオブジェクトのたびに呼ぶのは遅すぎるため、実用の
処理系はみな**自前のアロケータ**（allocator、領域割り当て器）を持ちます。

典型的な構成は二段階です。OS からは大きな**ページ**（page、固定長の
区画）単位でメモリをもらい、ページの内部を自分で小分けにします。

- **CRuby** は 16KB のヒープページを確保し、各ページを固定サイズの
  **スロット**（slot、オブジェクト1個ぶんの枠）に等分します。長らく
  スロットは 40 バイト固定でしたが、3.1 以降の **Variable Width
  Allocation** で複数のスロットサイズ（サイズプール）を持つようになり、
  大きめのオブジェクトもスロットに収められるようになりました。
- **Go** は約 70 種類の**サイズクラス**（8, 16, 24, 32, ... バイト）を
  用意し、確保要求を最も近いクラスに切り上げて、クラスごとのページ
  （span）から切り出します。これは高速 malloc 実装（tcmalloc）譲りの
  設計です。

固定サイズの枠に揃える理由は明快で、**空き領域の管理が劇的に単純になる**
からです。どの枠も同じ大きさなら、「空いている枠」を一列につないでおく
だけでよく、断片化（使える合計は足りるのに連続した空きがない状態）も
起きません。

## フリーリスト対バンプアロケーション

空き領域を覚えるデータ構造の代表が**フリーリスト**（free list）です。
空きスロットの先頭に「次の空きスロットへのポインタ」を書き込んで
連結リストにします。空きスロット自身を台帳として使うので、追加の
メモリが一切要らないのが妙味です。確保はリストの先頭を外すだけ、
解放は先頭につなぐだけ、どちらも O(1) です。CRuby の各ヒープページは
このフリーリストで空きスロットを管理しています。

```ruby
# 概念図：空きスロット自身をつないで台帳にするフリーリスト
class FreeList
  def initialize(slots)
    @head = nil
    slots.each { |s| free(s) }
  end

  def allocate
    slot = @head
    @head = slot.next_free   # 先頭を外す O(1)
    slot
  end

  def free(slot)
    slot.next_free = @head   # 先頭につなぐ O(1)
    @head = slot
  end
end
```

もう一つの方式が**バンプアロケーション**（bump allocation）です。
連続した空き領域の先端を指すポインタを一本持ち、確保のたびにポインタを
要求サイズぶん**進める（bump する）だけ**。フリーリストよりさらに速く、
ポインタ加算と境界検査だけで済みます。ただし「使用中の領域に穴が
開かない」ことが前提なので、後述するコピー GC や世代別 GC の若い世代と
組み合わせて使われます。Java（HotSpot）の若い世代の確保は、スレッド
ごとに切り出した領域（TLAB）へのバンプアロケーションで、わずか数命令です。

## 参照カウント：捨てどきを数えで知る

ここから「いつ捨てるか」の判定に入ります。最も直観的なのが
**参照カウント**（reference counting）です。各オブジェクトのヘッダに
「自分を指している参照の数」を持たせ、参照が増えたら +1、減ったら -1、
0 になった瞬間に回収します。

CPython がこの方式の代表です。「値の表現」の章で見たとおり、CPython の
全オブジェクトはヘッダに参照カウンタ `ob_refcnt` を持ち、C のマクロ
`Py_INCREF`/`Py_DECREF` がソース中の無数の場所でカウンタを上下させて
います。Swift の **ARC**（Automatic Reference Counting）も同族で、
こちらはコンパイラが retain/release 命令を自動挿入します。

参照カウントの長所は、ごみが**即座に**回収されること（メモリ使用量が
低く抑えられ、デストラクタの実行タイミングも予測できる）。短所は
二つあり、一つはカウンタ操作が**あらゆる代入に**ついて回るコスト、
もう一つが**循環参照**（cycle、互いに参照し合う孤島）を回収できない
ことです。

```ruby
# 循環参照の例：a と b は互いを指すので、参照カウントが 0 にならない
a = []
b = [a]
a << b
# a も b も外から到達不能になっても、カウンタは 1 のまま —— 回収されない
```

CPython はこのために、コンテナ系オブジェクトを連結リストでつないで
おき、定期的に「外から到達できない循環」を検出する**循環検出 GC** を
参照カウントに併設しています。純粋な参照カウントだけでは言語の
メモリ管理として完結しない、というのが歴史の教訓です。

> [!NOTE]
> 参照カウントはマルチスレッドとの相性も課題です。カウンタの上下を
> 複数スレッドから安全に行うには高価な不可分操作が要ります。CPython の
> GIL（「並行処理とデータ構造」の章）の存在理由の一つはカウンタ保護で、
> GIL を外す試み（PEP 703）が、スレッドごとのカウンタを分ける
> biased reference counting などの新しいデータ構造を必要としたのは
> 偶然ではありません [](#cite:pep703)。この作り直しの全貌は
> 「GIL を外す」の章で解剖します。

## マーク＆スイープ：根から辿って生死を判定

参照カウントと対をなすのが**トレーシング GC**（tracing GC、追跡型）で、
その基本形が**マーク＆スイープ**（mark and sweep）です。発想を変えて、
「死んだものを数える」のではなく「**生きているものを辿り尽くす**」ことに
します。

1. **マーク**（mark）：**ルート**（root、確実に生きている出発点 ——
   スタック上の値、グローバル変数、レジスタなど）から参照を辿り、
   到達できたオブジェクトすべてに「生きている」印を付ける。
2. **スイープ**（sweep）：ヒープ全体を走査し、印のないオブジェクトを
   フリーリストに戻す。

マーク段階は、グラフ探索そのものです。「これから辿るべきオブジェクト」を
積む**マークスタック**（mark stack）を使った反復で実装します。素朴に
再帰で書くと、深くつながったデータ（長大なリストなど）で処理系自身が
スタックあふれを起こすため、明示的なスタック＋あふれたときの再走査
という防御が実装には必須です。

ここまでの部品で、動くマーク＆スイープ GC が書けます。

```ruby
# 動く最小のマーク＆スイープ GC
class GCHeap
  Obj = Struct.new(:refs, :marked)   # refs: このオブジェクトが指す先たち

  def initialize = (@objects = []; @roots = [])
  attr_reader :roots

  def allocate(refs = [])
    obj = Obj.new(refs, false)
    @objects << obj                  # 全オブジェクトの台帳（スイープ用）
    obj
  end

  def collect
    work = @roots.dup                # マークスタック
    until work.empty?
      obj = work.pop
      next if obj.marked             # 訪問済みなら何もしない（循環対策）
      obj.marked = true
      work.concat(obj.refs)          # 子をスタックに積む
    end
    @objects.select!(&:marked)       # スイープ：印のないものを捨てる
    @objects.each { |o| o.marked = false }   # 印を消して次回に備える
  end

  def live_count = @objects.size
end

heap = GCHeap.new
a = heap.allocate                # a ← b から参照される
b = heap.allocate([a])
heap.allocate([])                # 誰からも参照されない孤児
c = heap.allocate; c.refs << c   # 自分しか参照しない循環
heap.roots << b
heap.collect
p heap.live_count   # => 2  b と、b から届く a だけが生き残る
                    #        循環ごみ c も正しく回収される（参照カウントとの違い！）
```

実物との距離は「台帳とフリーリストがページ＋ビットマップになる」
「ルートがスタック走査になる」程度で、骨格はこのままです。前節の
循環参照の例（参照カウントでは回収できなかった `c`）が、ここでは
何の特別扱いもなく回収されることを確かめてください。

「生きている印」をどこに置くかにも設計があります。オブジェクトの
ヘッダのビットを使うのが素朴ですが、CRuby は**ページごとのビットマップ**
（オブジェクト本体とは別の場所にまとめた印の表）に印を付けます。
これは Unix の `fork` と相性をよくするための工夫です。`fork` した
子プロセスは親とメモリを共有し、**書き込んだページだけ**が複製される
（コピーオンライト）ので、GC のたびに全オブジェクトのヘッダに印を
書き込むと、せっかくの共有が全ページ複製で台無しになります。印を
ビットマップに隔離すれば、GC が書き込むのはビットマップのページだけで
済むのです。プリフォーク型の Web サーバを多用する Ruby ならではの、
利用環境がデータ構造を決めた例です。

## コピーGC：生きているものだけ引っ越す

マーク＆スイープは「死んだものを掃く」ためにヒープ全体を走査します。
ごみが大半を占める場合、これは無駄です。**コピー GC**（copying GC）は
逆に、**生きているものだけを別の領域へ引っ越し**、元の領域を丸ごと
空き地にします。

古典的な**半空間方式**（semispace）では、ヒープを from 空間と to 空間に
二分し、from 側で生きているオブジェクトを to 側へ詰めて並べ直します。
Cheney のアルゴリズムは、この引っ越しを**to 空間自体を待ち行列として
使う幅優先探索**で、再帰もマークスタックもなしに実現する美しい方法です
[](#cite:cheney1970)。

コピー GC の魅力は、コストが**生きているオブジェクトの量にしか比例しない**
こと、引っ越し先で**オブジェクトが連続に詰まる**ので断片化が消え、
バンプアロケーションが使えるようになること。代償は、ヒープの半分しか
使えないことと、**オブジェクトのアドレスが変わる**ことです。アドレスが
変わると、そのオブジェクトを指していたすべての参照を書き換える必要が
あり、「どこに参照があるか」を正確に知らなければなりません。この
**正確な（precise）GC** の要件が、次の話につながります。

## 保守的GC：ポインタかどうか分からないなら

「スタック上のこのワードはポインタか、それともたまたまアドレスに見える
整数か」を、処理系が常に判別できるとは限りません。特に C で書かれた
処理系や C 拡張ライブラリの中では、値がレジスタや C のスタックに散らばり、
型情報がありません。

**保守的 GC**（conservative GC）は、これを開き直りで解決します。
「ポインタ**かもしれない**値は、すべてポインタと見なして生かしておく」の
です [](#cite:boehm1988)。間違って生かしても安全側ですが、間違って
殺すことはない、という非対称性を利用しています。CRuby は機械スタックを
保守的に走査します。これにより **C 拡張が GC をほとんど意識せずに
書ける**という大きな利便を得た代わりに、スタックから（かもしれない）
参照されたオブジェクトは動かせない、という制約を抱えました。CRuby に
後年加わったコンパクション（`GC.compact`）が「動かせないオブジェクトは
その場に残す（ピン留めする）」方式なのは、この歴史的経緯ゆえです。

## 世代別GC：「若いものほど早く死ぬ」に賭ける

実プログラムのオブジェクトの寿命を観測すると、強い偏りがあります ——
**ほとんどのオブジェクトは生まれてすぐ死ぬ**。一時的な文字列、ループ内の
中間配列、すぐ捨てる小さなオブジェクト。これを**世代仮説**
（generational hypothesis）と呼びます [](#cite:ungar1984)。

**世代別 GC**（generational GC）はこの仮説に賭けて、ヒープを**若い世代**と
**老いた世代**に分けます。GC の大半は若い世代だけを対象にした**マイナー
GC** で済ませ、そこを生き延びたオブジェクトだけを老いた世代へ**昇格**
（promote）させます。若い世代は小さいので、マイナー GC は短時間で
終わります。

ここで重大な問題が生じます。マイナー GC は若い世代しか辿りませんが、
**老いた世代のオブジェクトが若いオブジェクトを指している**かもしれません。
その参照を見逃すと、生きている若いオブジェクトを誤って回収してしまいます。
かといって老いた世代を全部走査したら、世代別にした意味がありません。

解決策が、**「老→若」の参照だけを記録しておく**データ構造 ——
**記憶集合**（remembered set）です。そして記憶集合を維持するために、
オブジェクトへの書き込みのたびに「老いたオブジェクトに若い参照を
書き込んでいないか」を検査する小さなコード片 —— **書き込みバリア**
（write barrier）—— をコンパイラ／VM が全代入に仕込みます。

```ruby
# 概念図：書き込みバリア。代入のたびにこの検査が走る
def write_field(obj, field, value)
  if old_gen?(obj) && young_gen?(value)
    remembered_set.add(obj)    # 「老→若」参照の発生を記録
  end
  obj[field] = value           # 本来の代入
end
```

記憶集合の実装にも幅があります。CRuby はページごとのビットマップで
「記憶されたオブジェクト」を管理し、HotSpot（Java）は**カードテーブル**
（card table）—— ヒープを 512 バイトの「カード」に区切り、書き込みの
あったカードに印を付ける粗い表 —— を使います。カードテーブルは
偽陽性（印のついたカードに実は老→若参照がない）を許す代わりに、
バリアを「1 バイト書くだけ」まで軽くした設計です。

> [!TIP]
> CRuby の世代別 GC（RGenGC、2.1 で導入）には、他の処理系にない
> 事情がありました。膨大な既存 C 拡張は書き込みバリアなしで書かれて
> おり、全部を一斉に書き換えるのは不可能です。RGenGC は「バリアを
> 持つオブジェクト」と「持たない（WB unprotected な）オブジェクト」の
> 共存を許し、後者はマイナー GC のたびに保守的に扱うことで、
> 互換性を保ったまま段階的に世代別化する道を選びました
> [](#cite:sasada2019)。理論より移行戦略がデータ構造を決めた例です。

## インクリメンタル・並行GC：止まらないために

GC 中、アプリケーションは止まります（**stop the world**）。ヒープが
ギガバイト級になると、一括のマークで数百ミリ秒止まることもあり、
対話的なアプリケーションやサーバには致命的です。そこで、マーク作業を
小刻みに分割する**インクリメンタル GC**や、アプリケーションと同時に
走る**並行 GC**（concurrent GC）が登場します。

ここでも理論的支柱になるのが**三色抽象化**（tri-color abstraction）です。
オブジェクトを「白（未到達）・灰（到達したが子を未走査）・黒（子まで
走査済み）」に塗り分け、「黒から白への参照を作らせない」という不変条件を
保てば、マークを中断・再開しても正しさが保たれます。この不変条件を
守る番人もまた書き込みバリアです。世代別 GC のバリアと併せて、
**現代の GC は「全代入に仕込んだ検査コード＋それを支える表」で
できている**と言ってよいほどです。

Go の GC は並行マーク＆スイープ（移動なし・世代なし）に振り切った設計、
HotSpot の G1 や ZGC はリージョン分割と並行コンパクションを組み合わせた
設計で、「止まらないこと」と「スループット」の配分が言語の用途ごとに
違うのが見て取れます。

## 最前線のGC：ポインタの中のメタデータ、再び

近年の GC 研究からも、データ構造として面白いものを三つ紹介します。

**カラードポインタとロードバリア（ZGC）**。HotSpot の低遅延 GC で
ある ZGC は、なんと**ポインタのビットの中に GC の状態**（この参照は
マーク済みか、移動済みの可能性があるか）を埋め込みます。「値の表現」の
章で見たタグ付きポインタの技法を、アプリケーションではなく **GC 自身の
メタデータ**に使うのです。検査は書き込み時ではなく**読み込み時**に
行います（**ロードバリア**）。参照をロードするたびに数命令で色を検査し、
「移動中のオブジェクト」を踏んだら**その場で転送先に修正してから**
使う。これにより、オブジェクトの移動（コンパクション）すら
アプリケーションと並行に進められ、ヒープがテラバイト級でも停止時間は
ミリ秒以下に収まります。

**Immix：マークとコピーの中間**。マーク＆スイープは断片化に弱く、
コピー GC は空間を倍食う。Immix [](#cite:blackburn2008) はヒープを
「ブロック（32KB）とその中のライン（128B）」の二段で管理し、
基本はその場でマークしつつ、**断片化したブロックだけ**中身を
よそへ退避（evacuate）します。全部は動かさない、しかし詰め直しは
できる —— 二分法を疑った中庸の設計で、研究用 VM から実用処理系へ
影響を広げています。

**不死オブジェクト（CPython の PEP 683）**。`None` や `True`、小整数
キャッシュのような「絶対に死なない」オブジェクトの参照カウンタを
特別な値に固定し、**カウンタ操作そのものを省略**します
[](#cite:pep683)。動機が振るっていて、カウンタ更新がオブジェクトの
ページを汚すと `fork` のコピーオンライト共有が壊れる（本章の
ビットマップマーキングと同じ問題！）、マルチスレッド化でカウンタが
競合する —— つまり**「書かない」ことが最高の同期・共有戦略**だから
です。

## GC をしないという選択：領域と所有権

最後に、視野を広げておきます。「実行時にごみを探す」のとは別に、
**コンパイル時にメモリの寿命を決めてしまう**研究の系譜があります。

理論的な源流が**領域ベースのメモリ管理**（region-based memory
management）です [](#cite:tofte1997)。プログラムを解析して各値の
寿命を**領域**（構文木の章のアリーナの一般化）に割り当て、領域ごと
まとめて確保・解放するコードをコンパイラが挿入します。GC なしで
メモリ安全 —— ただし寿命の推論が悲観的になりがちで、長生きする
領域にごみが溜まる弱点もありました。

この系譜の現代の到達点が **Rust の所有権システム**です。値には
唯一の所有者がいて、所有者がスコープを抜けたら解放。共有したければ
借用（コンパイル時に検査される参照）か、参照カウント（`Rc`/`Arc`）を
**明示的に**選ぶ。本章の言葉で言えば、Rust は「ごみ探し」の実行時
データ構造を、**型検査というコンパイル時の証明に置き換えた**言語です。
その代償は、二重連結リストのような「所有が循環する」構造を素直に
書けないこと —— `unsafe` か参照カウント＋弱参照の出番になります。
GC の章の締めくくりとして象徴的でしょう。**どのデータ構造が
書きやすいかは、メモリ管理の方式が決める**のです。逆もまた然り、
というのが本章で見てきたことでした。

## 弱参照とファイナライザ：生死の境界条件

最後に、GC と言語機能の境界に現れるデータ構造に触れます。

**弱参照**（weak reference）は「指してはいるが、生かす力はない」参照です。
キャッシュ（あれば使うが、メモリが苦しければ捨ててよい）の実装に
使われます。GC は弱参照を特別扱いし、マーク終了後に「死んだものを
指していた弱参照」を無効化します。Ruby の `ObjectSpace::WeakMap`、
Java の `WeakReference`、Lua の弱テーブルがこれです。

さらに微妙なのが、JavaScript の `WeakMap` や Lua の弱テーブルのような
**エフェメロン**（ephemeron）です。規則は「**キーが（このマップ以外の
経路で）生きているときに限り、値も生かす**」。なぜこんな回りくどい条件が
要るのか ── 素朴に「マップに入っている以上、値もマークする」とやると、
値がキーを参照していた場合に〈値が生きる → だからキーも生きる → だから
このエントリは消えない〉という自家撞着の循環に陥り、本当はもう誰も使って
いない組が永久に残ってしまうからです。

この条件は、ふつうの「たどって塗る」マークの一発では決まりません。GC は
エフェメロンに出会っても**値はすぐ塗らず脇に置き**、通常のマークが収束
してから「キーが塗られたエフェメロン」の値だけを塗ります。ところが値を
塗ると、その先で**別のエフェメロンのキー**が生きると判明することがあり、
塗り直しが要る ── つまり「変化がなくなるまで繰り返す」**不動点反復**に
なります。最後までキーが塗られなかったエフェメロンは、キーも値も死んだ
ものとしてエントリごと消えます。たかが連想配列の変種が、GC のマークを
単純な一巡から不動点計算へと押し上げる ── 言語機能とメモリ管理が
分かちがたく結びついている好例です。

**ファイナライザ**（finalizer、回収直前に実行されるフック）も、
「回収すべきオブジェクトのリスト」を GC からアプリケーションへ受け渡す
キューという、専用のデータ構造を必要とします。

なお、GC のマークと同じ「参照をたどってグラフを巡る」仕事が、
もう一つあります。ヒープ上のオブジェクトグラフをファイルや
ネットワークへ**書き出す**直列化（`Marshal` など）です。共有と
循環をどう写し取るかという問題も含めて、「直列化と永続化」の章で
まとめて扱います。

## まとめ：GCはデータ構造の総合芸術

ページとサイズクラス、フリーリスト、ビットマップ、マークスタック、
記憶集合、カードテーブル、待ち行列としての to 空間 —— GC は本書で
学ぶ基本データ構造の応用展示場です。そして値の表現（タグでポインタを
見分ける）、スタックとフレーム（ルートの所在）、オブジェクトの
ヘッダ設計と、第I部で見てきたすべての話題が GC に合流します。

次の章では、第I部の総仕上げとして、これらすべての構造の上で動く
**JIT コンパイラ**のデータ構造 —— 観測の台帳と、賭けに負けたとき
安全に後戻りするための対応表 —— を見ていきます。
