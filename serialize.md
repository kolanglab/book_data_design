# 直列化と永続化：データに外の世界を旅させる

## ヒープの外へ

ここまでの型は、すべて**一つのプロセスのヒープの中**で生きていました。
しかしデータは外へ出たがります。ファイルに保存したい、ネットワークの
向こうへ送りたい、別の言語のプログラムに渡したい。ポインタだらけの
オブジェクトグラフを**バイト列に写し取る**操作が**直列化**
（serialization、Ruby の `Marshal`、Python の `pickle`、Java の
Object Serialization）、その逆が**復元**（deserialization）です。

直列化は、本書で学んだ原則の総決算のような分野です。ポインタは
使えない（別のプロセスではアドレスが無意味）、名前は重い、グラフは
そのままでは書けない —— つまり「**ポインタをオフセットに、名前を
番号に、グラフを木＋後方参照に**」変換する必要があります。順に
見ていきましょう。

## 共有と循環：訪問済み表と後方参照

素朴な「再帰的に書き出す」実装は二つの理由で破綻します。**共有**
（同じオブジェクトが二ヶ所から参照されていたら、二度書くと復元後に
別物になる）と**循環**（`a` が `b` を、`b` が `a` を指していたら
無限ループ）です。解は GC のマーク（[メモリ管理とGCの章](memory.md)）と同じで、
**訪問済み表**を引き連れて歩くことです。

```ruby
# 巡回・共有に耐える直列化の骨格
def serialize(obj, seen = {})
  if (id = seen[obj.object_id])
    return [:backref, id]          # 二度目は「さっきの n 番」とだけ書く
  end
  seen[obj.object_id] = seen.size  # 訪問済み表に通し番号で登録
  [:object, obj.class.name, obj.fields.map { |f| serialize(f, seen) }]
end
```

書く側は「オブジェクト → 通し番号」のハッシュ、読む側は「通し番号 →
復元済みオブジェクト」の**配列**を持ち、**後方参照**（backreference）で
共有と循環を再現します。読む側が配列で済む（番号は登場順なので
添字アクセスできる）あたり、[識別子の章](identifier.md)の「名前 → ID → 配列」の
三段構えと同じ形です。

```ruby
a = []; a << a                  # 自分自身を含む配列
b = Marshal.load(Marshal.dump(a))
p b.equal?(b[0])                # => true  循環が正しく復元される
```

## Marshal の解剖：バイト列の中のインターン

Ruby の `Marshal` フォーマットは、この章の教材として最適です。
中身は「**1 バイトの型タグ＋型ごとのペイロード**」の再帰的な並び ——
つまり**バイト列に直列化されたタグ付き値**（[値の表現の章](values.md)）です。

```ruby
p Marshal.dump(42)[0, 4].bytes   # => [4, 8, 105, 47]
#  4.8  = フォーマットのバージョン（major.minor）
#  105  = 'i'：これは整数、という型タグ
#  47   = 整数本体（小さい整数は 1 バイトに収まる符号化）
```

見どころを三つ挙げます。

- **小さい整数の特別扱い**：整数のペイロードは可変長で、おおよそ
  −123〜122 は 1 バイト、それ以上は「続くバイト数＋本体」という
  符号化です。Fixnum を即値にした処理系の発想（小さい値は圧倒的に
  多いので最短にする）が、ファイル形式にも貫かれています。
- **ストリーム内シンボル表**：シンボルの初出はタグ `:` で綴りを
  書き、**二度目以降はタグ `;` と「何番目に出たシンボルか」だけ**を
  書きます。[識別子の章](identifier.md)のインターンが、バイト列の中で再演されて
  いるのです（オブジェクトの後方参照には別のタグ `@` が使われます）。
- **拡張用のフック**：独自の内部表現を持つクラスは `marshal_dump`／
  `marshal_load` で「外に出せる表現」へ自分で変換します。直列化
  できない値（IO、Proc など）はここで例外になります。OS のリソース
  ハンドルやコード（[クロージャの章](closures.md)で見た「コード＋環境」）は、
  バイト列に写し取れる範囲の外にあるからです。

## pickle はスタック機械である

Python の `pickle` は、さらに踏み込んだ設計です。pickle データは
「読まれるデータ」ではなく、**実行される命令列**——スタック機械の
バイトコード（[構文木と中間表現の章](syntax-tree.md)）そのもの——です。

```
# pickle のディスアセンブル（python -m pickletools）
0: \x80 PROTO   5          # プロトコル番号
2: ]    EMPTY_LIST         # 空リストをスタックに積む
3: \x94 MEMOIZE            # ← 訪問済み表（memo）への登録も命令！
4: K    BININT1 1          # 整数 1 を積む
6: K    BININT1 2
8: e    APPENDS            # スタック上の要素をリストに詰める
9: .    STOP               # スタックの先頭が結果
```

復元器は命令を順に解釈してスタックを操作し、`STOP` で先頭を返す
だけ。後方参照は「memo 表に積め（MEMOIZE）」「memo の n 番を積め」
という**命令**になっています。フォーマットの拡張は「新しい命令の
追加」であり、古い読み手との互換性はプロトコル番号で管理する ——
直列化形式の設計が、そのまま小さな VM の設計になっている例です。

## 実行できることの代償：デシリアライズ攻撃

「復元器が命令を実行する」設計には、暗い裏面があります。pickle の
命令には「**任意の呼び出し可能オブジェクトを呼べ**」（REDUCE）が
含まれており、細工したバイト列を `pickle.loads` に食わせると
**任意コード実行**になります。Python の文書が「信頼できないデータを
unpickle するな」と赤字で警告するのはこのためです。

命令型でない Marshal や Java の直列化も安全ではありません。復元とは
「バイト列の指示に従って、**任意のクラスのインスタンスを、
コンストラクタを通さずに**組み立てる」操作だからです。攻撃者は
アプリケーションにロード済みのクラスを部品に、復元の副作用
（`hash` の計算、ファイナライザ、`readObject` フック）を連鎖させて
攻撃を組み立てます（**ガジェットチェーン**と呼ばれます。Java では
これが大規模な脆弱性群になり、復元前にクラス名を検査するフィルタ
機構が後付けされました）。ハッシュ衝突攻撃（[ハッシュの章](hashes.md)）、
ReDoS（[正規表現の章](regexp.md)）に続く、「データ構造の機構そのものが攻撃面に
なる」第三の実例です。

> [!WARNING]
> 教訓は単純です。**言語ネイティブの直列化（Marshal・pickle・Java
> serialization）は、信頼できる相手とのプロセス間でだけ使う**。
> 外部と交換するデータは、「任意のクラスを組み立てる」能力のない、
> データ専用の形式（JSON・MessagePack・Protocol Buffers など）に
> 限ること。表現力の高さと安全性は、ここでは正面衝突します。

## 交換形式の設計：varint と zigzag

では、言語をまたぐ交換形式はどう設計されているのでしょう。テキスト
形式の代表 JSON は人間に読めて普遍的ですが、データ構造の視点では
制約が目立ちます。循環は表せず（書こうとするとエラー）、数値は
事実上 double なので **64 ビット整数が精度落ちする**（[数値の章](numbers.md)の
2⁵³ 問題が、フォーマットの仕様として再来します。だから多くの API は
ID を文字列で運びます）。

バイナリ交換形式の中身も覗いておきましょう。Protocol Buffers の
整数符号化 **varint** は、実装できるほど単純で、かつ本書的です。

```ruby
# varint：7 ビットずつ下位から書き、続きがあれば最上位ビットを立てる
def varint_encode(n)
  bytes = []
  loop do
    b = n & 0x7f
    n >>= 7
    bytes << (n.zero? ? b : b | 0x80)
    break if n.zero?
  end
  bytes
end
p varint_encode(300)   # => [0xAC, 0x02]  小さい数ほど短い
```

「小さい値は圧倒的に多い」に賭けた可変長表現 —— Marshal の整数や
UTF-8（[文字列の章](strings.md)）と同じ賭けです。負数は素朴にやると 64 ビット
全部が立って最長になるため、**zigzag 符号化**（`(n << 1) ^ (n >> 63)`
で −1→1、1→2、−2→3 …と絶対値の小さい順に並べ替える）を先に
かませます。ビットの並べ替え一つで「0 の近くが短い」性質を負数に
まで広げる、[値の表現の章](values.md)の流儀そのものです。

さらに Protocol Buffers の各フィールドは、名前ではなく**番号**
（field tag）で書かれます。「名前は早い段階で番号にする」原則
（[シンボルテーブルの章](symbol-table.md)）がスキーマ進化の道具になっていて、
読み手は知らない番号を読み飛ばせばよい（前方互換）、書き手は
古い番号を使い続ければよい（後方互換）。**名前→番号の対応表
（.proto ファイル）だけを人間の世界に残し、ワイヤ上は番号だけ**
という構成です。

## パースしないという最終手段：ゼロコピー形式

復元のコストすら払いたくない場面（ゲーム、モバイル、巨大データの
ランダムアクセス）には、**ゼロコピー形式**があります。FlatBuffers や
Cap'n Proto は、データを「**ポインタの代わりに相対オフセットで
つながった構造**」としてファイルに置き、復元せずに**その場で
たどって**読みます。`mmap` すれば、ギガバイトのファイルの 1 フィールド
だけを O(1) で読める。

これは[構文木の章](syntax-tree.md)で見た「フラットな AST」（添字でつなぐ木）の
ファイル版であり、Smalltalk の章のイメージや V8 のスナップショット
（ロード時のポインタ付け替えは要ります）と同じ系譜です。Apache
Arrow が列指向データ（[APL の章](apl.md)）をゼロコピーでプロセス間共有する
のも同じ思想 —— **「直列化」の究極は「直列化しないこと」**、
すなわちメモリ上の表現そのものを交換可能に設計しておくことなのです。

似た文脈で語られる CRuby の ISeq バイナリ（コンパイル済み
バイトコードを直列化する汎用機構で、Bootsnap による起動短縮は
その代表的な応用です）は、この系譜の**一歩手前**に位置します。字句解析・構文解析・コンパイルは丸ごと省けるものの、
ロード時には命令列のデコードやオブジェクトの再構築といった変換が
ひと通り走るので、「mmap してそのまま使う」わけではありません。
どこまでを保存時に済ませ、どこからをロード時に払うか —— ゼロ
コピーに寄せるほどロードは速くなる代わりに、フォーマットが
プロセス内表現（ポインタの幅、アドレス、処理系のバージョン）に
密着して互換性と安全性の検証が難しくなります。本章で見てきた
「互換性は形式の寿命を決める」という話が、ここでは逆向きの圧力に
なるのです。

## 同じ値が一意のバイト列になるとき：正準化と内容アドレス

ここまでは「バイト列に写して、復元できればよい」という前提でした。
ところが、**同じ値が何通りものバイト列に直列化されうる**という事実が、
別の場面で牙をむきます。連想配列のキーをどの順で書くか、整数を何バイトで
書くか、JSON の空白や数値の綴り（`1` か `1.0` か `1e0` か）、Unicode の
正規化 ── どれも復元すれば同じ値になるので、転送目的では問題になりません。
しかし、次の三つは**バイト列そのもの**を気にします。

- **署名・改ざん検出**：署名は値ではなく**バイト列**にかかる。受け取って
  読み込み、別の処理系で書き直すとバイトが変わり、署名が外れてしまう。
- **重複排除・キャッシュ**：同じ内容は同じキーに落ちてほしい。バイト列が
  ぶれると、同じデータが別物として二重に保存される。
- **分散合意・再現性**：ブロックチェーンのように、全ノードが**寸分違わぬ
  バイト列**を独立に再現できないと、ハッシュが一致せず合意できない。

これらが要求するのが、**正準形（canonical form）** ── 各値がただ一つの
バイト列に対応する決定的な直列化です。連想配列のキーを定めた順に並べ、
整数は最短符号で書き、無意味な空白を禁じ、数値表記と Unicode 正規化を
固定する。標準も整備されていて、JSON には JSON Canonicalization Scheme、
CBOR には決定的符号化の規定、ASN.1 には BER の正準部分集合である DER が
あります。逆に **Protocol Buffers は仕様上は正準ではなく**（フィールド順や
未知フィールドの扱いが一意でない）、「protobuf を署名・ハッシュの対象に
してはいけない」は有名な落とし穴です。

正準化が本領を発揮するのが**内容アドレス指定**（content addressing）です。
データの**正準バイト列のハッシュ値そのものを名前（識別子）にする**設計で、
Git がその代表です。Git のオブジェクトは「型とサイズのヘッダ＋中身」を
ハッシュした値を ID とし、**同じ内容は必ず同じ ID**になります ── だから
重複は自動的に一つに畳まれ、改変は ID の変化として即座に露見します。
IPFS の CID、Docker のレイヤ、Nix のストアパスも同じ原理です。識別子の
章では「名前を早く番号にする」と述べましたが、内容アドレスはその逆 ──
**中身そのものから名前を導く**。そして[ハッシュの章](hashes.md)のハッシュ関数が、
ここでは「データの指紋＝住所」として働きます。直列化の安定性が、
データの**同一性の土台**になるわけです。

## まとめ

- グラフは**訪問済み表＋後方参照**で木に直す（GC のマークと同型）
- フォーマットの中身は**タグ付き値**であり、小さい整数の最適化や
  **ストリーム内インターン**（Marshal のシンボル表）まで、ヒープ内の
  技法が再演される
- pickle は**スタック機械**。表現力は最強、ゆえに**デシリアライズ
  攻撃**という代償（外部入力にはデータ専用形式を）
- 交換形式は **varint・zigzag・フィールド番号**——「小さい値に
  賭ける」「名前を番号にする」原則のワイヤ版
- 究極は**ゼロコピー**：オフセットでつないだ構造を mmap して
  そのまま使う
- **正準形**は各値を一意のバイト列にし、署名・重複排除・合意を支える。
  その極が**内容アドレス指定**（Git など、中身のハッシュを名前にする）

ヒープの中の設計原則が、バイト列の上でそっくり繰り返される ——
直列化はこの本の縮図です。次の章では、複数の実行の流れが同じ
データ構造に触るとき何が起きるか —— **並行処理**を見ていきます。
