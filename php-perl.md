# PHP と Perl：万能配列と自動有効化

## 現場の言語のコレクション哲学

PHP と Perl は、Web と運用スクリプトの現場で鍛えられた言語です。
その現場感覚は、コレクションの設計に色濃く出ています。第II部で
見たように「配列・ハッシュ・リスト・集合を別の型として提供し、
適切に選ばせる」のが教科書的な設計だとすれば、この二言語は
**「考えなくても書ける」方向**へ振り切りました。その結果生まれた
データ構造は、率直に言って変わっていますが、内部を見ると合理の
塊です。

## PHP の array：一つで全部の順序付きハッシュ

PHP のコレクションは長らく実質一種類、`array` だけでした。これは
リストでも辞書でもなく、**両方**です。

```php
$a = [10, 20, 30];          // リストのつもり（キーは 0,1,2）
$a["name"] = "Alice";       // 同じ値に文字列キーも混ぜられる
$a[] = 40;                  // 「次の整数キー」= 3 が自動で振られる
foreach ($a as $k => $v) {} // 挿入順で回る
```

内部実装（zend のハッシュテーブル）は、ハッシュの章で見た
CPython/CRuby と同じ「**挿入順のエントリ配列＋ハッシュ索引**」です。
PHP 7 での内部刷新はまさにこの密な配列化で、メモリを激減させ
ました。さらに整数キーだけが順に並んでいる場合は索引を省いて
**ただの配列として**振る舞う最適化（packed array）もあり、
Lua の「配列部＋ハッシュ部」（ハッシュの章）と同じ着地点に、
別の道から到達しています。

PHP らしさが出るのは**「次の整数キー」カウンタ**です。`$a[] = x` の
ための「これまでに使われた最大整数キー＋1」を表自身が覚えています。
そして要素を削除してもこのカウンタは**戻りません**。`unset` で
末尾を消してから `[]` で足すと、キーが飛びます。「リストとしての
顔」と「辞書としての顔」を一つの構造に同居させた縫い目が、
こういう癖として表に出るのです。

> [!NOTE]
> PHP の array は**値意味論**で、代入や関数渡しで「コピーされた
> ように」振る舞います。実装は R の章で見たのと同じ参照カウント
> 付きコピーオンライトです。巨大な配列を関数に渡しても、書き換える
> までコピーは起きません。R・Swift・PHP という出自のまったく違う
> 三言語が同じ解に収斂しているのは、「値意味論＋大きなコレクション」
> という要求の組み合わせが必然的にこの構造を呼ぶからです。

## Perl の自動有効化：読むだけで木が生える

Perl（と PHP）には**自動有効化**（autovivification）という機能が
あります。ネストしたハッシュの奥に代入すると、**中間のハッシュが
すべて自動生成される**のです。

```perl
my %h;
$h{2026}{6}{8} = "memo";   # %h、$h{2026}、$h{2026}{6} が勝手に生える
```

多段の集計表を育てるコード（`$count{$year}{$month}++` など）が
1 行で書ける、現場には絶大にありがたい機能です。データ構造の
視点では、これは**「読み」と「書き」の中間にある「経路の確保」と
いう操作**を、言語が暗黙に提供しているということです。Ruby なら
`Hash.new { |h, k| h[k] = Hash.new(&h.default_proc) }` という
イディオム（再帰的デフォルトブロック）で同じものを**明示的に**
作ります —— ハッシュの章で見た「デフォルト値の仕組み」の応用です。

罠も同じ場所にあります。Perl では**読むだけのつもり**の式
（`exists $h{a}{b}`）でも、`$h{a}` までは中間ノードが生えます。
「観測が状態を変える」データ構造は、デバッグの直観を裏切ります。
利便と引き換えに「参照透過な読み取り」を手放す —— この設計判断の
損得は、いまも言語設計者の間で意見が割れるところです。

## Perl の each：イテレータが表の中に住む

Perl のハッシュ反復には、歴史的な珍構造があります。`each %h` は
呼ぶたびに「次のキーと値」を返しますが、その**反復位置（カーソル）が
ハッシュ自身の中に 1 個だけ**保存されているのです。

```perl
while (my ($k, $v) = each %h) { ... }   # 便利そうに見えるが…
# ループ途中で別の場所が each %h を呼ぶと、カーソルが共有されて大事故
```

反復の状態を「反復子オブジェクト」（外側）に持つか、「コレクション
自身」（内側）に持つか。Perl は内側を選び、再入できない・入れ子に
できないという制約を背負いました。現代の言語がほぼ例外なく外側
（Ruby の `Enumerator`、Python のイテレータ、Java の `Iterator`）を
選ぶのは、この種の先行の失敗から学んだ結果です。**状態の置き場所を
一つ間違えるだけで、データ構造は合成可能性を失う** —— 小さくて
深い教訓です。

なお Perl のハッシュは、ハッシュの章で見た衝突攻撃（hash-flooding）
への対応が最も早かった言語の一つでもあります（2003 年に乱数化を
導入）。現場の言語は、攻撃にも最初にさらされるのです。

## 型の三すくみ：Perl のシジルと文脈

Perl ではデータ構造の**種類が変数名に刻まれます**。`$x` はスカラー、
`@x` は配列、`%x` はハッシュ —— 三つは**別の変数**で、同名で共存
できます。さらに式の評価が「スカラー文脈」「リスト文脈」のどちらで
要求されたかで結果が変わります（配列をスカラー文脈で評価すると
要素数になる、など）。

これは「値の表現」の章の用語で言えば、**型タグを値ではなく変数と
構文に置いた**設計です。利点は、実行時の型判別が不要になり、
コンパイル時にアクセスする内部構造（スカラー用 SV・配列用 AV・
ハッシュ用 HV）が決まること。代償は、参照が絡んだときの記法の
複雑さ（`@{$h{list}}` のような脱参照地獄）です。型情報をどこに
置くか —— 値の中（動的タグ）、変数（シジル）、コンパイル時の型
（静的型付け）—— という三択の、二番目を選んだ稀有な実例として、
Perl は教材的価値があります。

## 系譜と教訓

- **万能配列**は「学習コストの最小化」という要求が生んだ構造で、
  内部は結局「エントリ配列＋索引」という主流の構造に収斂した。
  packed array 最適化は V8 の要素種別・Lua の配列部と同じ発想
- **コピーオンライト値意味論**は R・Swift と収斂進化
- **自動有効化**は「デフォルト値の自動生成」の言語組み込み版。
  明示（Ruby のブロック付き Hash）と暗黙（Perl/PHP）の好対照
- **each の内蔵カーソル**は、反復子の状態の置き場所という設計問題の
  反面教師

個性派言語の最後に、「一つで全部」をもう一段徹底した言語を訪ねます。
PHP の万能配列が単一の順序付きハッシュで押し通したのに対し、次章の
**Lua** は配列とハッシュを一枚の**テーブル**に同居させ、オブジェクトも
名前空間も、そのテーブル一種類でまかなってしまいます。
