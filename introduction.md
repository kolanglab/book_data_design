# はじめに：処理系とデータ構造

## 本書の位置づけ

本書は、[『言語処理系入門』](https://kolanglab.github.io/book_langimpl_intro/)の
**副読本**です。本編が字句解析・構文解析からコード生成・実行まで、処理系の
「作り方」を工程に沿って解説するのに対し、本書はその全工程を**データ構造**と
いう一本の軸で貫き直します。構文解析アルゴリズムやコード生成の一般論は
本編に譲り、ここでは「処理系の中で、データはどんな**形**をしていて、
なぜその形なのか」だけを徹底して掘り下げます。

そのため本書は、読者に次の素養を仮定しています。

- Ruby のコードがおおむね読めること（書けなくてかまいません）
- O 記法（計算量の表し方）の基本
- ポインタ・スタック・ヒープといったメモリのおおまかな直感

処理系そのものに初めて触れる方は、本編から読み始めることをお勧めします。
逆に「処理系は一度作った（読んだ）ことがある、内部のデータ構造を
もっと知りたい」という方は、本書だけで読み通せるはずです。

## 言語処理系とは何か

**言語処理系**（language processor）とは、プログラミング言語で書かれた
ソースコードを受け取り、それを実行可能な形へ変換したり、その場で実行したり
するソフトウェアの総称です。大きく分けると次の二種類があります。

- **コンパイラ**（compiler）：ソースコードを別の言語（多くは機械語や
  バイトコード）へ「翻訳」するプログラム。翻訳と実行が分かれています。
- **インタプリタ**（interpreter）：ソースコードを読み込みながら、その意味を
  解釈して直接実行するプログラム。

現実の処理系はこの二つの中間にいることが多く、たとえば Ruby や Python は
ソースコードをいったん**バイトコード**（bytecode、仮想機械向けの中間的な
命令列）に変換し、それを仮想機械（VM）が解釈実行します。Java も同様に
バイトコードへコンパイルし、JVM（Java仮想機械）が実行します。
コンパイラの古典的な教科書である Dragon Book [](#cite:aho2006) は、この
変換の各段階を体系的に論じています。

処理系の内部は、おおよそ次のような段階（フェーズ）に分かれます。

```mermaid
graph LR
    A[ソースコード<br/>文字列] --> B[字句解析<br/>トークン列]
    B --> C[構文解析<br/>構文木]
    C --> D[意味解析<br/>注釈付き木]
    D --> E[コード生成<br/>バイトコード]
    E --> F[実行<br/>仮想機械]
```

- **字句解析**（lexical analysis）：文字の列を、意味のある最小単位である
  **トークン**（token、`if`・`+`・`123` のような語）に切り分けます。
- **構文解析**（parsing）：トークン列が文法に合っているかを調べ、プログラムの
  構造を**構文木**として組み立てます。
- **意味解析**（semantic analysis）：変数が宣言されているか、型が合っているかと
  いった「意味」のチェックを行います。
- **コード生成**（code generation）：実行できる命令列を出力します。
- **実行**（execution）：生成された命令を仮想機械や CPU が動かします。

## なぜデータ構造が主役なのか

これらの段階を貫いて言えるのは、**各段階は前の段階が作ったデータ構造を
受け取り、次の段階のためのデータ構造を作る**、ということです。
字句解析はトークンの列を作り、構文解析はそれを木に変え、コード生成は木を
命令の列に変えます。処理系とは、いわば**データ構造から
データ構造への変換器**の連なりなのです。

> [!NOTE]
> 「アルゴリズム + データ構造 = プログラム」という有名な等式があります。
> 言語処理系はこの言葉が最もよく当てはまる分野のひとつで、適切なデータ構造を
> 選ぶことが、処理系の速度・メモリ使用量・実装のしやすさを直接決めます。

本書では、言語処理系に登場するデータ構造を **3つの立場** から眺めます。

1. **処理系を実装するためのデータ構造**（第I部）
   名前から情報を引く**シンボルテーブル**、識別子を軽い印に変える
   **インターン**、プログラムの構造を表す**構文木**と中間表現。さらに
   実行を支える土台として、あらゆる値を一様に持ち運ぶ**値の表現**、
   関数呼び出しを管理する**スタックとフレーム**、オブジェクトの生と死を
   司る**メモリ管理と GC**、観測と投機で高速化する **JIT コンパイラ**、
   そしてプログラムを部品から組み立てる**モジュールとロード**。
   これらは利用者から見えませんが、処理系の骨格です。
2. **言語が提供するデータ構造**（第II部）
   整数・浮動小数点数・文字列・配列・ハッシュ・リスト・集合・平衡木・
   オブジェクト・クロージャ・遅延評価のサンク・時間・正規表現・
   直列化、そして並行処理のための構造。締めくくりには、CPython の
   GIL 撤廃を題材に、要求の変化がデータ構造を総取り替えさせる現場を
   解剖します。言語の利用者が日々使うデータ型
   たちも、内部では緻密に設計されたデータ構造です。Ruby を軸に、Python・JavaScript・
   Java・Go・Rust・Haskell・OCaml・Erlang・Clojure・Lua など、
   できるだけ多くの処理系を見比べます。
3. **個性派言語のデータ構造**（第III部）
   すべてが配列の APL、変数の束縛を取り消せる Prolog、実行中の
   フレームまでオブジェクトの Smalltalk、辞書とスタックだけの Forth、
   すべてが文字列の Tcl、失敗が制御構造の Icon、引数がすべて遅延される
   R、配列が一種類しかない PHP。独自の世界観が生んだ独自の構造と、
   それらが現代の主流言語へ転生していく系譜を追います。

> [!NOTE]
> **用語について**：本書で「データ型」（または文脈上紛れがない場合は
> 単に「型」）と言うときは、整数・文字列・配列のような**実行時の
> 値の種類と、その内部表現**を指します。静的型付け言語の型システム
> （コンパイル時の型検査・型推論）の「型」とは別の概念で、そちらに
> 言及するときは「静的型」「型システム」「型推論」のように明示します。
> 両者が交差する場面（静的型がデータ表現を決める、型推論の実装に
> データ構造が要る、など）こそ本書の見どころですが、概念としては
> 区別して読んでください。

## 本書の読み方

本書のコード例は、特に断らない限り **Ruby** で示します。Ruby を選ぶのは、
記述が簡潔で擬似コードに近く、かつ Ruby 自身の処理系（CRuby、MRI とも
呼ばれます）が本書のテーマである豊富なデータ構造を備えているからです。
たとえば次のコードは、文字列の中に現れる単語を数える短いプログラムです。

```ruby
def word_count(text)
  counts = Hash.new(0)        # 値の初期値を 0 にしたハッシュ
  text.scan(/\w+/) do |word|  # 正規表現で単語を取り出す
    counts[word.downcase] += 1
  end
  counts
end

p word_count("the cat sat on the mat")
# => {"the" => 2, "cat" => 1, "sat" => 1, "on" => 1, "mat" => 1}
```

この数行のなかにも、本書で扱うデータ構造がいくつも顔を出しています。
`Hash.new(0)` の**ハッシュ**、`/\w+/` の**正規表現**、
`"the cat..."` という**文字列**、`counts` の値である**整数**、
そして `do ... end` の**ブロック**（クロージャ）。ふだん何気なく
使っているこれらの型が、処理系の中でどう実装されているのかを知ることが
本書の目的です。

各章は独立して読めるように書いていますが、用語は前の章で導入したものを
後の章で使うことがあります。章をまたぐ参照は「[ハッシュ（辞書）の章](hashes.md)」の
ように章の名前で示しますので、気になったら遠慮なく前後を行き来して
ください。実装をとことん追いたい読者には、Nystrom の
*Crafting Interpreters* [](#cite:nystrom2021) や、計算の意味そのものを
扱う *SICP* [](#cite:abelson1996) を併読すると理解が深まります。

それでは、まず処理系の内部を支えるデータ構造から見ていきましょう。
